# --interactive--

The main camera is a window into the world. As your player walks east, the camera scrolls east, and every game object the camera sees scrolls with it. That's exactly what you want for trees, enemies, and platforms — but it's exactly **wrong** for the score, the heart icons, and the pause button.

You can pin individual objects to the camera with `setScrollFactor(0)`:

```js
this.scoreText = this.add
  .text(8, 8, "Score: 0")
  .setScrollFactor(0);
```

That works for one or two HUD elements. For a real HUD — score, lives, mini-map, dialogue boxes, pause overlay — you have a few problems with the per-object approach:

1. **You repeat yourself.** Every UI object needs `setScrollFactor(0)`. Forget one and it drifts off-screen the first time the player moves.
2. **You can't shake the world without shaking the HUD.** `cameras.main.shake()` shakes everything attached to the main camera, including any object with `scrollFactor: 0`. Scroll factor only blocks scroll, not shake.
3. **You can't tint the world without tinting the HUD.** A red flash for damage applied to `cameras.main` colours the HUD too.
4. **Layering gets ambiguous.** UI must be drawn over world objects, but `setDepth(9999)` mixes UI and world objects in the same depth space — refactors get fragile.

The clean solution is a **second camera dedicated to the UI**. The main camera renders the world. The UI camera covers the same screen area but is a separate `Camera` instance. You add the camera once and configure how each object is shared between cameras.

```js
this.uiCamera = this.cameras.add(0, 0, 480, 320);
this.uiCamera.setScroll(0, 0);
```

`cameras.add(x, y, width, height)` creates a new viewport. By default it sees every game object in the scene. The next two lectures show how to make the main camera ignore HUD objects and the UI camera ignore world objects — once that's done, the HUD is rock-stable while the world scrolls, shakes, and flashes.

:::interactive_editor

```js
const main = { scroll: { x: 0, y: 0 }, shake: 0 };
const ui = { scroll: { x: 0, y: 0 }, shake: 0 };

function moveWorld(dx) {
  main.scroll.x += dx;
}

function shakeWorld(amount) {
  main.shake = amount;
}

moveWorld(120);
shakeWorld(0.02);

console.log("main camera:", main);
console.log("ui camera (untouched):", ui);
```

:::

The mental model: think of cameras as render targets pointing at the same scene. Each one has independent scroll, zoom, shake, flash, and tint state. Whatever you do to the main camera does **not** affect the UI camera — that's the whole point.

Common bug: you create the UI camera but never tell either camera what to ignore. Result — every object renders twice, once per camera. Worst case is invisible (UI hides world); best case is doubled and slightly misaligned because the cameras have different scrolls. Lecture 2 fixes this with `camera.ignore(...)`.

# --questions--

## --text--

Why isn't `setScrollFactor(0)` enough for a real game HUD?

## --answers--

It blocks scroll only — shake and flash on the main camera still affect HUD objects, and you must remember to set it on every single UI object.

---

It works perfectly; a second camera is just for performance.

### --feedback--

Performance is not the reason. The reason is that scroll factor only addresses scroll — not shake, flash, or tint.

---

`setScrollFactor(0)` only works on `Image` objects, not `Text`.

### --feedback--

It works on every game object that supports scroll factor, including Text. The issue is what scroll factor *doesn't* cover.

---

`setScrollFactor` is deprecated.

### --feedback--

It is not deprecated — it has its uses. But for a full HUD, a dedicated UI camera is cleaner.

## --text--

What does `this.cameras.add(0, 0, 480, 320)` do?

## --answers--

Creates a new camera viewport at `(0, 0)` covering 480 × 320 pixels in addition to the main camera.

---

Replaces the main camera with a new one of that size.

### --feedback--

`cameras.add` adds a camera; it doesn't replace the main one. Use `cameras.main` for the main camera.

---

Adds a UI element at that position.

### --feedback--

That would be `this.add.something`, not `this.cameras.add`.

---

Resizes the canvas.

### --feedback--

Canvas size is set in `Phaser.Game` config. `cameras.add` configures camera viewports.

## --text--

You shake the main camera and the player sees the score text shake too — even though it has `setScrollFactor(0)`. Why?

## --answers--

Scroll factor only stops scroll, not shake. Shake is applied at the camera level and affects every object that camera renders, including pinned ones.

---

Phaser bug — file an issue.

### --feedback--

Not a bug. Shake and scroll are different transforms.

---

You forgot to call `setScrollFactor(0, 0)` with both x and y.

### --feedback--

`setScrollFactor(0)` is a shorthand for both axes. The issue is shake, not scroll.

---

The text is a child of a container that scrolls.

### --feedback--

If that were the cause it would also drift on scroll, not just shake. Shake propagates per-camera.
