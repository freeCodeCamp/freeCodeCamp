# --interactive--

You have two cameras. Now: which one draws **on top**?

Phaser draws cameras in the order they were added to the camera manager. The main camera (`cameras.main`) is added first, automatically, when the scene boots. Every camera you `add()` after it is drawn over the cameras already in the list. That's why the typical pattern is:

```js
this.uiCamera = this.cameras.add(0, 0, 480, 320);
```

…right after `create()` starts. The UI camera comes after main, so it renders last, and the HUD always sits above the world. If you accidentally call `cameras.add` *before* setting up the world, you can flip the order; the world ends up on top of the HUD. The fix is either to add the UI camera last or to call `setRenderOrder` to reorder them — both work, both are explicit.

Within a single camera, `setDepth()` controls draw order. Higher depth draws later (on top). The two systems compose:

- **Camera draw order:** `main` → `ui` (later cameras paint on top)
- **Within `main`:** `background (-100)` → `world (0)` → `world-fx (100)` → `world-text (200)`
- **Within `ui`:** `hud-bg (0)` → `hud-text (10)` → `toast (50)`

The world's text is *under* the HUD's background even though it has a higher depth — depth is per-camera. That's important: you never have to compete with HUD elements when picking world depth values.

A multi-camera scene also clarifies what "screen space" means. The UI camera scrolls at `(0, 0)` and stays there forever. To draw a HUD element at the top-left, you give it world coordinates `(8, 8)` — there's no separate "screen space." Because the UI camera doesn't scroll, world `(8, 8)` *is* screen `(8, 8)` for that camera. No transforms, no math.

:::interactive_editor

```js
const layers = [
  { name: "world-bg", camera: "main", depth: -100 },
  { name: "platforms", camera: "main", depth: 0 },
  { name: "particles", camera: "main", depth: 100 },
  { name: "hud-bg", camera: "ui", depth: 0 },
  { name: "score", camera: "ui", depth: 10 },
  { name: "toast", camera: "ui", depth: 50 }
];

const drawOrder = layers
  .slice()
  .sort((a, b) => {
    if (a.camera !== b.camera) return a.camera === "main" ? -1 : 1;
    return a.depth - b.depth;
  })
  .map(l => `${l.camera}/${l.name}@${l.depth}`);

console.log(drawOrder.join("\n"));
```

:::

Two camera-level effects you'll reach for routinely:

```js
this.cameras.main.shake(200, 0.012);
this.cameras.main.flash(120, 255, 64, 64);
```

Apply them to the **main camera only** — the HUD sits on the UI camera and stays still. That's the payoff for the setup work: the world can shake and flash all it wants while the score, lives, and pause hint never blink.

Common pitfall: a developer adds a "screen-shake the *whole* screen" effect by calling `this.cameras.shake()`. There's no such method on the manager — only on individual cameras. To shake everything (rare, but useful for cinematic moments) you'd loop:

```js
this.cameras.cameras.forEach(c => c.shake(200, 0.012));
```

But for the standard "world shakes, HUD doesn't" — just shake `cameras.main`.

# --questions--

## --text--

Which camera renders on top by default?

## --answers--

The camera added most recently — main is added first, so any `cameras.add()` after that draws over the main camera.

---

The main camera always renders on top.

### --feedback--

It renders first, not last. It ends up *under* every camera you add after it.

---

The camera with the highest `depth` value.

### --feedback--

Cameras don't have a `depth`; that's a game-object property. Camera order is set by the camera manager.

---

Whichever camera the active scene is using.

### --feedback--

Multiple cameras can be active at once in the same scene; that's the whole point.

## --text--

You want the world to shake but the HUD to stay still. What do you call?

## --answers--

`this.cameras.main.shake(200, 0.012)` — only the main camera shakes; the UI camera (which renders the HUD) is untouched.

---

`this.cameras.shake(200, 0.012)`

### --feedback--

There is no `shake` on the camera manager. Shake is per-camera.

---

`this.cameras.main.shake(200, 0.012); this.uiCamera.shake(200, 0.012);`

### --feedback--

The second call would shake the HUD too — the opposite of what you want.

---

`this.scene.shake(200, 0.012)`

### --feedback--

Scenes don't shake; cameras do.
