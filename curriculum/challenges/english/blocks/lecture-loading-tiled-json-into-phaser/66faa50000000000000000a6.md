# --interactive--

Two layers, painted in the right order, are usually enough to read a level. The floor goes down first; the walls go on top of the floor; the player goes on top of the walls. The order is decided by the order you call `createLayer` *and* the explicit `setDepth` you assign to each.

```js
const floor = map.createLayer("floor", tileset, 0, 0);
floor.setDepth(0);

const walls = map.createLayer("walls", tileset, 0, 0);
walls.setDepth(10);

this.player.setDepth(20);
```

Phaser draws by depth — lower numbers paint first. Two objects with the same depth fall back to creation order. That second rule sounds harmless but it is the one that bites you: the moment you spawn the player after the layers, the player sits on top by accident, and the moment you spawn it before the layers, the player is buried. Pinning each layer to an explicit depth is the cure.

:::interactive_editor

```js
function create() {
  const map = this.make.tilemap({ key: "room1" });
  const tileset = map.addTilesetImage("dungeon", "tiles");

  const floor = map.createLayer("floor", tileset, 0, 0);
  floor.setDepth(0);

  const walls = map.createLayer("walls", tileset, 0, 0);
  walls.setDepth(10);

  this.player = this.physics.add.sprite(40, 60, "hero");
  this.player.setDepth(20);
}
```

:::

If you have decorations — torches, banners, pillars — that should appear *in front* of the walls but *behind* the player, you give them depth `15` (between walls and player). Keep depths spaced (`0`, `10`, `20`, `30`) so you have room to slot in new layers later without re-numbering everything.

There is one more dial: `scrollFactorX` / `scrollFactorY`. By default, layers move with the camera at the same speed as the player. Setting `scrollFactor` to `0` pins the layer to the screen — useful for HUDs and minimaps. Setting it to a fraction (`0.5`) makes the layer scroll at half-speed, which is the recipe for parallax backgrounds (you will see this again in chapter 6). For a normal dungeon map, leave it at the default.

# --questions--

## --text--

You spawn the player after `createLayer("walls", ...)`. The wall tiles render *on top* of the player. Why?

## --answers--

Both have depth `0` (the default), so creation order tied them — and the walls layer was created later because of an out-of-order call somewhere. Set the player's `setDepth` higher than the walls layer.

---

`physics.add.sprite` always places below tilemap layers.

### --feedback--

It does not — depth is symmetric across game-object types.

---

Tilemap layers are unsortable.

### --feedback--

They sort by depth like everything else.

---

The walls texture is opaque on alpha 0xff and that takes priority.

### --feedback--

Alpha does not change the depth sort.

## --text--

Why pick `setDepth(0)`, `setDepth(10)`, `setDepth(20)` instead of `0`, `1`, `2`?

## --answers--

The wider gaps leave room to slot new things in (decorations between walls and player at `15`) without re-numbering anything.

---

Phaser truncates depths smaller than `5`.

### --feedback--

It does not. Depths are plain numbers.

---

The depth has to be a multiple of ten.

### --feedback--

It does not have to be — but spacing is a convention for editability.

---

Larger numbers render faster.

### --feedback--

Render time is depth-agnostic.

## --text--

You want the floor layer to stay locked to the camera viewport (e.g. for a debug grid overlay). Which property do you set?

## --answers--

`floor.setScrollFactor(0)` — a scroll factor of zero pins the layer to the screen so the camera does not move it.

---

`floor.setDepth(-1)` — a negative depth glues the layer to the camera.

### --feedback--

Depth controls paint order, not screen-pinning.

---

`floor.alpha = 0`.

### --feedback--

That hides the layer; it does not pin it.

---

`this.cameras.main.ignore(floor)`.

### --feedback--

That tells the main camera *not* to render it — opposite of what we want.
