# --interactive--

A Tiled map needs two assets in the loader: the **map JSON** (the grid, layers, and objects) and the **tileset image** (the PNG the gids point at). Phaser exposes one call for each. Both go in `preload`:

```js
preload() {
  this.load.image(
    "tiles",
    "/curriculum-assets/phaser/tilemaps/tileset-dungeon.png"
  );
  this.load.tilemapTiledJSON(
    "room1",
    "/curriculum-assets/phaser/tilemaps/room-01.json"
  );
}
```

The `key` you pass to `load.image` (`"tiles"`) is the cache key Phaser uses to find the texture later. The `key` for the JSON (`"room1"`) is how you ask `make.tilemap` to build the parsed tilemap. These are separate caches with separate names, so you can name them anything — many projects pick one shared name like `"dungeon"` for both, but that overlaps two distinct concepts in the cache and trips up other tools.

:::interactive_editor

```js
function preload() {
  this.load.image(
    "tiles",
    "/curriculum-assets/phaser/tilemaps/tileset-dungeon.png"
  );
  this.load.tilemapTiledJSON(
    "room1",
    "/curriculum-assets/phaser/tilemaps/room-01.json"
  );
}
```

:::

`load.image` and `load.tilemapTiledJSON` are both fire-and-forget — they queue the asset and Phaser fetches it before `create` runs. If either path is wrong, you do not see an error in `preload`; you see the canvas mount and immediately throw inside `create` when `make.tilemap` cannot find the cache entry, or when `addTilesetImage` cannot find the texture. The fix is always the same: re-check the path strings.

One subtle thing: the path you pass is **resolved relative to the page that loads Phaser**, not relative to the JSON. So a JSON file at `/tilemaps/room-01.json` whose `tilesets[].image` points at `tileset-dungeon.png` will work as long as the *page* serves the PNG from the same directory. With Gatsby's static asset folder, both live at `/curriculum-assets/phaser/tilemaps/` and the relative-image path inside the JSON resolves correctly.

# --questions--

## --text--

What two assets does a Tiled map need from the loader?

## --answers--

The map JSON via `this.load.tilemapTiledJSON(...)` and the tileset PNG via `this.load.image(...)`.

---

Just the JSON — Phaser pulls the PNG from the JSON automatically.

### --feedback--

Phaser does *resolve* the JSON's `image` field, but the PNG still has to be in the loader cache as an image asset.

---

Just the PNG — Phaser embeds the JSON.

### --feedback--

The JSON is a separate file Phaser needs to fetch and parse.

---

A `.tmx` file and a sprite sheet.

### --feedback--

`.tmx` is Tiled's native XML format; Phaser uses Tiled's **JSON** export.

## --text--

You name your image cache key `"tiles"` and your JSON cache key `"room1"`. Where do those keys reappear?

## --answers--

`"room1"` goes into `make.tilemap({ key: "room1" })` and `"tiles"` goes into `map.addTilesetImage(<tilesetName>, "tiles")`.

---

Both go into `make.tilemap({ key })`.

### --feedback--

Only the JSON key goes there. The image key is for `addTilesetImage`.

---

Both go into `addTilesetImage`.

### --feedback--

Only the image key goes there. The JSON key is for `make.tilemap`.

---

You never use them again — they are just for the loader's debug log.

### --feedback--

You absolutely use them. They are how your `create` code retrieves the assets.

## --text--

The page renders fine in `preload`, but `create` throws on `addTilesetImage`. What is almost always wrong?

## --answers--

The image cache key passed to `addTilesetImage` does not match the key you used in `load.image` — re-check the spelling.

---

Tiled exported the JSON in a version Phaser does not understand.

### --feedback--

Possible but rare. Check the cache key first.

---

Phaser's WebGL renderer is missing.

### --feedback--

Renderer issues throw at config time, not at `addTilesetImage`.

---

The map is too large.

### --feedback--

Size limits are different errors. Cache miss is the common one.
