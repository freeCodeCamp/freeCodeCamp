# --interactive--

Once a clip is loaded there are two ways to play it. Both work. They are not the same.

```js
this.sound.play("sfx-coin");
```

`sound.play(key)` is **fire-and-forget**. Phaser allocates a fresh sound instance every call, plays it, and disposes it when it ends. You don't get a reference back. You can't pause it, change its volume mid-flight, or hook a listener to its `complete` event. It's the right tool for "every coin pickup plays the same coin SFX, untouched, end of story."

```js
const music = this.sound.add("music-ambient", { loop: true, volume: 0.6 });
music.play();
music.setVolume(0.3);
music.stop();
```

`sound.add(key, config)` returns a **`Sound` instance**. You keep the reference. You can call `setVolume`, `setRate`, `setDetune`, `pause`, `resume`, `stop`, listen for `complete` and `looped` events, swap config mid-playback. It's the right tool for music, ambience, looping engine sounds, anything you'll touch after `play()`.

Default config object on `add`:

```js
{
  mute: false,
  volume: 1,
  rate: 1,
  detune: 0,
  seek: 0,
  loop: false,
  delay: 0
}
```

You can also pass a config to `play` directly: `this.sound.play("sfx-jump", { volume: 0.4 })` plays once at half volume without needing to keep a reference. Use this for one-shots that need a tweak.

:::interactive_editor

```js
function decideAddOrPlay(useCase) {
  const needsControl =
    useCase.loops ||
    useCase.duckable ||
    useCase.fadeable ||
    useCase.pauseable;
  return needsControl ? "sound.add" : "sound.play";
}

console.log(decideAddOrPlay({ loops: true })); // music
console.log(decideAddOrPlay({ loops: false, duckable: false })); // coin SFX
console.log(decideAddOrPlay({ loops: false, fadeable: true })); // ambience that ducks
```

:::

Memory note: clips played via `sound.play(key)` get auto-cleaned. Clips obtained via `sound.add(key)` live until you call `.destroy()`. If you `add` 50 instances and never destroy them, the `SoundManager` ends up with 50 dormant `Sound` objects, each holding a buffer reference. Always destroy long-lived instances on `scene.shutdown`:

```js
this.events.once("shutdown", () => music.destroy());
```

A subtle one: `sound.play("key")` does **not** play the instance you created with `sound.add("key")` earlier. It plays a *new* one. If you want to play the same instance twice, call `instance.play()` (with no args) on the reference. Forgetting this leads to "why is my music starting from the beginning even though I called `play` again?" — because you weren't playing the instance, you were creating a new one.

# --questions--

## --text--

You're playing a coin pickup SFX. Every coin sounds identical, you don't need to fade it, and there's no reason to keep a reference. Which API do you reach for?

## --answers--

`this.sound.play("sfx-coin")` — fire-and-forget, Phaser cleans up automatically.

---

`this.sound.add("sfx-coin").play()` — keep the reference for safety.

### --feedback--

You'd be hand-managing a `Sound` instance you don't actually need to control. `sound.play` is the lighter tool here.

---

`new Phaser.Sound.WebAudioSound(this.sound, "sfx-coin").play()`.

### --feedback--

That bypasses the manager entirely. Always go through `this.sound`.

---

`this.cache.audio.get("sfx-coin").play()`.

### --feedback--

The cache holds the decoded buffer, not a `Sound` instance. You can't `play` it directly.

## --text--

You stored a music instance with `const music = this.sound.add("music")` and now want to restart it from the beginning. Which call does that?

## --answers--

`music.play()` — playing a `Sound` instance that's already created restarts it; passing a key to `sound.play` would create a new instance instead.

---

`this.sound.play("music")` — same effect.

### --feedback--

That creates a *new* `Sound` instance each call. Your original `music` reference is now orphaned (still in the manager, never cleaned up).

---

`music.start()`.

### --feedback--

There is no `start` method. The Web Audio source has one internally, but Phaser exposes `play`.

---

`music.replay()`.

### --feedback--

There is no `replay` method. `play()` on an existing instance restarts it.

## --text--

You called `this.sound.add("music")` 50 times in a level and never destroyed any of them. What's the consequence?

## --answers--

The `SoundManager` keeps 50 dormant instances and their buffer references; eventually the level's GC pressure or `seek` lookup time grows.

---

Phaser auto-cleans them on `scene.shutdown`.

### --feedback--

Only the buffer cache is cleared on scene shutdown; instances created with `add` need explicit `destroy`.

---

The audio element gets garbage-collected after one second.

### --feedback--

Web Audio `AudioBufferSourceNode`s do not get GC'd while the manager holds references. You leak.

---

Calling `add` more than once silently returns the same instance.

### --feedback--

Each call creates a new instance. That's exactly the leak.
