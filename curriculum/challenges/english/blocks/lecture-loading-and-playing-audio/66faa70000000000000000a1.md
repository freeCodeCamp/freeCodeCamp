# --interactive--

Audio loads in `preload` the same way images do, but the call accepts an array of URLs instead of a single one:

```js
preload() {
  this.load.audio("sfx-jump", [
    "audio/sfx-jump.ogg",
    "audio/sfx-jump.mp3"
  ]);
}
```

Phaser walks the array left-to-right and uses the **first** format the browser claims it can decode. Why ship two? Because no single audio container plays everywhere:

| Format | Chrome | Firefox | Safari      | Notes                           |
| ------ | ------ | ------- | ----------- | ------------------------------- |
| OGG    | yes    | yes     | **no**      | Open, smaller files at low bps. |
| MP3    | yes    | yes     | yes         | Universal; slightly larger.     |
| WAV    | yes    | yes     | yes         | Uncompressed; bloats bundles.   |

`OGG` is roughly half the size of an equivalent-quality `MP3`, so listing it first lets Chrome and Firefox win on file size while Safari falls through to the `MP3`. Ship both, list `OGG` first.

A second concept worth knowing now is the **audio sprite**. Like a sprite sheet for images, an audio sprite is one big audio file plus a JSON map describing where each clip lives inside it:

```js
this.load.audioSprite(
  "sfx-bundle",
  "audio/sfx-bundle.json",
  ["audio/sfx-bundle.ogg", "audio/sfx-bundle.mp3"]
);
this.sound.playAudioSprite("sfx-bundle", "jump");
```

One HTTP request, one decode pass, instant playback for every clip. For a soundboard with six SFX you can use either approach; for a 100-SFX RPG, audio sprites are non-negotiable.

Finally, `load.audio` *decodes* on the main thread. Decoding 20 sound effects synchronously can stall a low-end phone for hundreds of milliseconds. Two mitigations:

1. Defer non-critical SFX to a second `preload` (e.g. load menu SFX in `BootScene`, gameplay SFX in `LevelScene`).
2. Use audio sprites — one decode instead of N decodes.

:::interactive_editor

```js
function pickFormat(supports) {
  const order = ["ogg", "mp3", "wav"];
  return order.find(fmt => supports[fmt]) ?? null;
}

console.log("chrome:", pickFormat({ ogg: true, mp3: true, wav: true }));
console.log("safari:", pickFormat({ ogg: false, mp3: true, wav: true }));
console.log("none:  ", pickFormat({ ogg: false, mp3: false, wav: false }));
```

:::

The fallback function above mirrors what Phaser does internally — it picks the first format the browser supports.

A common pitfall: people ship only `MP3` "because it works everywhere," then their bundle is twice the size it should be. The other common pitfall: `load.audio("jump", "audio/sfx-jump.ogg")` (no array) will silently fail in Safari with no error — the resource is "loaded" as `null` and `sound.add("jump").play()` is a no-op.

# --questions--

## --text--

Why do you ship both `OGG` and `MP3` for the same SFX?

## --answers--

Safari does not decode `OGG`; without an `MP3` fallback the file silently fails to play there.

---

`OGG` is faster to decode than `MP3`.

### --feedback--

Decode speed is roughly the same. The reason is browser support.

---

Phaser plays both files at once for a richer sound.

### --feedback--

Phaser picks one. The other is a fallback the browser may never even fetch.

---

Mixing two formats halves the file size.

### --feedback--

Each format is fetched separately on demand, but only one ever plays. The point is browser coverage.

## --text--

You ship `load.audio("jump", "audio/sfx-jump.ogg")` (single string, no array). What happens in Safari?

## --answers--

Safari cannot decode `OGG`, the audio entry resolves to `null`, and `sound.play("jump")` does nothing without throwing.

---

Phaser throws a `Format not supported` error on `preload`.

### --feedback--

It does not throw. The decode just fails silently and `sound.play` becomes a no-op.

---

Safari falls back to a built-in beep.

### --feedback--

There is no built-in fallback. Silence is the default.

---

Safari converts `OGG` to `MP3` on the fly.

### --feedback--

Browsers do not transcode. You must ship the format they can decode.

## --text--

You have 100 SFX in a single scene. What's the cleanest way to keep `preload` fast?

## --answers--

Bundle them as one audio sprite plus a JSON map of clip offsets, then call `playAudioSprite`.

---

Load each one with its own `load.audio` call.

### --feedback--

That triggers 100 HTTP requests and 100 decodes. The whole point of audio sprites is to fold those into one.

---

Use 100 separate `MP3` files but mark them `lazy: true`.

### --feedback--

Lazy-load helps with initial load, but at the moment the level needs them you still pay 100 decodes. An audio sprite pays one.

---

Convert all SFX to `WAV` for instant playback.

### --feedback--

`WAV` skips decoding but bloats the bundle ~10×. The right answer is one bundled file with a map.
