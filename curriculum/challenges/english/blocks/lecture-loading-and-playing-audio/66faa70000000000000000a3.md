# --interactive--

Open your game on iOS Safari and call `this.sound.play("music")` from `create`. Silence. The user opens the inspector, sees no errors, and is convinced something is broken.

Nothing is broken. Mobile (and desktop) browsers refuse to start audio that wasn't triggered by a user gesture. The Web Audio `AudioContext` boots in the **suspended** state and only transitions to **running** after a `pointerdown`, `keydown`, `touchstart`, or `click` originating from the user. Any `play` calls before that transition are queued or dropped depending on the browser.

Phaser exposes the manager state and a helper to handle the unlock:

```js
console.log(this.sound.locked); // true on first frame in iOS Safari

this.input.once("pointerdown", () => {
  if (this.sound.locked) {
    this.sound.unlock();
  }
});
```

`this.sound.locked` is `true` until the first user input. `this.sound.unlock()` resumes the context if it's suspended and triggers a one-shot inaudible silent buffer, which forces iOS to actually transition the context to `running`. After that, every subsequent `sound.play` works as you'd expect.

The standard pattern: show a "TAP TO START" overlay, listen for the first input, then unlock and dismiss the overlay.

:::interactive_editor

```js
function shouldShowUnlockOverlay(soundManager, hasInteracted) {
  return soundManager.locked && !hasInteracted;
}

console.log(shouldShowUnlockOverlay({ locked: true }, false));
console.log(shouldShowUnlockOverlay({ locked: true }, true));
console.log(shouldShowUnlockOverlay({ locked: false }, false));
```

:::

Two things commonly go wrong:

1. **Autoplaying music in `create` and assuming it'll catch up.** It won't. Even when `unlock` runs later, the `play` call you made before unlock was a no-op — there's no queue. You have to issue `play` *after* unlock.
2. **Listening for the wrong event to unlock.** `pointerdown` covers mouse and touch on the game canvas. But if your unlock UI is HTML overlaid on the canvas, you also need a `domcontentadded` event or the `unlock` won't fire. Phaser's own `this.sound` listens to `pointerdown`, `pointerup`, and `keydown` on the game canvas; if your overlay is HTML, capture the click yourself and call `this.sound.unlock()` from there.

Phaser also fires a `unlocked` event you can hook for analytics or to start the music exactly once:

```js
this.sound.once("unlocked", () => {
  music.play();
});
```

Lastly: even desktop Chrome enforces this in the `Site Settings → Sound` flag now, and Firefox respects `media.autoplay.default`. Treat audio unlock as a universal boot step, not an iOS-specific hack.

# --questions--

## --text--

`this.sound.play("music")` from `create` produces silence on iOS Safari. The console shows no errors. What's the cause?

## --answers--

The `AudioContext` is suspended until the first user gesture; the `play` call ran but produced no sound, and there is no automatic queue.

---

The audio file failed to load.

### --feedback--

If the file failed to load, `this.sound.play` would warn `Cannot play missing audio key`. The console is silent.

---

Phaser disables audio on small screens.

### --feedback--

Phaser does no such check. The browser is the gatekeeper.

---

Safari requires `MP3` and you shipped `OGG`.

### --feedback--

That's a different bug. This one is the autoplay restriction; the symptom is identical regardless of file format.

## --text--

You bound `this.input.once("pointerdown", ...)` on the game canvas, but your "TAP TO START" overlay is an HTML `<div>` on top of the canvas. The user taps the overlay and audio is still locked. Why?

## --answers--

The HTML overlay swallows the pointer event before it reaches the Phaser canvas, so Phaser never fires `pointerdown` and `unlock` is never called.

---

Phaser unlocks audio synchronously and the listener fired too late.

### --feedback--

The listener doesn't run if the event never reaches the canvas.

---

iOS only unlocks on `keydown`.

### --feedback--

iOS unlocks on any user gesture, including `touchstart` on a DOM element.

---

You need to call `setInteractive` on the overlay.

### --feedback--

`setInteractive` is for Phaser game objects, not HTML.

## --text--

What's the cleanest way to start music exactly once after the user's first gesture?

## --answers--

Listen for the `this.sound.once("unlocked", ...)` event and call `music.play()` from inside that callback.

---

Call `music.play()` in `create`; Phaser will replay it after unlock.

### --feedback--

There is no replay-after-unlock queue. The call is a no-op.

---

Poll `this.sound.locked` in `update` and call `play` when it flips to false.

### --feedback--

Polling works but races with the unlock; the `unlocked` event is the deterministic hook.

---

Use `setTimeout(() => music.play(), 1000)` to wait for unlock.

### --feedback--

Time-based hacks are flaky. The user might tap in 200 ms or 30 seconds.
