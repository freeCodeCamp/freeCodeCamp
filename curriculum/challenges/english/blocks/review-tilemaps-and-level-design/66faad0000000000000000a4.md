## Tilesets and tilemaps

A **tileset** is one image sliced into a grid of equal-sized tiles. A **tilemap** is a 2D array of indices into that tileset — `0` means "tile 0 from the set", `1` means "tile 1", and so on. The same tileset can paint thousands of distinct levels because the level data is just numbers. The browser fetches one image, the GPU uploads one texture, and Phaser stamps tiles by index every frame.

Phaser loads tilemaps in two halves: the tileset image (a normal `this.load.image` call), and a Tiled JSON map file via `this.load.tilemapTiledJSON`.

:::interactive_editor

```js
function preload() {
  this.load.image(
    "tiles",
    "/curriculum-assets/phaser/tilesets/dungeon.png"
  );
  this.load.tilemapTiledJSON(
    "dungeon-01",
    "/curriculum-assets/phaser/tilemaps/dungeon-01.json"
  );
}

function create() {
  const map = this.make.tilemap({ key: "dungeon-01" });
  const tileset = map.addTilesetImage("dungeon", "tiles");
  const floor = map.createLayer("floor", tileset, 0, 0);
  const walls = map.createLayer("walls", tileset, 0, 0);

  walls.setDepth(10);
}
```

:::

`map.addTilesetImage(nameInTiled, keyInPhaser)` is where Phaser binds the JSON's tileset reference to a loaded image. If the names disagree, Phaser will tell you in the console — pay attention to the casing.

## Index-based vs property-based collision

The simplest way to mark walls is by tile index: `walls.setCollision([1, 4, 7, 12])`. It works, and for tiny maps it's fine. The cost is **fragility** — every time you re-arrange the tileset, every level breaks.

The robust pattern is to add a `collides: true` custom property to the wall tiles in Tiled, then ask Phaser to read it:

:::interactive_editor

```js
function create() {
  const map = this.make.tilemap({ key: "dungeon-01" });
  const tileset = map.addTilesetImage("dungeon", "tiles");
  const walls = map.createLayer("walls", tileset, 0, 0);

  walls.setCollisionByProperty({ collides: true });
}
```

:::

Now adding a new wall tile is one box-tick in Tiled. The map data and the collision data move together.

When you only need *most* tiles to collide and a few exceptions, `setCollisionByExclusion([2, 9])` flips it the other way: collide with everything *except* the listed indices.

## Per-tile collision rectangles

A pillar tile that is visually 16×16 but should only block the lower 12×8 needs a tighter hit-box. Tiled lets you draw collision rectangles per tile in the **Tile Collision Editor**. Phaser reads them automatically — there is no extra Phaser API call.

If you don't have access to Tiled, you can override at runtime with `tile.setCollision(left, right, up, down, recalc)` per tile, but that's a rescue tool — the source of truth should live in the map file.

## Object layers

Tile layers describe terrain. **Object layers** describe everything else: spawn points, NPCs, pickups, doors, triggers, patrol paths, exits. An object can be a point, a rectangle, a polyline, or a polygon, and every object can carry custom properties.

In Phaser, `map.findObject(layerName, predicate)` returns the first match; `map.filterObjects(...)` returns every match. The key insight: **objects are data, not display nodes**. You read them in `create` and decide what to render — usually a sprite or a rectangle with a physics body attached.

:::interactive_editor

```js
function create() {
  const map = this.make.tilemap({ key: "dungeon-01" });

  const spawn = map.findObject("objects", (o) => o.type === "spawn");
  this.player = this.add.rectangle(spawn.x, spawn.y, 12, 12, 0xff66cc);
  this.physics.add.existing(this.player);

  const torches = map.filterObjects(
    "objects",
    (o) => o.type === "torch"
  );
  torches.forEach((t) => {
    this.add.sprite(t.x + 8, t.y + 8, "tiles", 25);
  });
}
```

:::

The `+ 8` is because Tiled stores tile-object positions at the bottom-left corner; Phaser sprites are origin-centred by default. Always shift by half a tile when rendering tile objects.

## Custom properties

Every Tiled object can carry a `properties` array. They are how the *level designer* configures the *programmer's* code without touching JavaScript: a door tile carries `locked: true`, a chest carries `loot: "gold"`, a trigger carries `event: "boss-fight"`. Phaser flattens them onto the object so you can read them as plain fields.

:::interactive_editor

```js
function create() {
  const map = this.make.tilemap({ key: "dungeon-01" });

  const doors = map.filterObjects("objects", (o) => o.type === "door");
  doors.forEach((d) => {
    if (d.properties && d.properties.find((p) => p.name === "locked").value) {
      this.add.sprite(d.x + 8, d.y + 8, "tiles", 13).setTint(0xff5566);
    } else {
      this.add.sprite(d.x + 8, d.y + 8, "tiles", 13);
    }
  });
}
```

:::

In recent Phaser versions, `properties` may already be flattened into `o.locked` directly when imported from a Tiled v1.10+ JSON. If `o.locked` is `undefined`, fall back to scanning the array as above.

## Polylines for patrol paths

A patrol path is a list of waypoints. Tiled's polyline tool produces an array of `{ x, y }` points relative to the object's origin; convert to world space by adding the origin:

:::interactive_editor

```js
function create() {
  const map = this.make.tilemap({ key: "dungeon-01" });

  const slimeObj = map.findObject("objects", (o) => o.type === "slime");
  this.slime = this.add.rectangle(slimeObj.x, slimeObj.y, 14, 12, 0x88ee66);
  this.physics.add.existing(this.slime);

  this.slimePoly = slimeObj.polyline.map((p) => ({
    x: slimeObj.x + p.x,
    y: slimeObj.y + p.y
  }));
  this.slimeTarget = 1;
}
```

:::

Each frame, point the body toward `slimePoly[slimeTarget]`; when within a few pixels, advance `slimeTarget` to the next waypoint (wrapping with `% length`). That single trick produces every "guard pacing along a path" enemy you'll ever need.

## Doors and one-shot triggers

A door is a special tile in the walls layer plus a one-shot trigger in your code. The pattern: read the door object, watch for the player getting close, look at custom properties to decide what happens, and remove the wall tile when the door opens.

:::interactive_editor

```js
function create() {
  const map = this.make.tilemap({ key: "dungeon-01" });
  const doorObj = map.findObject("objects", (o) => o.type === "door");

  this.events.on("update", () => {
    const close =
      Phaser.Math.Distance.Between(
        this.player.x,
        this.player.y,
        doorObj.x,
        doorObj.y
      ) < 20;
    if (close && this.inventory.has("key")) {
      const tx = doorObj.properties.find((p) => p.name === "tx").value;
      const ty = doorObj.properties.find((p) => p.name === "ty").value;
      this.walls.removeTileAt(tx, ty);
    }
  });
}
```

:::

`removeTileAt(tx, ty)` deletes the wall tile at that grid cell. Phaser's collision data updates immediately — the player can walk through next frame.

## Camera, world bounds, and minimaps

Set the world bounds and the camera bounds to the map size, then `startFollow` the player. `setLerp(0.1)` smooths the chase — the camera trails the player at 10% of the gap per frame, which feels natural.

:::interactive_editor

```js
function create() {
  const map = this.make.tilemap({ key: "dungeon-01" });

  this.physics.world.setBounds(0, 0, map.widthInPixels, map.heightInPixels);
  this.cameras.main.setBounds(0, 0, map.widthInPixels, map.heightInPixels);
  this.cameras.main.startFollow(this.player);
  this.cameras.main.setLerp(0.1);

  this.minimap = this.cameras
    .add(252, 4, 60, 40)
    .setZoom(0.1)
    .setBackgroundColor(0x000000);
  this.minimap.setBounds(0, 0, map.widthInPixels, map.heightInPixels);
  this.minimap.startFollow(this.player);
}
```

:::

A minimap is just a second camera. `cameras.add(x, y, w, h)` carves a small viewport at screen coordinates `(x, y)`. Tell each camera to `ignore` the entities you don't want it to render — minimaps usually skip the swing rectangle, the dialogue box, and other gameplay-only objects so the map shows clean walls and floor.

## Multi-scene HUDs and event bridges

The HUD lives in its own scene so it can render at depth 100 without interfering with the gameplay scene. Launch it with `this.scene.launch("UIScene")`, and bridge gameplay → HUD with the gameplay scene's event emitter.

:::interactive_editor

```js
function gameplayCreate() {
  this.scene.launch("UIScene");
  this.events.emit("ui:hp", this.hp);
  this.events.emit("ui:score", this.score);
}

function uiCreate() {
  const main = this.scene.get("MainScene");

  main.events.on("ui:hp", (hp) => this.drawHp(hp));
  main.events.on("ui:score", (s) =>
    this.scoreText.setText("Gold: " + s)
  );

  this.events.on("shutdown", () => {
    main.events.off("ui:hp");
    main.events.off("ui:score");
  });
}
```

:::

The shutdown listener is non-negotiable. Without it, every scene restart leaves stale handlers behind that double-fire on the next run.

## Save and load with localStorage

Persistence in a browser game is `JSON.stringify` to `localStorage` on save, `JSON.parse` on load. Sets serialize as arrays — convert back with `new Set(array)`.

:::interactive_editor

```js
function create() {
  this.saveGame = () => {
    localStorage.setItem(
      "dungeon-save",
      JSON.stringify({
        x: this.player.x,
        y: this.player.y,
        hp: this.hp,
        score: this.score,
        inventory: [...this.inventory],
        flags: [...this.flags]
      })
    );
  };

  this.loadGame = () => {
    const raw = localStorage.getItem("dungeon-save");
    if (!raw) return;
    const data = JSON.parse(raw);
    this.player.setPosition(data.x, data.y);
    this.hp = data.hp;
    this.score = data.score;
    this.inventory = new Set(data.inventory);
    this.flags = new Set(data.flags);
  };
}
```

:::

Save points are just object-layer markers that call `this.saveGame()` on overlap. Load on scene start so a refresh restores the player to the last save without losing inventory.
