# --interactive--

Scene-level pointer events tell you *something* happened, somewhere on the canvas. Most of the time you want a tighter contract: "the player tapped *this button*." For that you flip a sprite's interactive flag and listen on the sprite itself.

```js
const button = this.add.image(240, 320, "play-button");
button.setInteractive();
button.on("pointerdown", () => {
  this.scene.start("MainScene");
});
```

`setInteractive()` does two things behind the scenes:

1. Tells Phaser to hit-test pointer events against this object every frame.
2. Registers a default rectangular hit area matching the texture bounds.

Once an object is interactive, you get four extra events:

- `pointerover` ŌĆö pointer entered the hit area (mouse hover or first frame of a touch).
- `pointerout` ŌĆö pointer left the hit area.
- `pointerdown` / `pointerup` ŌĆö same as the scene-level events, but only when the pointer is over this object.

That makes hover effects almost free:

```js
button.on("pointerover", () => button.setTint(0xaaffaa));
button.on("pointerout", () => button.clearTint());
```

For non-rectangular sprites, pass a **hit area shape** so the cursor only "sees" the visible art:

```js
button.setInteractive(
  new Phaser.Geom.Circle(button.width / 2, button.height / 2, 32),
  Phaser.Geom.Circle.Contains
);
```

The first argument is the geometry; the second is the function Phaser uses to test "is this point inside this geometry?" For circles, polygons, and ellipses, Phaser ships matching `Contains` helpers.

Two performance traps worth knowing:

- Calling `setInteractive` on a sprite means Phaser hit-tests every pointer event against it. If you have a thousand decorative coins on screen and only the player can pick them up, *don't* mark the coins interactive ŌĆö let the physics overlap do that work.
- Per-object `pointerdown` runs on top of any scene-level `pointerdown`. If both your background and your buttons listen, the background listener still fires when the user clicks a button.

If you really want only the button's handler to run, return `event.stopPropagation()` from the per-object handler ŌĆö Phaser checks for it before invoking the scene-level chain.

# --questions--

## --text--

Why does calling `setInteractive()` on a `Phaser.GameObjects.Image` not require an explicit hit area?

## --answers--

Phaser falls back to a default rectangular hit area matching the texture bounds.

---

Phaser refuses to make the object interactive without one and throws.

### --feedback--

It accepts the call without complaint and uses a default rectangle.

---

Images are always interactive by default, so the call is a no-op.

### --feedback--

Images are *not* interactive by default; `setInteractive` is what turns them on.

---

Phaser uses pixel-perfect detection by default, which needs no hit area.

### --feedback--

Pixel-perfect detection is opt-in via `setInteractive({ pixelPerfect: true })` and is much more expensive.

## --text--

You want a hover effect on a circular play button so the cursor only registers when it's truly over the circle. Which call do you reach for?

## --answers--

`button.setInteractive(new Phaser.Geom.Circle(cx, cy, r), Phaser.Geom.Circle.Contains)`

---

`button.setInteractive({ shape: "circle" })`

### --feedback--

Phaser's API expects a geometry object plus a Contains callback, not a string.

---

`button.setInteractive(true, "circular")`

### --feedback--

That's not a Phaser signature. The second argument is a Contains function, not a string.

---

`button.setHitArea(Phaser.Geom.Circle)`

### --feedback--

There is no `setHitArea` method on game objects.
