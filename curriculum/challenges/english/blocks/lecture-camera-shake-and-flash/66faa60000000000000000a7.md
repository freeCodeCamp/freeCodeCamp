# --interactive--

"Juice" is the umbrella term for the small, non-essential effects that make a game feel responsive — screen shake on impact, a quick white flash when the player takes a hit, a colour fade between scenes, particles bursting on a kill. None of it is required to play the game, but a game without juice feels lifeless, and a game with too much juice feels exhausting.

Camera juice is the cheapest, most universal kind. The camera is already in every scene, so `cameras.main.shake`, `flash`, and `fade` cost almost nothing to add and immediately tighten the moment-to-moment feel of the game.

The rules of thumb that almost always hold:

1. **Tie juice to discrete events**, not continuous state. Shake on impact, not while damage is being dealt every frame. Flash on hit, not while at low health. The pattern that breaks your game is "every frame the player is on fire, shake by 1 pixel" — that's not juice, that's nausea.
2. **Keep durations short.** 100–300 ms is the sweet spot for shake and flash. Anything longer competes with the player's input — they cannot read the screen mid-effect.
3. **Match intensity to importance.** A small enemy dying = `shake(100, 0.005)`. The boss exploding = `shake(800, 0.04)`. A pickup = no shake at all, maybe a flash. If everything shakes the same amount, nothing reads as important.
4. **Provide a way to disable it.** Vestibular and visual sensitivities are real. Shipping a "reduce motion" toggle is now table stakes; the workshop in this module wires it in.

When juice **hurts**:

- During exploration. Shake while the player is just walking around feels like a bad joystick.
- For low-stakes events. Picking up a coin should not shake the screen; that exhausts the budget.
- In tutorials. The first 30 seconds of a game is when the player is calibrating; gratuitous juice during that window confuses them.
- Long, repeated shakes. Even at low intensity, screen shake accumulated across many enemies in a wave is the fastest path to motion sickness reports.

A useful test: if you remove the juice, does the game still read clearly? If the answer is "no, the player can't tell what hit them" — the shake is masking a missing visual signal, not amplifying an existing one. Add a sprite flash, an enemy stun frame, or particles first; let the shake be the cherry on top, not the whole sundae.

:::interactive_editor

```js
function juiceFor(event) {
  switch (event) {
    case "small-enemy-die":
      return { shakeMs: 100, intensity: 0.005, flashMs: 0 };
    case "player-hit":
      return { shakeMs: 200, intensity: 0.012, flashMs: 80 };
    case "boss-explode":
      return { shakeMs: 800, intensity: 0.04, flashMs: 200 };
    case "level-clear":
      return { shakeMs: 0, intensity: 0, flashMs: 400 };
    default:
      return { shakeMs: 0, intensity: 0, flashMs: 0 };
  }
}

console.log("walking:", juiceFor("walking"));
console.log("hit:    ", juiceFor("player-hit"));
console.log("boss:   ", juiceFor("boss-explode"));
```

:::

The pattern: short, discrete, scaled to importance, optional. Everything that violates one of those four properties is the candidate for the first round of "tone it down" feedback.

Which of these is the wrong place to add screen shake?

While the player walks across a flat floor, shake by 1 pixel for "weight."

When the player takes a hit, shake for 150 ms.

That is exactly the right place — a short, discrete event with stakes.

When the boss explodes, shake hard for 800 ms.

### --feedback--

That is a high-importance moment that deserves heavy juice.

On level clear, a 200 ms flash plus a small shake.

### --feedback--

A short flash on a discrete event is appropriate; small shake is fine if the team agrees the moment warrants it.

## --text--

You ship a "reduce motion" toggle. What should it do?

## --answers--

Disable shakes entirely; keep flashes and fades only when they convey gameplay information.

---

Reduce shake intensity by 50%.

### --feedback--

Half-strength is still motion. Players who turn the toggle on want zero unprompted screen movement.

---

Disable all visuals including HUD updates.

### --feedback--

Way too aggressive. The toggle is about non-essential motion, not gameplay readability.

---

Pause the game until the player toggles back.

### --feedback--

A toggle is a setting, not a pause condition.

You're juicing the moment a player picks up a coin. What's the right amount?

A small particle burst and a sound. No shake. Maybe a one-frame sprite flash.

---

A 200 ms shake at intensity `0.01` plus a 100 ms flash.

### --feedback--

Coins are low-stakes and frequent. Shaking on every pickup exhausts the player's juice budget within a minute.

A 600 ms fade to black so the pickup feels weighty.

### --feedback--

Fades are for transitions. A pickup is not a transition; cutting the camera black for over half a second is jarring.

### --feedback--

Too far the other way. Some feedback is needed; just not screen shake.
