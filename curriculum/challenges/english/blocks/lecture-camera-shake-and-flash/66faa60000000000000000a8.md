# --interactive--

`Phaser.Cameras.Scene2D.Camera#shake` is the simplest juice primitive. Two arguments — duration in milliseconds and intensity as a fraction of the camera viewport — and the camera jitters on its own for that long, then settles back where it started.

```js
this.cameras.main.shake(200, 0.012);
```

That shakes the main camera for 200 milliseconds. The `0.012` intensity is read as a fraction of `(width + height) / 2`. On a 320 × 240 canvas, `0.012 * 280 ≈ 3.4` pixels of peak displacement. On a 1920 × 1080 canvas, the same `0.012` produces `~18` pixels — visibly stronger.

That scaling is a feature, not a bug. The same intensity number reads the same way across screen sizes. But it means the **right** intensity depends on what feels right at your *target* canvas size; tune at the resolution you actually ship.

A starter palette for arcade games:

| Event              | Duration | Intensity |
| ------------------ | -------- | --------- |
| Bullet hit (small) | 80 ms    | 0.003     |
| Player damage      | 200 ms   | 0.012     |
| Big enemy death    | 350 ms   | 0.02      |
| Boss explosion     | 800 ms   | 0.04      |

These are starting points. Tune in-engine; numbers always feel different from imagination.

:::interactive_editor

```js
function compoundShake(events) {
  return events.reduce(
    (acc, e) => ({
      durationMs: Math.max(acc.durationMs, e.durationMs),
      intensity: Math.min(0.05, acc.intensity + e.intensity)
    }),
    { durationMs: 0, intensity: 0 }
  );
}

const burst = [
  { durationMs: 200, intensity: 0.012 },
  { durationMs: 200, intensity: 0.012 },
  { durationMs: 200, intensity: 0.012 }
];

console.log("three hits in same frame:", compoundShake(burst));
```

:::

The compound-shake function above shows a real production gotcha. If three damage events fire on the same frame, calling `shake(200, 0.012)` three times does **not** produce a single shake — Phaser overwrites the previous shake. The intensity does not stack naturally. Either accumulate yourself (clamp to a sane maximum) or have a single "dispatcher" that decides the strongest shake to apply each frame.

Two more API knobs:

```js
this.cameras.main.shake(200, 0.012, true);
```

The third argument is `force` — `true` restarts the shake even if one is already running. Without it, a new `shake` call is ignored when one is in flight. For deliberately chaining shakes you need `force: true`; for the "no compounding" property, leave it `false`.

```js
this.cameras.main.once("camerashakecomplete", () => {});
```

`camerashakecomplete` fires when the shake naturally ends. Use it to chain juice (shake → flash → particle burst) without timeouts, or to gate UI elements that should only show after the shake settles.

Common bug: the shake is jittering not just the player but everything that should be screen-fixed (HUD, dialogue boxes, score). Anything attached to the main camera shakes with it. Move HUD elements onto a second camera (Module 2 covers this) or use `setScrollFactor(0)` on the HUD object. Without one of those, your dialogue text rattles every time the boss explodes.

## --text--

`cameras.main.shake(200, 0.012)` — what does the `0.012` represent?

## --answers--

A fraction of `(width + height) / 2`, so the same number scales across canvas sizes.

---

A fraction of one second of duration.

### --feedback--

Duration is the first argument. The second is intensity.

---

The number of pixels of shake.

### --feedback--

It's a fraction; pixels depend on the canvas size.

---

The frequency in Hz.

### --feedback--

Phaser handles the frequency internally; you only set duration and intensity.

## --text--

Three damage events fire on the same frame and you naively call `shake(200, 0.012)` three times. What happens?

## --answers--

Each call overwrites the previous one, so you see one 200 ms shake at intensity `0.012`, not the compound shake you might expect.

---

The shakes stack and the intensity becomes `0.036`.

### --feedback--

They do not stack automatically. Phaser replaces the active shake.

---

The shakes queue and play sequentially.

### --feedback--

There is no built-in queue. To queue, you'd write your own dispatcher.

---

Phaser throws an error.

### --feedback--

It does not throw; it just replaces the active shake silently.
