# --interactive--

Where shake is for *impact*, **flash** and **fade** are for *transitions and signals*. Both paint the camera's viewport with a colour, but with very different timings and intents.

`flash(duration, r, g, b)` snaps the colour to full opacity instantly, then fades it out over `duration` milliseconds. It's a **brief, percussive overlay** for hits and pickups.

```js
this.cameras.main.flash(150, 255, 255, 255);
```

A 150-ms white flash, perfect for "the player just took damage." Common short flashes:

- White flash on damage — `flash(150, 255, 255, 255)`.
- Red flash on critical damage — `flash(200, 255, 0, 0)`.
- Yellow flash on pickup — `flash(100, 255, 240, 80)`.

`fade(duration, r, g, b)` is the *slow* version. Phaser smoothly ramps the colour up to full opacity over `duration` ms, and *holds* it (the camera stays masked until you fade back).

```js
this.cameras.main.fade(800, 0, 0, 0);
this.cameras.main.once("camerafadeoutcomplete", () => {
  this.scene.start("NextScene");
});
```

That fades to black over 800 ms, then starts the next scene. The `once("camerafadeoutcomplete")` handler is how you sequence the transition. To fade *back in*, the destination scene calls:

```js
this.cameras.main.fadeIn(500, 0, 0, 0);
```

`fadeIn` ramps the camera *out* of the colour mask. Without that call the camera stays black; only the new scene's `fadeIn` clears the overlay.

:::interactive_editor

```js
function describeFx(api, duration, color) {
  if (api === "flash") {
    return `paint ${color} at full opacity, fade it out over ${duration} ms`;
  }
  if (api === "fade") {
    return `paint ${color} at zero opacity, fade it in over ${duration} ms`;
  }
  return "no-op";
}

console.log(describeFx("flash", 150, "white"));
console.log(describeFx("fade", 800, "black"));
```

:::

Two events worth knowing:

- `camerafadeoutcomplete` — fires when `fade` reaches full opacity. Sequence the next step here.
- `camerafadeincomplete` — fires when `fadeIn` is fully back to normal. Useful if you want to enable input only after the transition finishes.

`flash` does not have completion events you usually need; if you do, `cameras.main.once("camerashakecomplete", …)` is the closest analogue, but the typical pattern is "fire and forget."

A common combo: damage feedback. Fire shake + flash on the same frame.

```js
this.cameras.main.shake(200, 0.012);
this.cameras.main.flash(120, 255, 0, 0);
```

The shake is the haptic; the flash is the colour signal. Together they read as "you got hit" with no need for a sprite tint. With either alone, the moment is visually weaker.

A trap: `fade` is asymmetric. The camera holds the colour after fade-out completes, so if you `fade` and then forget to `fadeIn`, the next scene appears under a black mask. The defensive pattern is to call `fadeIn` from the *new* scene's `create`, not from the old scene's transition handler.

# --questions--

## --text--

You want a brief white flash to signal damage. Which API call is right?

## --answers--

`cameras.main.flash(150, 255, 255, 255)`.

---

`cameras.main.fade(150, 255, 255, 255)`.

### --feedback--

`fade` ramps *into* the colour and *holds* it. You'd see the screen go white and stay white.

---

`cameras.main.shake(150, 0.012)`.

### --feedback--

That shakes the camera; it does not paint a colour overlay.

---

`cameras.main.setBackgroundColor("#fff")`.

### --feedback--

That changes the persistent background, not a transient effect.

## --text--

You call `cameras.main.fade(800, 0, 0, 0)` to transition to a new scene. The new scene loads, but the screen stays black. What's missing?

## --answers--

The new scene's `create` must call `cameras.main.fadeIn(duration, r, g, b)` to lift the mask.

---

`cameras.main.flash(800, 0, 0, 0)` to bookend the fade.

### --feedback--

`flash` is unrelated; it would briefly paint another colour.

---

`scene.events.emit("readyToPlay")`.

### --feedback--

There is no such event by default; `fadeIn` is the right call.

---

The new scene needs `cameras.main.setBackgroundColor("#000")`.

### --feedback--

The black is the camera's fade mask, not the background. Lifting the mask is the fix.
