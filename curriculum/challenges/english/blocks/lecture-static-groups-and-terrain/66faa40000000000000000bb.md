# --interactive--

A platformer needs floors, walls, and ceilings — bodies that the player collides with but that never themselves move. Arcade physics gives you a dedicated container for this: the **static group**, created with `physics.add.staticGroup()`.

A static body is a normal Arcade body with two switches flipped: `body.immovable = true` and `body.moves = false`. The physics engine never integrates velocity, never applies gravity, and never resolves penetration *on* the static body — only on whatever dynamic body collided with it. That makes static bodies extremely cheap: hundreds of them cost roughly nothing per frame.

```js
function create() {
  this.platforms = this.physics.add.staticGroup();
  this.platforms.create(80, 200, "tile");
  this.platforms.create(160, 200, "tile");
  this.platforms.create(240, 200, "tile");
}
```

:::interactive_editor

```js
function create() {
  this.platforms = this.physics.add.staticGroup();
  this.platforms.create(120, 180, "tile");
  this.platforms.create(200, 180, "tile");

  const player = this.physics.add.sprite(160, 60, "hero");
  player.setCollideWorldBounds(true);
  this.physics.add.collider(player, this.platforms);
}
```

:::

Two rules of thumb. First: **anything the level geometry needs goes in a static group**. Second: if a "platform" should move (elevators, conveyor belts, falling blocks), it is *not* static — use a regular dynamic body with `body.allowGravity = false` and `body.immovable = true` instead. That distinction comes back in step 21 of the workshop, when we add a moving platform.

The other thing static bodies don't do is change shape automatically. If you scale a static sprite after `staticGroup.create`, you have to call `body.refresh()` or the collision rectangle stays at the original size. New Phaser developers hit this constantly when they tile a sprite at 2× and watch the player phase through it.

# --questions--

## --text--

What two flags does a static body have set, compared to a dynamic body?

## --answers--

`body.immovable = true` and `body.moves = false`.

---

`body.allowGravity = false` and `body.bounce = 0`.

### --feedback--

Those are knobs you can tune on a dynamic body — they do not, on their own, make a body static.

---

`body.isStatic = true` and `body.kinematic = true`.

### --feedback--

Those property names do not exist in Arcade physics.

---

`body.fixed = true` and `body.frozen = true`.

### --feedback--

Invented names. Static bodies live behind `immovable` + `moves = false`.

## --text--

You scale a tile in a static group up to 2× after `staticGroup.create`. The player phases through it. Why?

## --answers--

The static body's collision rectangle was sized at the original scale and you forgot to call `body.refresh()`.

---

Static bodies do not collide with dynamic bodies; you need a dynamic group instead.

### --feedback--

They absolutely do collide — that is their whole purpose.

---

`physics.add.collider` does not accept static groups.

### --feedback--

It does. A collider between a dynamic body and a static group is the standard pattern.

---

The tile texture is not loaded yet.

### --feedback--

A missing texture would render the tile as a green-and-magenta debug box, not phase through it.

## --text--

Which of the following belongs in a static group?

## --answers--

The level's permanent floors and walls.

---

The player.

### --feedback--

The player is a dynamic body — it moves.

---

A spinning saw blade that travels back and forth.

### --feedback--

It moves, so it cannot be static. Use a dynamic body with `allowGravity = false` and `immovable = true`.

---

A coin pickup that floats and bobs up and down.

### --feedback--

Bobbing means it moves — dynamic body or a tweened image without physics.
