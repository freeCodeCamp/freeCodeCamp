# --interactive--

In 3D engines, what's nearer the camera occludes what's further away — the GPU resolves it for you with a depth buffer. 2D doesn't get that for free. Phaser draws every game object onto a flat plane, in order, like a painter laying brush strokes on canvas: each new stroke covers what was already there.

The order of those strokes is called the **display list**. By default, Phaser pushes new objects to the back of the list as you create them, which means the last `this.add.*` call paints on top of every earlier one.

That works until it doesn't. If you create a hero before its shadow, the shadow paints on top of the hero. If you create a UI element before a particle burst, the burst occludes your UI.

The lever you reach for is `setDepth`. Every game object has a numeric depth that overrides creation order:

```js
const bg = this.add.rectangle(240, 180, 480, 360, 0x223344);
const player = this.physics.add.sprite(240, 180, "hero", 0);
const ui = this.add.text(10, 10, "Score: 0", { fill: "#fff" });

bg.setDepth(0);
player.setDepth(10);
ui.setDepth(100);
```

Higher depth numbers paint later, which means they paint on top. The exact numbers don't matter; the relative ordering does. Most games use a small set of conventional depths — 0 for backgrounds, 10 for entities, 100 for UI — and leave gaps between them so future content can slot in without renumbering everything.

When two objects share a depth, Phaser falls back to creation order: whoever was added first paints first. That fallback is convenient for sprite groups (every coin in a coin group at depth 10 just sorts by spawn order), but it bites you when you create something later than you wanted.

The takeaway: never rely on creation order for anything that matters visually. The first thing every Phaser project should agree on is a depth scheme.

## --text--

You add a background image, then a hero sprite, then a HUD text — in that order, with no `setDepth` calls. What does the player see?

## --answers--

The HUD text on top, then the hero, then the background — because Phaser paints in the order you add objects.

---

The background on top, hiding the hero and the HUD.

### --feedback--

The background was added first, so it paints first — anything added later covers it.

---

All three at the same time, blended together.

### --feedback--

Phaser doesn't blend by default; later strokes occlude earlier ones.

---

The hero on top — sprites always render above text and shapes.

### --feedback--

There's no automatic priority by object type. Order alone decides.

You call `bg.setDepth(0)` and `hud.setDepth(100)`. What does that depth difference of 100 communicate to Phaser?

## --answers--

That `hud` should be drawn after `bg`. The exact numbers are arbitrary; only the order matters.

That `hud` should be drawn 100 frames after `bg`.

### --feedback--

Depth is a render-order key, not a time-based delay.

---

That `hud` should be 100 pixels closer to the camera.

### --feedback--

There is no "camera distance" in Phaser's 2D pipeline.

---

That `hud` should be 100% opaque and `bg` semi-transparent.

Depth has no effect on alpha or transparency.
