# --interactive--

You have two scenes. The controls scene knows when the player taps the fire button. The main scene knows how to fire. How do they talk?

The wrong answer is to give the controls scene a reference to the main scene's player and call `mainScene.player.fire()` directly. That couples the scenes — change the player class and the controls break — and it bypasses the abstraction that the rest of the chapter is built on.

The right answer is **events**. Each scene has its own event emitter at `this.events`, and any scene can read another scene's emitter via `this.scene.get(name).events`:

```js
class ControlsScene extends Phaser.Scene {
  create() {
    const fire = this.add.circle(240, 580, 50, 0xff6644, 0.6)
      .setInteractive();
    fire.on('pointerdown', () => {
      this.scene.get('MainScene').events.emit('action:fire');
    });
  }
}

class MainScene extends Phaser.Scene {
  create() {
    this.events.on('action:fire', () => {
      this.fireBullet();
    });
  }
}
```

Three principles to internalize:

- **Events name actions, not buttons.** Use `action:fire`, not `button:tap-bottom-right`. The naming should match the action map. That way you can swap a different controls scene in (e.g., a gamepad scene that emits the same event names) and the gameplay does not change.
- **Use a colon-prefixed namespace.** `action:fire`, `score:change`, `level:complete`. This avoids collisions with Phaser's built-in events (`update`, `shutdown`, etc.) and makes the source obvious to future readers.
- **Detach in `shutdown`.** When `MainScene` shuts down (level transition, game over), Phaser will clean up its own listeners, but if `MainScene` listens to events on `ControlsScene` it should `off` them in `shutdown`. Otherwise you'll leak listeners across scene restarts.

For "held" actions (left/right movement) you have two options:

```js
let leftHeld = false;
left.on('pointerdown', () => leftHeld = true);
left.on('pointerup', () => leftHeld = false);
left.on('pointerout', () => leftHeld = false);
this.events.on('update', () => {
  if (leftHeld) this.scene.get('MainScene').events.emit('action:left');
});
```

Or, more elegantly, expose the `held` state directly:

```js
this.scene.get('MainScene').data.set('virtualLeft', leftHeld);
```

…and read `mainScene.data.get('virtualLeft')` inside the action map. Both work; the event-emit-per-frame version is more explicit, the data-store version avoids per-frame allocations.

A subtle but important detail: don't forget `pointerout`. If the player slides their finger off the button without releasing, the `pointerup` may fire on the **scene** (not the button) and your `leftHeld` variable will be stuck on `true`. Listen for both `pointerup` _and_ `pointerout`, or hook the global `pointerup` on `this.input` and reset all held flags.

This event-bridge pattern works for any cross-scene coordination: HUD updates, pause toggles, level transitions. Once you internalize it, your scenes stay decoupled even as the game grows.

# --questions--

## --text--

Why route virtual button presses through events instead of calling `mainScene.player.fire()` directly?

## --answers--

To keep the scenes decoupled — gameplay code can change without breaking the controls scene, and vice versa.

---

Because `this.scene.get('MainScene')` is asynchronous.

### --feedback--

It returns the scene instance synchronously.

---

Because direct calls violate Phaser's scene isolation rules.

### --feedback--

There are no such rules. Direct calls are technically possible — they're just bad design.

---

Because the events queue is faster than function calls.

### --feedback--

It's not. The reason is structural decoupling, not performance.

## --text--

You attach `pointerdown` and `pointerup` to a "left" virtual button. Why also attach `pointerout`?

## --answers--

If the player slides their finger off the button without releasing, `pointerup` may not fire on the button — `pointerout` ensures the held state resets.

---

Because `pointerup` doesn't fire on touch devices.

### --feedback--

It does — touch raises `pointerup` correctly.

---

Because `pointerout` is required to detect button release.

### --feedback--

`pointerup` is the canonical release event. `pointerout` is a backstop for the slide-off case.

---

Because Phaser deprecated `pointerup` in 3.80.

### --feedback--

It is not deprecated.

## --text--

What's the recommended naming convention for cross-scene events?

## --answers--

Colon-prefixed, action-named: `action:fire`, `score:change`, `level:complete`.

---

Phaser-style camelCase: `actionFire`.

### --feedback--

That collides too easily with Phaser's built-in events. Namespace with `:` instead.

---

Match the button's position: `button:bottom-right`.

### --feedback--

Names should describe the action, not the physical button. Otherwise rebinding breaks the listeners.

---

Use UPPER_SNAKE_CASE constants.

### --feedback--

That's a stylistic preference, not a semantic one. The colon-prefixed namespace is the load-bearing rule.
