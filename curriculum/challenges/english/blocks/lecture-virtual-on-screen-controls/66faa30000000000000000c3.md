# --interactive--

Most "mobile-friendly" web games are desktop games with a control hint that reads "use a keyboard." This lecture is about why that fails and how to actually design for two input devices at the same time.

The core idea is the same one from the previous block: **gameplay code should care about actions, not input devices**. If your update loop reads "is `left` active?" then it does not matter whether `left` is being driven by an arrow key, a `pointermove` event, an on-screen button, or a swipe gesture. As long as something somewhere sets the `left` flag, the gameplay is identical.

Concretely, your input pipeline becomes a small fan-in. The keyboard plugin, the pointer event stream, the on-screen virtual buttons, and the swipe detector all write into the same `actions` table; gameplay reads from `actions` and never asks which device wrote there. There is no `if (mobile)` branch in `update`. Ever.

This pays off in practice in three ways:

- **Testing is easier.** You can unit-test gameplay by directly setting `actions.left = true`. No fake DOM events.
- **Input feels consistent.** A swipe and an arrow key both feed the same dead-zone, the same lerp, the same response curve. Players feel the game's "feel" the same way on every device.
- **Adding a new device is one file.** Want to support gamepads? Add a fourth resolver. Gameplay doesn't change.

There are, however, three places where you _do_ branch on device:

1. **Layout.** A virtual D-pad takes 200×200 px in the bottom-left corner; that real estate doesn't exist on desktop. So the layout differs even though the actions are the same.
2. **Hints.** The "Press SPACE to fire" tutorial line should read "Tap to fire" on touch. Detect via `Phaser.Device.os.android || Phaser.Device.os.iOS`.
3. **Tuning.** Touch tends to need slightly more lerp smoothing; keyboard wants snappier acceleration. Two tuning constants is fine — two `update` bodies is not.

A common anti-pattern: writing a desktop game first, then "porting" it to touch by adding a virtual D-pad on top. That works, but it almost always feels worse than designing both modes simultaneously. Touch input has much less pixel precision than a mouse; if your game requires sub-pixel paddle alignment to clear a level, the touch experience will be miserable. Design the difficulty around the **least precise** input you intend to support.

The workshop in this module uses this exact pipeline: keyboard, pointer, and on-screen buttons all funnel into the same action set, and gameplay never asks which device the player is using.

A player swipes left on touch, then immediately presses LEFT on the keyboard. What does the action map do?

## --answers--

Both inputs feed the same `actions.left` flag — the action stays active as long as either device says so.

---

The keyboard input wins; the swipe is canceled.

### --feedback--

There's no priority by default. Both feed the same flag.

---

You get a "conflict" event you have to handle.

### --feedback--

Made up. The action map just OR-s its inputs.

---

Phaser throws because two devices cannot drive the same action.

### --feedback--

Phaser doesn't track action ownership. Your code does, and OR is the natural composition.
