# --interactive--

Once you decide to ship virtual buttons, the next decision is _where_ they live. The cleanest option is to split your game into two scenes: a `MainScene` that owns the gameplay (paddle, ball, bricks) and a `ControlsScene` that owns the on-screen UI (D-pad, fire button, pause). They run **in parallel**, not in sequence — both are active simultaneously, drawing into the same canvas at different depths.

```js
class MainScene extends Phaser.Scene {
  constructor() {
    super('MainScene');
  }
  create() {
    // gameplay lives here.
    this.scene.launch('ControlsScene');
  }
}

class ControlsScene extends Phaser.Scene {
  constructor() {
    super('ControlsScene');
  }
  create() {
    this.add.rectangle(60, 580, 80, 80, 0x556677, 0.4).setInteractive();
    this.add.rectangle(420, 580, 80, 80, 0x556677, 0.4).setInteractive();
  }
}

const config = {
  // …
  scene: [MainScene, ControlsScene]
};
```

Why split scenes instead of just adding the buttons as part of `MainScene`? Three reasons:

- **Independent lifecycle.** When you `pause` the gameplay, you do **not** want to pause the pause button. Two scenes give you two pause states.
- **Camera independence.** A scrolling game's camera moves the world; the controls should not move with it. Two scenes mean two cameras, and the controls scene's camera stays put.
- **Layering.** The controls scene renders **on top** of the gameplay scene because it's listed second in the `scene` array. You don't have to fight `setDepth` on every UI sprite.

`launch` (not `start`) is the key word. `start` stops the current scene; `launch` runs the new scene **in parallel**. There's also `run`, which is essentially `launch` with an explicit "if already running, restart it" flag.

A common starter template for the controls scene:

```js
create() {
  // hide the entire scene on desktop
  if (!Phaser.Device.os.android && !Phaser.Device.os.iOS) return;

  const left = this.add.rectangle(60, 580, 80, 80, 0x556677, 0.4)
    .setInteractive();
  const right = this.add.rectangle(420, 580, 80, 80, 0x556677, 0.4)
    .setInteractive();
  const fire = this.add.circle(240, 580, 50, 0xff6644, 0.6)
    .setInteractive();

  this.add.text(60, 580, '◀', { fontSize: '32px' }).setOrigin(0.5);
  this.add.text(420, 580, '▶', { fontSize: '32px' }).setOrigin(0.5);
}
```

A few practical notes:

- `setInteractive()` accepts a hit-area shape. For triangular D-pad buttons, pass `new Phaser.Geom.Triangle(...)` and `Phaser.Geom.Triangle.Contains` as the shape callback.
- Keep buttons **big** — the iOS Human Interface Guidelines suggest 44 pt minimum, but for game buttons 80 px or more is comfortable. Smaller and players will miss them.
- Use **transparency**. A solid button at the bottom of the screen draws too much attention; opacity 0.4–0.6 is a sweet spot.
- The "pause" button often deserves to be in the top-right, away from the action buttons, because misclicking it during gameplay is catastrophic.

## --text--

Why use `this.scene.launch('ControlsScene')` instead of `this.scene.start('ControlsScene')`?

## --answers--

`launch` runs the new scene in parallel with the current one. `start` would stop `MainScene`.

`launch` is faster.

Performance is the same. The semantic difference is parallel-vs-replace.

### --feedback--

That's not the relevant difference. Both can skip a loading screen.

## --text--

What's the practical reason to put on-screen controls in a separate scene rather than as part of the gameplay scene?

## --answers--

Independent pause/resume, independent camera, and predictable rendering depth without `setDepth` everywhere.

---

Phaser doesn't allow input objects in the same scene as physics objects.

### --feedback--

It does. The split is a design choice, not a constraint.

---

Performance — two scenes render faster than one.

### --feedback--

Two scenes are slightly slower, not faster. The benefit is structural.

---

It's required to use `setInteractive`.

### --feedback--

`setInteractive` works in any scene.
