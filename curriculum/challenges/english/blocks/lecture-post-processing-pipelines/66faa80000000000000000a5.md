# --interactive--

Phaser 3.60+ ships fifteen-ish built-in post-FX pipelines, all callable via `gameObject.postFX.addX(...)` or `camera.postFX.addX(...)`. You don't write GLSL — you tweak parameters.

The ones you'll reach for in match-3 and most arcade games:

| Method                                                              | What it does                                                                                                       | Typical use                                                            |
| ------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------- |
| `addBloom(color, offsetX, offsetY, blurStrength, strength, steps)`  | Extracts bright pixels and blurs them outward, adding a halo.                                                      | Special-gem creation, win screen.                                      |
| `addBlur(quality, x, y, strength, color, steps)`                    | Box-blurs a region. `quality: 0` is cheapest.                                                                      | Pause menu background, victory blur.                                   |
| `addGlow(color, outerStrength, innerStrength, knockout, distance, quality)` | Soft outer/inner glow around the alpha edge.                                                            | Hint pulses, hovered gems, special-gem aura.                           |
| `addPixelate(amount)`                                               | Snaps pixels to an `amount × amount` block.                                                                        | Stylized retro look, scene transitions.                                |
| `addVignette(x, y, radius, strength)`                               | Darkens corners of a region toward black, focusing the eye on the playfield center.                                | Permanent playfield fence.                                             |
| `addShine(speed, lineWidth, gradient, reveal)`                      | A diagonal animated highlight band sweeping across.                                                                | "New high score!" banners.                                             |
| `addBarrel(amount)`                                                 | Convex barrel distortion. Negative `amount` = pinch.                                                               | CRT screens, fish-eye effects.                                         |
| `addTiltShift(radius, amount, contrast, blurX, blurY, strength)`    | Selective focus simulating a tilt-shift lens.                                                                      | Cinematic intros.                                                      |
| `addCircle(thickness, color, backgroundColor, scale, feather)`      | Renders a circular vignette mask.                                                                                  | Scene reveal/hide transitions.                                         |
| `addReveal(width, color, alpha)`                                    | Linear gradient mask that scrolls across.                                                                          | Wipe transitions.                                                      |

Three patterns you'll write in the workshop:

**Permanent vignette on the playfield camera:**

```js
this.cameras.main.postFX.addVignette(0.5, 0.5, 0.6, 0.4);
```

`(x, y)` is the center in normalized `[0,1]` space, `radius` the falloff, `strength` how dark the corners get.

**Bloom on special-gem creation:**

```js
const stripe = this.spawnSpecialGem(...);
const bloom = stripe.postFX.addBloom(0xffffff, 1, 1, 0.5, 1.5, 6);
this.tweens.add({
  targets: bloom,
  strength: { from: 1.5, to: 0 },
  duration: 800,
  onComplete: () => stripe.postFX.remove(bloom)
});
```

The handle returned by `addBloom` exposes `strength`, `steps`, and other live parameters you can tween. Always remove the FX with `postFX.remove(handle)` when the animation ends — leaving it on costs every frame after.

**Blur backdrop for pause menu:**

```js
const blur = this.cameras.main.postFX.addBlur(0, 2, 2, 1, 0xffffff, 4);
// later, on resume:
this.cameras.main.postFX.remove(blur);
```

`quality: 0` is single-pass and cheapest; `quality: 2` is multi-pass and noticeably softer.

A common trap: `addX` returns a *handle* (a Phaser FX controller), not the camera. So you can't chain — `cam.postFX.addBloom(...).addVignette(...)` will throw because `addBloom` returned a Bloom object, not the FX manager.

Right way:

```js
const cam = this.cameras.main;
cam.postFX.addBloom(0xffffff, 1, 1, 0.5, 1.5);
cam.postFX.addVignette(0.5, 0.5, 0.7, 0.5);
```

:::interactive_editor

```js
function pickPostFX(intent) {
  const map = {
    "focus the eye on the playfield":        "addVignette",
    "make a special gem feel powerful":      "addBloom",
    "blur the background of a pause menu":   "addBlur",
    "add a soft outline to a hovered gem":   "addGlow",
    "stylize the entire game retro":         "addPixelate",
    "celebrate a high score with a shine":   "addShine"
  };
  return map[intent] ?? "no built-in covers this; write a custom shader";
}

const intents = [
  "focus the eye on the playfield",
  "make a special gem feel powerful",
  "blur the background of a pause menu",
  "add a soft outline to a hovered gem",
  "make every sprite invert color"
];
intents.forEach((i) => console.log(i, "→", pickPostFX(i)));
```

:::

The "every sprite invert color" intent has no stock pipeline — that's exactly the gap a custom `PostFXPipeline` subclass fills. For everything in the table above, `addX` covers it without writing GLSL.

A subtle gotcha with `addBloom` parameters: `strength` of `1` is sometimes hard to see, `2` is bordering on overdone. Tween from `2 → 0` over 800ms for a "burst then fade" feel — that's the `bloom on special-gem creation` recipe in step 31 of the workshop.

## --text--

What does `addBloom` actually return?

## --answers--

A `Phaser.FX.Bloom` controller (a handle) that exposes the live parameters of that bloom effect (`color`, `strength`, `steps`, `offsetX`, `offsetY`). You can tween its `strength` to fade the bloom in/out, and call `postFX.remove(handle)` to dispose of it.

---

The camera or game object it was applied to.

### --feedback--

That'd let you chain `addX` calls, which is exactly what doesn't work — `addX` returns the controller, not the FX manager.

---

A boolean indicating success.

### --feedback--

It returns the controller object.

---

A promise that resolves when the bloom finishes.

### --feedback--

Bloom is continuous, not animated by default. It runs as long as it's attached.

## --text--

You add a Bloom for a brief 800ms flash on a special gem and then... never remove it. What happens?

## --answers--

The bloom keeps running every frame for the rest of the gem's life. Cost compounds across many gems — eventually frame rate drops. Always pair `add*` with `postFX.remove(handle)` when the animation ends, or destroy the whole gem (which removes the FX too).

---

Phaser garbage-collects unused FX after a few seconds.

### --feedback--

There's no auto-collector for post-FX. They're attached until you remove them or destroy the host.

---

Multiple bloom effects merge into one.

### --feedback--

Each `addBloom` is independent and additive — you can stack them, but each costs.

---

The bloom auto-fades to zero strength after `duration: 800`.

### --feedback--

`addBloom` has no duration — it's a continuous effect. The 800ms came from your tween, not the FX itself.

## --text--

Why does `cam.postFX.addBloom(...).addVignette(...)` throw?

## --answers--

`addBloom` returns the Bloom controller, not the FX manager. The Bloom object has no `addVignette` method, so the chained call hits an undefined property and throws.

---

Phaser only allows one post-FX per camera.

### --feedback--

You can stack many — just not via chaining.

---

`addVignette` requires a sprite, not a camera.

### --feedback--

Cameras and sprites both expose `postFX` with the same `addX` methods.

---

Bloom must finish before vignette starts.

### --feedback--

Both run continuously and can coexist on the same camera. The chain bug is unrelated to ordering.
