# --interactive--

Post-FX is a fragment-shader pass over the framebuffer. On a flagship phone GPU at 1080p, you can stack two or three full-screen passes and stay at 60 FPS. On a budget device at the same resolution, the same stack may drop to 35 FPS. Mobile budgets matter.

The cost equation, in rough order of dominance:

| Factor                       | Impact on cost                                                    |
| ---------------------------- | ----------------------------------------------------------------- |
| **Framebuffer pixel count**  | Linear. 1080p costs 4× the pixels of 540p.                        |
| **Number of passes**         | Linear. Bloom (2 passes) costs ~2× a vignette (1 pass).           |
| **Render-target allocations**| Each pipeline allocates at least one full-screen texture in VRAM. |
| **Texture sample count**     | Box blur with `quality: 2` samples the framebuffer 9 times per output pixel; `quality: 0` samples once. |
| **Effect chaining**          | Two stacked pipelines re-read the same framebuffer twice.         |

Per-sprite vs per-camera matters more than people think:

- **Per-sprite post-FX** runs on the sprite's bounding rectangle. A 64×64 gem with a glow costs ~4096 pixels per pass.
- **Per-camera post-FX** runs on the camera's render target. A 1080p camera with a glow costs ~2,073,600 pixels per pass.

Three orders of magnitude difference. Apply post-FX to *the smallest game object that needs it* — a special gem, the score banner — not the whole camera, unless the effect is truly global (vignette, color grading).

A practical decision tree for mobile:

1. Does the effect need to apply to the whole frame? If yes, use camera. If no, use the smallest game object.
2. Is the effect *animated*? Tween the controller's parameters; never re-create the effect each frame.
3. Is the effect *transient* (lasts < 1s)? `addX(...)` then `postFX.remove(handle)` when the tween finishes.
4. Is the effect *permanent* (vignette, retro pixelate)? `addX(...)` once at scene start, leave it.
5. Are you stacking >2 camera-level pipelines? Profile on the lowest-spec device you target. Cut what you can't justify.

A "cheap" stack:

- `addVignette(0.5, 0.5, 0.6, 0.4)` on the main camera — one pass, 1 mip.

A "moderate" stack:

- Vignette + per-special-gem bloom (transient).

A "wreck-mobile" stack:

- Vignette + full-camera bloom + full-camera blur (`quality: 2`) + per-gem glow on every gem.


`prefers-reduced-motion` is *not* the same as low-end-device — but you can use it as a hint for "user wants a simpler experience" and skip non-essential post-FX:

```js
const reduce = window.matchMedia?.("(prefers-reduced-motion: reduce)").matches;
if (!reduce) {
  this.cameras.main.postFX.addBloom(0xffffff, 1, 1, 0.5, 1.0);
}
this.cameras.main.postFX.addVignette(0.5, 0.5, 0.7, 0.4);
```

The vignette stays (it's free); the bloom only fires if the user hasn't asked for less motion. This is a generous default.

For real device budgeting, expose a `lowFx` toggle in your settings menu and gate optional pipelines on it. Save the choice to `localStorage`. Players on weak hardware will appreciate the option more than they appreciate the bloom.

:::interactive_editor

```js
const PASSES = {
  vignette: 1,
  bloom: 2,
  blur: 2,
  glow: 2,
  pixelate: 1
};

function frameCost(width, height, effects, perGem = 0) {
  const cameraCost = effects.reduce(
    (sum, e) => sum + (PASSES[e] ?? 0) * width * height, 0
  );
  const gemCost = perGem * 64 * 64 * (PASSES["glow"] ?? 0);
  return {
    cameraOps:    cameraCost,
    perGemOps:    gemCost,
    totalOps:     cameraCost + gemCost,
    fpsBudgetMs:  16.7,
    estMs:        ((cameraCost + gemCost) / 25_000_000).toFixed(2)
  };
}

console.log("light",     frameCost(1080, 1920, ["vignette"]));
console.log("moderate",  frameCost(1080, 1920, ["vignette", "bloom"]));
console.log("heavy",     frameCost(1080, 1920, ["vignette", "bloom", "blur"], 64));
```

:::

The "heavy" config above runs ~33M shader invocations per frame. At 25M ops/ms (a midrange mobile GPU), that's ~1.3ms — *just* the post-FX, before sprites, physics, and game logic. A 16.7ms budget at 60 FPS leaves you ~15ms for everything else, which is fine. Bump that to 1080p with two more pipelines and you're cooked.

# --questions--

## --text--

You're shipping match-3 to mobile. Which post-FX placement strategy gives the best feel-to-cost ratio?

## --answers--

A permanent low-strength `addVignette` on the main camera (cheap, one pass, frames the playfield), plus transient `addBloom` on individual special gems via `postFX.remove(handle)` when their birth animation finishes. Both stay tiny — ~2.07M ops camera + ~8K ops per gem.

---

Stack vignette, bloom, and blur on the camera permanently.

### --feedback--

Three full-screen pipelines = ~5 passes × 1080p ≈ 10M ops/frame just for post-FX. On budget mobile that's a frame-rate problem.

---

Skip post-FX entirely on mobile.

### --feedback--

A single vignette is essentially free and dramatically improves the feel. Skipping all FX is over-correcting.

---

Apply bloom to every sprite, even base gems.

### --feedback--

64 gems × bloom × 2 passes is per-sprite cost, but it adds up; and on idle gems the bloom does nothing useful. Reserve bloom for moments that matter.

## --text--

Why does per-camera bloom cost ~500× more than per-sprite bloom on a single 64×64 gem?

## --answers--

Per-camera bloom samples every pixel of the framebuffer (1080p ≈ 2M pixels). Per-sprite bloom samples only the sprite's bounding box (64×64 = 4K pixels). Both use the same shader — the difference is the area being shaded. Roughly 2M / 4K ≈ 500.

---

Per-camera passes are slower per pixel.

### --feedback--

The shader runs at the same speed per pixel. Difference is pixel count.

---

Per-sprite blooms skip frames.

### --feedback--

Both pipelines run every render frame.

---

Per-camera bloom requires double-buffering.

### --feedback--

Both pipelines render to a texture; the buffering pattern is the same.

## --text--

A user has `prefers-reduced-motion` enabled. Which post-FX strategy best honors that hint?

## --answers--

Skip optional, high-energy effects (full-screen bloom, screen shake, elastic tweens) but keep low-energy contextual effects like a permanent vignette. The user wants less stimulation, not zero feedback.

---

Disable all post-FX.

### --feedback--

Vignettes and color framing aren't motion — keeping them is fine. Disabling everything is over-correcting.

---

Reduce frame rate to 30 FPS.

### --feedback--

Frame rate is independent of the user's preference; cutting it would just make the game feel worse.

---

Replace post-FX with screenshots.

### --feedback--

Static art breaks the game. The right answer is to pare back the *flashy* FX, not the gameplay rendering.
