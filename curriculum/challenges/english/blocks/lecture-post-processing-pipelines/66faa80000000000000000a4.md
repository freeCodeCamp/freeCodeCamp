# --interactive--

When Phaser draws a sprite, it asks the GPU: *"here's a quad, here's a texture, sample the texture and write each pixel to the screen."* That program — the recipe the GPU runs per pixel — is a **fragment shader**, and a **pipeline** is the wrapper Phaser puts around it.

Three pipeline categories matter:

| Pipeline kind   | When it runs                                      | What you'd write in it                                |
| --------------- | ------------------------------------------------- | ----------------------------------------------------- |
| **Sprite**      | While drawing each game object's quad.            | Custom per-sprite color math (rare in 99% of games).  |
| **Multi**       | Default sprite path. Batches lots of quads.       | Already written for you. Don't replace it.            |
| **Post-FX**     | After the scene/object is drawn, on a render target. | Bloom, blur, vignette, glow, color grading.        |

A **post-FX pipeline** takes the already-rendered pixels of a camera, scene, or game object, runs them through one more fragment-shader pass, and writes the result to the screen. That second pass is where `bloom`, `blur`, `vignette`, and friends live.

Conceptually:

```bash
[ scene render ]  →  [ render target ]  →  [ post-FX shader pass ]  →  [ screen ]
                          (texture)             (one or more passes)
```

A "pass" is one shader execution sampling the entire frame. A bloom effect is typically two passes: extract bright pixels, then blur and combine. Each pass costs roughly one screen-sized texture sample per pixel, so the cost scales with framebuffer area, not sprite count.

In Phaser 3.60+, the easiest entry point is the `postFX` accessor on game objects and cameras:

```js
this.cameras.main.postFX.addBloom(0xffffff, 1, 1, 1, 1);
this.gem.postFX.addGlow(0xffff00, 6, 0, false, 0.1, 16);
```

The first line slaps a bloom on every pixel the main camera renders. The second adds a yellow glow to one specific gem — all other sprites render unchanged.

You can also work with the underlying class directly for a custom shader:

```js
class MyPipeline extends Phaser.Renderer.WebGL.Pipelines.PostFXPipeline {
  constructor(game) {
    super({
      game,
      fragShader: "/* GLSL here */"
    });
  }
}
this.game.renderer.pipelines.addPostPipeline("MyPipeline", MyPipeline);
this.cameras.main.setPostPipeline("MyPipeline");
```

That route is for grain/CRT/distortion effects you can't build from the stock toolkit. For 95% of games, the built-in `postFX.addX` shortcuts cover the use case.

:::interactive_editor

```js
function passCount(effect) {
  const passes = {
    "bloom":     2,
    "blur":      2,
    "glow":      2,
    "vignette":  1,
    "pixelate":  1,
    "shine":     1,
    "tiltShift": 2
  };
  return passes[effect] ?? 0;
}

function totalCost(width, height, effects) {
  const pixels = width * height;
  let totalPasses = 0;
  effects.forEach((e) => totalPasses += passCount(e));
  return {
    pixels,
    totalPasses,
    pixelOps: pixels * totalPasses
  };
}

console.log(totalCost(1920, 1080, ["vignette"]));
console.log(totalCost(1920, 1080, ["vignette", "bloom"]));
console.log(totalCost(1920, 1080, ["vignette", "bloom", "blur", "glow"]));
```

:::

A vignette alone reads ~2.07M pixels on 1080p (one pass × the frame). Add bloom and you're at ~6.2M ops. Stack four effects and you're at ~14.5M per frame — at 60 FPS that's ~870M shader invocations per second. On a flagship phone GPU that's fine; on a budget laptop it can drop you below 60 FPS.

The price is *per-pixel × passes*. A vignette is cheap because it's one pass. Bloom is roughly twice as expensive. Sticking three or four pipelines on the camera at once is where mobile budgets get blown.

# --questions--

## --text--

What does a "pass" cost in a post-FX pipeline?

## --answers--

Roughly one shader invocation per output pixel of the framebuffer — the cost scales with `width × height` of the render target, not with the number of sprites in the scene.

---

One invocation per sprite.

### --feedback--

Sprite count doesn't enter the equation. The pass operates on the *rendered* framebuffer, regardless of how many sprites contributed to it.

---

Cost depends on the texture format only.

### --feedback--

Format matters for bandwidth, but the per-pixel work is the dominant cost.

---

Browsers cap pipeline cost at 1ms.

### --feedback--

There's no such cap; the GPU just runs slower if you ask too much.

## --text--

Why does Phaser ship `MultiPipeline` as the default for sprite rendering?

## --answers--

It batches many sprite quads into a single draw call (up to a per-batch cap), so 100 sprites cost roughly one GPU draw call instead of 100. That batching is the win — replacing it with a custom sprite pipeline usually loses the batch benefit and tanks performance.

---

It looks better than alternatives.

### --feedback--

Default rendering looks identical regardless of pipeline; the difference is throughput.

---

It supports more textures.

### --feedback--

Texture count is bound by hardware, not pipeline.

---

It's the only pipeline that supports physics.

### --feedback--

Pipelines and physics are independent systems.

## --text--

You apply `addBloom` to a single gem game object. What changes versus applying it to the entire camera?

## --answers--

The bloom shader runs only on the bounding region of that one gem, not on the whole framebuffer. Cost stays small (a tiny render target instead of a full-screen one), and surrounding sprites render at zero post-FX cost.

---

The bloom is brighter on a single sprite.

### --feedback--

Brightness is a parameter of the effect, independent of where it's applied.

---

Per-sprite post-FX is unsupported in 3.60+.

### --feedback--

Phaser 3.60+ explicitly added `gameObject.postFX.addX()` for this.

---

The camera version skips frames; the per-sprite version doesn't.

### --feedback--

Neither skips frames; both run every render tick.
