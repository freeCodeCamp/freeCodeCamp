# --interactive--

Lerp smooths out the camera's response to fast moves, but the camera still moves for *any* movement — even when the player just shuffles a few pixels left and right. In a platformer or top-down shooter that twitchy "always-tracking" feel is fatiguing.

The fix is a **deadzone**: a rectangle, centred on the camera, where the player can move freely without nudging the camera at all. Only when the player crosses an edge of the rectangle does the camera start scrolling, just enough to pull the player back inside.

```js
this.cameras.main.setDeadzone(200, 100);
```

That is a 200 × 100 pixel rectangle. While the player stays inside it, the camera does not move. Once the player walks past the right edge, the camera scrolls right just enough to keep the player on the rectangle's right edge.

:::interactive_editor

```js
const camera = { x: 200, y: 100 };
const deadzone = { width: 100, height: 80 };
const player = { x: 240, y: 130 };

function update() {
  const halfW = deadzone.width / 2;
  const halfH = deadzone.height / 2;
  if (player.x > camera.x + halfW) camera.x = player.x - halfW;
  if (player.x < camera.x - halfW) camera.x = player.x + halfW;
  if (player.y > camera.y + halfH) camera.y = player.y - halfH;
  if (player.y < camera.y - halfH) camera.y = player.y + halfH;
}

console.log("before:", { ...camera }, "player:", { ...player });
update();
console.log("after :", { ...camera });

player.x = 320;
update();
console.log("player past edge:", { ...camera });
```

:::

The first update doesn't move the camera — the player is inside the deadzone. The second update slides the camera just enough to put the player on the deadzone's right edge.

Deadzones combine cleanly with lerp. You set both:

```js
this.cameras.main.startFollow(this.player, true, 0.1, 0.1);
this.cameras.main.setDeadzone(200, 100);
```

The deadzone defines *when* the camera reacts; lerp defines *how* it reacts. Inside the deadzone, lerp does nothing because the camera has no target to chase. Crossing the edge gives lerp a target and it smoothly catches up.

Sizing is the only hard part:

- A small deadzone (50 × 50) feels almost like no deadzone — every step nudges the camera.
- A wide deadzone (400 × 200) feels stable for exploration but means the player can run a long way before the camera reacts, which makes them feel like they're outrunning the screen.
- Tune the deadzone visually — Phaser exposes `cameras.main.deadzone` so you can render a `Graphics` rectangle in dev mode to see exactly where the boundary sits.

To turn the deadzone off, call `setDeadzone()` with no arguments, or `setDeadzone(0, 0)`.

# --questions--

## --text--

What is a camera deadzone, in one sentence?

## --answers--

A rectangle centred on the camera; the camera does not move while the followed target stays inside it.

---

A region the player cannot enter.

### --feedback--

Deadzones do not block the player; they only affect when the camera reacts.

---

A camera mode that freezes for the duration of a level transition.

### --feedback--

That is a different feature (`stopFollow` / scene fade). The deadzone is per-frame movement filtering.

---

A graphical filter that fades the screen edges to black.

### --feedback--

That is vignette. Deadzones are about camera motion, not rendering.

## --text--

You want a slightly stable feel for exploration but the camera should still react when the player breaks into a run. What's the best knob mix?

## --answers--

A modest deadzone like `setDeadzone(150, 80)` plus a small lerp like `startFollow(target, true, 0.1, 0.1)` — the deadzone soaks up small wandering movements, lerp smooths the catch-up when the player accelerates past the edge.

---

A huge deadzone like `setDeadzone(800, 600)` and no lerp.

### --feedback--

The camera will only move when the player nearly leaves the screen — that feels like the player is outrunning the camera, which is exactly the disorienting experience deadzones are meant to prevent.

---

`setDeadzone(0, 0)` and `lerp: 0.5`.

### --feedback--

A zero deadzone means every footstep nudges the camera — the twitchy feel deadzones exist to fix.

---

No deadzone and no lerp.

### --feedback--

That gives you exact tracking, which is the rigid baseline lerp and deadzones improve on.
