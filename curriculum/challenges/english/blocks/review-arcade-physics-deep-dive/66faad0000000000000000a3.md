## Gravity, velocity, and drag

Arcade physics gives every dynamic body three forces you can dial. **World gravity** is set on the game `config` and applied to every body that opts in. **Per-body gravity** stacks on top via `body.gravity.x` / `body.gravity.y` and lets one body live by different rules than the rest of the scene. **Drag** scrubs energy from the body each frame; it slows whatever the body is doing, regardless of direction.

:::interactive_editor

```js
function create() {
  this.physics.world.gravity.y = 600;
  const ball = this.add.circle(120, 40, 8, 0xffaa44);
  this.physics.add.existing(ball);
  ball.body.setBounce(0.6);
  ball.body.setDrag(40, 0);
  ball.body.setVelocity(120, 0);
  this.add.rectangle(120, 170, 240, 8, 0x66ddff).body =
    this.physics.add.existing(this.add.rectangle(120, 170, 240, 8, 0x66ddff), true);
}
```

:::

The bounce coefficient `body.bounce` is the **restitution** of the collision: `1` is perfectly elastic (energy preserved), `0` is perfectly inelastic (the body sticks), anything in-between dampens the bounce by that fraction. Combine drag and bounce for the bounce-then-rest sandbox you built in `workshop-bouncing-ball-lab` step 6 — gravity pulls the ball down, bounce returns it (now slower), drag scrubs horizontal velocity, ball comes to rest.

## Colliders vs overlaps vs process callbacks

A **collider** (`physics.add.collider`) is hard contact: Phaser separates the bodies and resolves penetration. Use it for floors, walls, paddles, ball-vs-brick, anything where the bodies should not pass through each other.

An **overlap** (`physics.add.overlap`) is a trigger: the callback fires while the bodies intersect, but Phaser does not separate them. Use it for coin pickups, damage zones, level-complete flags — anything where you want detection without resolution.

The **process callback** (the 4th argument to `collider`/`overlap`) is a yes/no filter that runs *before* Phaser resolves the contact. Return `true` to let the contact happen, `false` to skip it. That's how you build one-way platforms, friend-vs-foe filtering, or "only damage the player when invincibility is over":

:::interactive_editor

```js
function create() {
  this.player = this.add.rectangle(60, 80, 16, 16, 0xffffff);
  this.physics.add.existing(this.player);
  this.player.body.setVelocityY(180);
  this.platform = this.add.rectangle(120, 140, 100, 6, 0x88ffaa);
  this.physics.add.existing(this.platform, true);
  this.physics.add.collider(
    this.player,
    this.platform,
    null,
    (player, platform) => player.body.bottom <= platform.body.top + 2,
    this
  );
}
```

:::

The process callback runs every overlap frame, so keep it cheap.

## Static groups and terrain

A **static group** (`physics.add.staticGroup()`) is a container for bodies the physics engine treats as immovable — gravity is ignored, velocity is never integrated, penetration is resolved on whatever collided with it, not on the static body. That makes static bodies effectively free; you can have hundreds of platform tiles without a frame-time cost.

The two rules of thumb: anything the level geometry needs goes in a static group, and if a "platform" should move, it is *not* static — use a regular dynamic body with `body.allowGravity = false` and `body.immovable = true`. That second case is exactly the moving platform you built in step 21 of the platformer workshop:

:::interactive_editor

```js
function create() {
  this.movingPlatforms = this.physics.add.group({
    allowGravity: false,
    immovable: true
  });
  const plat = this.add.rectangle(120, 120, 60, 8, 0xffaa44);
  this.movingPlatforms.add(plat);
  this.tweens.add({
    targets: plat,
    x: 200,
    duration: 1500,
    yoyo: true,
    repeat: -1
  });
}
```

:::

If you scale a static sprite after `staticGroup.create`, you must call `body.refresh()` — otherwise the collision rectangle stays at the original size and the player phases through.

## Body offsets and collision sides

The physics body does not have to match the sprite. `body.setSize(width, height)` resizes the body without touching the visual; `body.setOffset(dx, dy)` shifts the body inside the sprite's bounding box. Together they let you tune hit-boxes for game-feel — a player whose body is slightly smaller than the sprite feels generous, a hazard whose body is slightly larger than the sprite feels punishing.

:::interactive_editor

```js
function create() {
  this.player = this.physics.add.sprite(120, 60, "hero", 12);
  this.player.body.setSize(10, 14);
  this.player.body.setOffset(3, 2);
  const platform = this.add.rectangle(120, 140, 200, 8, 0x66ddff);
  this.physics.add.existing(platform, true);
  this.physics.add.collider(this.player, platform);
}
```

:::

After every `update`, Arcade physics fills in two booleans on each body: `body.touching` and `body.blocked`. They look interchangeable but are not.

- `body.touching.{up,down,left,right}` is **true while the body is in contact with another body** on that side this frame.
- `body.blocked.{up,down,left,right}` is **true while the body is in contact with the world bounds** on that side.

Read `touching.down` to detect "standing on a platform". Read `blocked.down` to detect "standing on the world floor". Most ground checks should `||` them together: `body.touching.down || body.blocked.down`.

## One-way platforms

A one-way platform lets the player jump up *through* it but stand on it on the way down. The cleanest implementation is a static group plus a process callback that returns `true` only when the player is above the platform:

:::interactive_editor

```js
function create() {
  this.player = this.physics.add.sprite(60, 40, "hero", 12);
  this.oneWay = this.physics.add.staticGroup();
  this.oneWay.add(this.add.rectangle(120, 100, 80, 4, 0xaaffaa));
  this.physics.add.collider(
    this.player,
    this.oneWay,
    null,
    (player, plat) => player.body.bottom <= plat.body.top + 2,
    this
  );
}
```

:::

The `+ 2` epsilon absorbs sub-pixel jitter; without it the player can pop through on the very last frame of contact.

## Coyote time and jump buffering

Two grace windows make jumps feel right.

**Coyote time** keeps the jump alive a few frames after the player has left a ledge. Stamp `lastGroundedAt = time` while on the ground; allow the jump while `time - lastGroundedAt < N` (80–150ms is the typical range).

**Jump buffer** keeps an early jump press alive until the player lands. Stamp `lastJumpPressedAt = time` on the press; consume it when both grace windows say yes.

Combine both and the jump fires whenever a recent press meets a recent ground state:

:::interactive_editor

```js
function update(time) {
  const onGround =
    this.player.body.touching.down || this.player.body.blocked.down;
  if (onGround) this.lastGroundedAt = time;
  if (Phaser.Input.Keyboard.JustDown(this.cursors.up)) {
    this.lastJumpPressedAt = time;
  }
  const canJump = time - this.lastGroundedAt < 120;
  const buffered = time - this.lastJumpPressedAt < 120;
  if (canJump && buffered) {
    this.player.setVelocityY(-310);
    this.lastGroundedAt = -Infinity;
    this.lastJumpPressedAt = -Infinity;
  }
}
```

:::

Stamping the timestamps to `-Infinity` after a jump prevents the buffer from chaining a phantom second jump out of the same press.

**Variable jump height** stacks on top: when the player releases `up` while still rising, clamp `velocity.y` to a smaller-magnitude negative number. Tap = short hop, hold = full jump. The single line that does it:

:::interactive_editor

```js
function update() {
  if (
    Phaser.Input.Keyboard.JustUp(this.cursors.up) &&
    this.player.body.velocity.y < -120
  ) {
    this.player.setVelocityY(-120);
  }
}
```

:::

These three forgiveness windows — coyote time, jump buffer, variable jump height — are the difference between a platformer that frustrates and a platformer that feels good.
