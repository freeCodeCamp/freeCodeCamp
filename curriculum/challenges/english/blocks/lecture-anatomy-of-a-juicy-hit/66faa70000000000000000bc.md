# --interactive--

**Hit-pause** (also called "hitstop", "frame freeze", or "impact pause") is the trick where time briefly stops on contact, then resumes. It's the single highest-impact-per-LOC effect in the juice stack, but devs skip it because they don't know which clock to pause.

In Phaser, three independent clocks govern motion:

| Clock                              | What it drives                          | Pause method                                 |
| ---------------------------------- | --------------------------------------- | -------------------------------------------- |
| `physics.world` (Arcade)           | Gravity, velocity, overlap callbacks    | `physics.world.pause()` / `.resume()`        |
| `tweens` (per-tween)               | Active tweens (scale, tint, position)   | `tween.pause()` / `tween.resume()` per-tween |
| `time.timeScale` (master scale)    | All `time.delayedCall`s + `time.events` | Set `time.timeScale = 0` then back to `1`    |

The cleanest hit-pause uses `physics.world.pause()` ONLY:

```js
this.physics.world.pause();
this.time.delayedCall(40, () => this.physics.world.resume());
```

40ms is the sweet spot. Shorter than that and the player doesn't perceive it; longer than ~80ms and it starts to feel like a stutter rather than impact. For heavy hits — boss attacks, finishing blows — you can scale up to 120ms.

The trick: `time.delayedCall` runs on the `time` clock, NOT the physics clock. So even though `physics.world` is paused, the resume call still fires on schedule. If you used `time.timeScale = 0` instead, you'd freeze your own resume timer too — the game would pause forever.

```js
// The wrong way - freezes its own unfreezing.
this.time.timeScale = 0;
this.time.delayedCall(40, () => this.time.timeScale = 1);
// ↑ This delayedCall never fires because the time clock is paused.
```

```js
// The right way - physics pauses, time keeps running.
this.physics.world.pause();
this.time.delayedCall(40, () => this.physics.world.resume());
```

If you want to pause tweens too (e.g., enemies tweening their squash/stretch should freeze with the world), pause individual active tweens or call `this.tweens.pauseAll()` / `.resumeAll()`.

:::interactive_editor

```js
function explainHitPause(approach) {
  const recipes = {
    "physics-only": {
      pauses: ["enemy velocities", "physics overlaps"],

    },
    "timeScale-zero": {

    },
    "all-three": {

}
```

For a heavy hit you can also pause all active tweens to freeze the squash/stretch alongside the physics:

```js
this.physics.world.pause();
this.tweens.pauseAll();
this.time.delayedCall(80, () => {
  this.physics.world.resume();
  this.tweens.resumeAll();
});
```

This makes the freeze look total — even the enemy's squash mid-deformation halts. Use sparingly; on multi-enemy combos this can feel heavy.

A subtle gotcha: `physics.world.pause()` does NOT stop your `update(time, delta)` callback. If your update reads input and updates sprite positions directly (without using physics velocity), those still advance. For a "true freeze" you'd need to gate your update body on a `this.frozen` flag — but in 95% of games, "physics pause" is the right scope.

# --questions--

## --text--

Why does `this.time.timeScale = 0; this.time.delayedCall(40, () => this.time.timeScale = 1);` cause the game to freeze permanently?

## --answers--

`time.delayedCall` runs on the time clock that we just zeroed. The resume callback can never fire because its scheduler is paused.

---

Phaser throws an error when `timeScale` reaches 0.

### --feedback--

It accepts 0 (and even negative values) without error.

---

`delayedCall` only accepts integer milliseconds.

### --feedback--

It accepts any number.

---

The garbage collector deletes the timer when paused.

### --feedback--

Pausing doesn't garbage-collect timers; they sit in the queue waiting for the clock to advance.

## --text--

You want a 40ms hit-pause that freezes physics movement but lets the enemy's squash/stretch tween continue playing. Which approach?

## --answers--

`this.physics.world.pause()` followed by `this.time.delayedCall(40, () => this.physics.world.resume())`. Tweens run on a separate manager, so they keep advancing.

---

`this.tweens.pauseAll()` then resume after 40ms.

### --feedback--

That pauses tweens but leaves physics running. The opposite of what you want.

---

`this.scene.pause()` then resume.

### --feedback--

That pauses the entire scene including the resume timer.

---

`this.cameras.main.fadeOut(40)`.

### --feedback--

That fades the camera, not pause anything.

## --text--

A common mistake on heavy "boss" hits is a 200ms hit-pause. What goes wrong?

## --answers--

200ms reads as a stutter or frame-rate drop, not an impact. The hit-pause sweet spot is ~40ms (light) to ~120ms (heavy). Beyond that it stops feeling like a "freeze" and starts feeling like the game broke.

---

Phaser's physics engine corrupts colliders if paused longer than 100ms.

### --feedback--

It does not.

---

200ms is too short for the player to notice.

### --feedback--

It's actually too long; the perceptual window for "impact freeze" is closer to 50ms.

---

Audio desyncs from physics if paused longer than 100ms.

### --feedback--

Audio doesn't drive physics; this is unrelated.
