# --interactive--

Bounce is how much of a body's velocity survives a collision. Set it with `body.setBounce(x, y)` (or pass one number for both axes). The value is the **coefficient of restitution**: `0` means "kill all velocity along that axis on impact," `1` means "perfectly elastic — the body returns at the same speed it arrived," anything in between is partial.

:::interactive_editor

```js
function create() {
  const ball = this.add.circle(120, 30, 8, 0xffaa44);
  this.physics.add.existing(ball);
  ball.body.setBounce(0.7);
  ball.body.setCollideWorldBounds(true);
}
```

:::

Combine bounce with `setCollideWorldBounds(true)` and the world's edges become rebounders. World-bounds bounce is the same coefficient as collider bounce; flip the bounce value live and the next wall hit picks up the change immediately.

There is no native friction in arcade physics. A platformer character does not "stick" to a floor because of friction — it stops because the player is no longer pressing left/right and `setDrag` is doing the deceleration. If you need surface-specific friction (an icy platform vs. a grippy one), the trick is to swap the player's drag value when they overlap a surface zone.

:::interactive_editor

```js
function create() {
  this.player = this.add.rectangle(120, 90, 16, 16, 0xffffff);
  this.physics.add.existing(this.player);
  this.player.body.setDragX(800);
  this.player.body.setVelocityX(200);
}
```

:::

A `setBounce` value greater than `1` is unusual but legal — the body *gains* velocity on each hit. Use this for the trampoline / spring-pad pattern (a one-shot boost on overlap), not for a generic "bouncy" surface, because energy will run away within a few hits and the simulation breaks down.

The recipe for a calm, predictable bouncing body: `setBounce(0.6)`, `setCollideWorldBounds(true)`, `setDrag(40, 0)`. It bounces, it eventually settles, it does not phase through walls.

# --questions--

## --text--

A body with `setBounce(1)` hits a stationary wall at 200 px/s. What's its rebound speed in arcade physics, ignoring drag?

## --answers--

`200` px/s — bounce `1` is perfect elasticity.

---

`100` px/s — bounce halves the speed.

### --feedback--

Bounce of `0.5` would do that. `1` preserves all speed.

---

`0` px/s — anything that hits a wall stops.

### --feedback--

That's bounce of `0`.

---

`400` px/s — bounce doubles speed by default.

### --feedback--

Bounce values above `1` are uncommon and *gain* energy; the default is at most `1`.

## --text--

You want a ball that bounces off the floor at full speed but loses energy on the side walls. What's the right call?

## --answers--

`ball.body.setBounce(0.5, 1)` — independent X and Y bounce.

---

`ball.body.setBounce(0.5)` — set the average and Phaser interpolates per axis.

### --feedback--

`setBounce(x)` sets both axes to the same value; Phaser does not interpolate.

---

`ball.body.setBounceX(0.5)` only — the Y axis defaults to `0`.

### --feedback--

`setBounceX` works, but `Y` defaults to whatever you previously set, not `0`.

---

You cannot vary bounce per axis in arcade physics.

### --feedback--

`setBounce(x, y)` is exactly the API for this.

## --text--

How do you simulate a "slippery ice" surface for the player in arcade physics?

## --answers--

Lower the player's drag while they overlap the ice zone, and restore it when they leave.

---

Set the ice tile's `friction` property to `0`.

### --feedback--

Arcade physics has no native `friction` property — friction is simulated through drag.

---

Add gravity to the ice tile so it pulls the player along.

### --feedback--

That doesn't model friction; it models a magnet.

---

Reduce the player's `setBounce` to `0` over the ice.

### --feedback--

Bounce affects collisions, not surface stickiness.
