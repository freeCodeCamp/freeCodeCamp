# --interactive--

A door is an object that swaps you between two pieces of level. Sometimes that means a different scene, sometimes a different region of the same map, sometimes the same coordinates with different state. The pattern that scales to every shape of "room transition" is simple: each door names its **destination**, the player overlap fires a transition, and the receiving door names where to spawn.

```js
const doors = [
  {
    name: "to_crypt_north",
    type: "door",
    x: 384,
    y: 112,
    properties: [
      { name: "to", value: "crypt_north_entry" },
      { name: "fade", value: 250 }
    ]
  },
  {
    name: "crypt_north_entry",
    type: "door_landing",
    x: 64,
    y: 64
  }
];
```

The transition is a three-step sequence: read the door's destination → fade the camera → set the player's position to the destination's coordinates → fade back in. With two cameras this is also where you'd drop a "Loading…" overlay, but for same-map transitions the fade is plenty.

:::interactive_editor

```js
async function transitionToDoor(player, fromDoor, doors, fadeMs) {
  const destName = fromDoor.properties.find((p) => p.name === "to").value;
  const dest = doors.find((d) => d.name === destName);
  console.log(`fade out (${fadeMs}ms)`);
  console.log(`teleport ${player.x},${player.y} -> ${dest.x},${dest.y}`);
  player.x = dest.x;
  player.y = dest.y;
  console.log(`fade in (${fadeMs}ms)`);
  return { x: player.x, y: player.y };
}

const doors = [
  {
    name: "to_crypt_north",
    type: "door",
    x: 384,
    y: 112,
    properties: [
      { name: "to", value: "crypt_north_entry" },
      { name: "fade", value: 250 }
    ]
  },
  {
    name: "crypt_north_entry",
    type: "door_landing",
    x: 64,
    y: 64
  }
];

const player = { x: 384, y: 112 };
transitionToDoor(player, doors[0], doors, 250).then((p) =>
  console.log("landed at", p)
);
```

:::

Three rules keep this from rotting as the project grows.

**1. Don't teleport into a door zone.** If destination `crypt_north_entry` overlaps the doorway, the player will trigger the *return* transition the moment they land. Always position the destination one tile *past* the door, in the direction the player is meant to be facing.

**2. Add a transition cooldown.** A 250 ms ignore window after a transition prevents the player walking back through the same overlap zone and getting bounced. The simplest version is a `this.transitionLockedUntil = time + 500` flag and an early return in the overlap callback.

**3. Persist what should outlive a transition.** Inventory, HP, score, and quest flags belong on a long-lived object — typically the scene's `data` registry or a singleton store. The tilemap doesn't survive the transition, but the player's state should. The same idea applies to "room visited" flags: store them on the scene, not on the tilemap object.

For genuinely huge worlds, doors that point at *different scenes* (`scene.start("CryptNorth", { spawnAt: "crypt_north_entry" })`) become the right call. The pattern is identical — destination by name, fade, set position — but each scene loads its own tilemap. Same vocabulary, bigger scope.

# --questions--

## --text--

Why does the destination object live in the same object layer (or a sibling layer) instead of being a hardcoded coordinate on the door itself?

## --answers--

Designers can move the landing spot in Tiled without touching code, and the door declares only *which* destination it points at — not *where* it is.

---

Phaser disallows hardcoded coordinates.

### --feedback--

It permits them; it's just not the maintainable approach.

---

It saves memory.

### --feedback--

A few extra fields per door is negligible; the win is editorial flexibility.

---

Hardcoded coordinates don't render.

### --feedback--

Coordinates aren't rendered either way. The reason is editorial, not visual.

## --text--

A player walks into a door, fades out, teleports, fades in — and immediately fades out again. What's the bug?

## --answers--

The destination object overlaps the receiving door's trigger zone, so the overlap callback fires again on landing.

---

The fade duration is too short.

### --feedback--

Even an instant fade should not loop forever. The position is overlapping the trigger.

---

The camera bounds aren't set.

### --feedback--

That would scroll past the map, not loop transitions.

---

`physics.add.overlap` was called twice.

### --feedback--

That'd double-fire the callback once, not loop forever. The cause is positional.

## --text--

Inventory and HP need to survive a room-to-room transition. Where do they live?

## --answers--

On a long-lived object — typically the scene instance (`this.inventory`) or the global `game.registry` — *not* on a tilemap object that gets torn down with the room.

---

On the door object's properties.

### --feedback--

Door properties are static level data. Player state changes at runtime.

---

In the tilemap layer's data array.

### --feedback--

The data array is gid integers. It can't store anything else.

---

In a global variable that you reset on every scene start.

### --feedback--

Globals work — `game.registry` is the structured version — but you must *not* reset on every scene start. That'd wipe what you're trying to persist.
