# --interactive--

Object type and name are the headline data. Everything else — *which* key unlocks *which* door, what the NPC says, how fast the slime patrols, which sound the chest plays — lives on **custom properties**. Tiled exposes them as a properties panel on each object; the JSON stores them as an array of `{ name, type, value }` records.

```js
const lockedDoor = {
  name: "locked_door",
  type: "door",
  x: 384,
  y: 112,
  properties: [
    { name: "locked", type: "bool", value: true },
    { name: "tx", type: "int", value: 24 },
    { name: "ty", type: "int", value: 7 }
  ]
};
```

Phaser hands you the array as-is. You'll write the same lookup helper a hundred times in your career, so just learn it now:

```js
function getProp(obj, key, defaultValue = undefined) {
  if (!obj.properties) return defaultValue;
  const found = obj.properties.find((p) => p.name === key);
  return found ? found.value : defaultValue;
}
```

:::interactive_editor

```js
function getProp(obj, key, defaultValue = undefined) {
  if (!obj.properties) return defaultValue;
  const found = obj.properties.find((p) => p.name === key);
  return found ? found.value : defaultValue;
}

const lockedDoor = {
  name: "locked_door",
  type: "door",
  x: 384,
  y: 112,
  properties: [
    { name: "locked", type: "bool", value: true },
    { name: "tx", type: "int", value: 24 },
    { name: "ty", type: "int", value: 7 }
  ]
};

const npc = {
  name: "elder",
  type: "npc",
  x: 128,
  y: 96,
  properties: [
    {
      name: "lines",
      type: "string",
      value: "Welcome traveler.|Find the bronze key.|Then return east."
    }
  ]
};

console.log("locked?", getProp(lockedDoor, "locked"));
console.log("door tile:", getProp(lockedDoor, "tx") + "," + getProp(lockedDoor, "ty"));
console.log("dialogue:", getProp(npc, "lines").split("|"));
console.log("missing:", getProp(npc, "voice", "default"));
```

:::

Three patterns this design unlocks:

**1. Designers configure gameplay without touching code.** A level designer toggling `locked: true` on one door doesn't need to ship JS. The same code path handles "this door is locked" and "this door isn't" — the predicate just reads the property.

**2. The property bag is a thin wire format, not a structure to fight.** Resist the urge to mirror the property array into a typed object on load. The flat array is small, JSON-friendly, and Tiled-native; reading it directly via `getProp(o, "key")` is fine.

**3. Strings can encode lists.** Tiled doesn't have a "string array" type. The convention is to store the array as a delimiter-joined string — `"a|b|c"` for a `lines` property — and `.split("|")` it on read. That's how the NPC's dialogue lines travel from Tiled to runtime.

# --questions--

## --text--

The locked-door object has `properties: [{ name: "tx", value: 24 }]`. You want to read `tx` in your code. What's the idiomatic call?

## --answers--

Iterate (or `.find`) the `properties` array looking for the entry whose `name === "tx"`, then read `.value`.

---

`door.tx`

### --feedback--

Phaser does not flatten property entries onto the object. They live in the array.

---

`door.properties.tx`

### --feedback--

`door.properties` is an *array*, not an object keyed by property name.

---

`door.getProperty("tx")`

### --feedback--

Phaser objects from `findObject` are plain data — there's no `getProperty` method.

## --text--

A designer wants to encode an NPC's three dialogue lines in Tiled. Tiled has no "string array" property type. What's the conventional encoding?

## --answers--

A single string property like `"line1|line2|line3"` — split on a delimiter at read time.

---

Three separate properties named `line1`, `line2`, `line3`.

### --feedback--

Workable but rigid — extending to four lines means a code change. The single delimited string scales without code edits.

---

Encode as JSON and parse at runtime.

### --feedback--

That works too, but a delimited string is simpler, easier to author, and survives copy-paste better.

---

Use the object's `name` field.

### --feedback--

The `name` is for identifying the object, not for cargo data.

## --text--

What's the right default-value strategy when reading a property that may or may not exist?

## --answers--

A lookup helper that takes a default — `getProp(obj, key, fallback)` — so missing properties are handled at the call site, not by crashing.

---

Trust that every object has every property and let it crash if not.

### --feedback--

Brittle. A new object type without that property crashes the level.

---

Add every property to every object, even when not relevant.

### --feedback--

You drown the property panel and confuse designers. Use defaults at the call site instead.

---

Catch the error in a `try/catch` and ignore.

### --feedback--

A `try/catch` masks real bugs. The default-value helper is more explicit.
