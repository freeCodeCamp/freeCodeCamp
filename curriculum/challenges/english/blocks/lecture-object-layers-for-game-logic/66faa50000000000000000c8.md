# --interactive--

A tilemap layer answers one question: "what tile sits in cell `(tx, ty)`?". For floor and walls, that's enough. For everything else — where does the player spawn? where does the NPC stand? where do the doors lead? where does the slime patrol? — you need data that doesn't fit on a 16×16 grid. That's what **object layers** are for.

An object layer holds free-form objects with pixel coordinates, a name, a type, and a properties bag. They are not rendered as tiles. Phaser doesn't even draw them by default — they're pure level *data*.

```js
// In Tiled, you draw a 16x16 box and tag it:
//   name: "spawn"  type: "spawn"  x: 64  y: 64
// In Phaser:
const spawn = map.findObject("objects", (o) => o.type === "spawn");
this.player.setPosition(spawn.x, spawn.y);
```

The object layer in the dungeon you'll build looks roughly like this:

```js
const objectLayer = [
  { name: "spawn", type: "spawn", x: 64, y: 64 },
  { name: "elder", type: "npc", x: 128, y: 96 },
  { name: "key", type: "key", x: 128, y: 512 },
  { name: "locked_door", type: "door", x: 384, y: 112,
    properties: [{ name: "locked", value: true }] },
  { name: "slime_patrol", type: "slime", x: 608, y: 448,
    polyline: [{ x: 0, y: 0 }, { x: 96, y: 0 }] },
  { name: "chest", type: "chest", x: 800, y: 96 },
  { name: "save", type: "save", x: 160, y: 512 },
  { name: "goal", type: "goal", x: 864, y: 512 }
];
```

The two access patterns Phaser ships are `findObject` (returns the first match) and `filterObjects` (returns all matches).

```js
const spawn = map.findObject("objects", (o) => o.type === "spawn");
const allDoors = map.filterObjects("objects", (o) => o.type === "door");
```

:::interactive_editor

```js
function findObject(layer, predicate) {
  return layer.find(predicate) || null;
}
function filterObjects(layer, predicate) {
  return layer.filter(predicate);
}

const objects = [
  { name: "spawn", type: "spawn", x: 64, y: 64 },
  { name: "elder", type: "npc", x: 128, y: 96 },
  { name: "key", type: "key", x: 128, y: 512 },
  { name: "door_a", type: "door", x: 384, y: 112 },
  { name: "door_b", type: "door", x: 784, y: 64 }
];

console.log("spawn:", findObject(objects, (o) => o.type === "spawn"));
console.log("doors:", filterObjects(objects, (o) => o.type === "door").length);
console.log("by name:", findObject(objects, (o) => o.name === "elder"));
```

:::

Two design rules carry this pattern a long way.

**`type` is the verb, `name` is the proper noun.** `type: "door"` says how the object behaves. `name: "locked_door"` says which specific door it is. Code dispatches on `type`; designers identify by `name`.

**The object's pixel coordinates are world-space, not grid-space.** A 16×16 object on tile `(4, 4)` has `x: 64, y: 64`. If you mistakenly compare to grid coordinates, every object will look like it's at the origin. Always treat object `x/y` as pixels.

# --questions--

## --text--

Why does the spawn point live on the object layer instead of the tilemap layer?

## --answers--

Because the spawn point is a single point of data, not a 16×16 image — and you want pixel-precise placement, custom properties, and a stable name to reference it by.

---

Because the tilemap layer can't store the player's coordinates.

### --feedback--

It could store a "spawn" tile, but you'd be wasting an entire grid cell to encode one point of data. Object layers are the better fit.

---

Because Phaser refuses to read the spawn from a tile layer.

### --feedback--

Phaser would read it fine; the question is which layer the data *belongs* on.

---

Because object layers render faster than tile layers.

### --feedback--

Object layers don't render at all by default — that's not the reason.

## --text--

A level has 8 doors. Which API should you reach for to get all of them?

## --answers--

`map.filterObjects("objects", (o) => o.type === "door")` — returns the array of all matching objects.

---

`map.findObject("objects", (o) => o.type === "door")`

### --feedback--

That returns only the *first* door.

---

`map.getLayer("objects").objects.find((o) => o.type === "door")`

### --feedback--

`find` returns one. You want `filter`.

---

`map.getObjectLayer("doors")`

### --feedback--

Object layers are looked up by their layer name (not the object type). And you'd still need to filter the resulting `objects` array.

## --text--

You see this object: `{ type: "npc", name: "elder", x: 128, y: 96 }`. Where in world-space is the NPC?

## --answers--

At pixel `(128, 96)` — object coordinates are pixels.

---

At grid cell `(128, 96)`.

### --feedback--

That would put it 128 tiles to the right of origin — way outside any reasonable map.

---

At pixel `(8, 6)` — divide by 16.

### --feedback--

The reverse: tile `(8, 6)` is at pixel `(128, 96)`. The object's stored coordinate is already in pixels.

---

At the camera's center.

### --feedback--

Object coordinates are absolute world-space, not relative to the camera.
