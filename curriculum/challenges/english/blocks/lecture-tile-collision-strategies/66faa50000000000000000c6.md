# --interactive--

The boolean property from the previous lecture says "this tile is solid". It does *not* say *what shape* is solid. By default, Phaser uses the entire 16×16 square as the hitbox. For irregular tiles — a half-wall, a sloped ramp, a pillar that's only solid in its central column — that default is wrong.

Tiled's **Tile Collision Editor** lets you draw custom collision shapes on individual tiles. Each tile can carry one or more rectangles (or polygons), and the JSON export stores them under `tile.objectgroup.objects`.

```json
{
  "id": 33,
  "objectgroup": {
    "objects": [
      { "x": 5, "y": 0, "width": 6, "height": 16 }
    ]
  }
}
```

That's a 6×16 vertical column centered on the tile — exactly what you want for a pillar. Phaser reads the `objectgroup` automatically when the layer is built; tiles flagged with custom shapes use those shapes for collision instead of the bounding box.

:::interactive_editor

```js
function tileHitbox(tile) {
  if (!tile.objectgroup || !tile.objectgroup.objects.length) {
    return { x: 0, y: 0, width: 16, height: 16, kind: "default" };
  }
  const shape = tile.objectgroup.objects[0];
  return {
    x: shape.x,
    y: shape.y,
    width: shape.width,
    height: shape.height,
    kind: "custom"
  };
}

const tiles = [
  { id: 12, label: "wall" },
  {
    id: 33,
    label: "pillar",
    objectgroup: { objects: [{ x: 5, y: 0, width: 6, height: 16 }] }
  },
  {
    id: 40,
    label: "half-wall",
    objectgroup: { objects: [{ x: 0, y: 8, width: 16, height: 8 }] }
  }
];
for (const t of tiles) {
  console.log(t.label, JSON.stringify(tileHitbox(t)));
}
```

:::

The everyday alternative — and the right call most of the time — is to keep tiles square and design *around* the grid. A 6-wide pillar can just be a 16-wide pillar tile with art that makes it look thinner. Players read collision through the visual, and the visual can lie a little. Rectangle overrides are best reserved for tiles where the lie would be obvious: spikes, slopes, half-floors that the player needs to hop onto.

There is a related feature for irregular sprites (not tiles): `body.setSize(w, h).setOffset(ox, oy)`. That tweaks a *sprite's* hitbox at runtime. Tile rectangle overrides do the same job for *map tiles* at author-time — once, in the editor, instead of repeatedly in code.

# --questions--

## --text--

You have a tile that is visually a half-floor — only the bottom 8 pixels should be solid. What is the cleanest fix?

## --answers--

Tiled Tile Collision Editor: draw a `16 × 8` rectangle at `(0, 8)` on that tile. Phaser respects it automatically.

---

`setCollisionBetween(40, 40)` and call it solid.

### --feedback--

That makes the whole 16×16 square solid; the player will collide with empty air above the floor.

---

In `update`, manually push the player out of the top half if they overlap.

### --feedback--

Fragile and per-frame. Use the editor's per-tile rectangle once.

---

Use a separate sprite for the half-floor instead of a tile.

### --feedback--

That defeats the point of tilemaps. Per-tile rectangles are exactly the editor feature for this case.

## --text--

Custom per-tile rectangles defined in Tiled live where in the JSON?

## --answers--

Inside the tileset definition, on the specific tile entry: `tile.objectgroup.objects[]`.

---

Inside each map's `objectgroup` layer.

### --feedback--

That layer is for *world-level* objects (NPCs, doors). Per-tile rectangles travel with the tileset.

---

Inside the layer's `data` array, packed into the gid bits.

### --feedback--

The high gid bits encode flip flags, not collision shapes.

---

Inside `map.properties`.

### --feedback--

Map-level properties are for things like room name and ambient sound, not per-tile collision.

## --text--

When should you reach for per-tile rectangle overrides?

## --answers--

When the *visual* of a tile makes a 16×16 hitbox feel obviously wrong — slopes, spikes, half-floors — and changing the art alone would not fix it.

---

For every wall tile, to be safe.

### --feedback--

Walls are the case where 16×16 is correct. Adding overrides everywhere just slows the load.

---

For decorative tiles like torches.

### --feedback--

Decorative tiles should not collide at all. The fix is to omit `collides: true`, not to add a hitbox.

---

When the tileset has more than 100 tiles.

### --feedback--

Tileset size has nothing to do with whether overrides help.
