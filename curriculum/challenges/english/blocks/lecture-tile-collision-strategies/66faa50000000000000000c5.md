# --interactive--

A tilemap layer renders fine without any collision setup — but nothing blocks the player. Telling Phaser which tiles are solid is a separate, explicit step. There are three APIs for this, and the choice between them is one of those small decisions that defines whether your level data is fun to maintain or painful.

Option 1 is **index-based**: hardcode the local tile ids that should collide.

```js
walls.setCollision(12);
walls.setCollisionBetween(12, 23);
walls.setCollision([12, 13, 14, 15]);
```

Option 2 is **exclusion-based**: collide with every tile *except* a list of safe ones.

```js
walls.setCollisionByExclusion([0, 1, 2]);
```

Option 3 is **property-based**: in Tiled, tag each solid tile with a custom boolean and let Phaser walk the layer.

```js
walls.setCollisionByProperty({ collides: true });
```

The pragmatic answer for nontrivial maps is **always property-based**. Index lists rot the moment an artist adds a tile, renumbers a tileset, or swaps in new art. Exclusion lists rot the moment you add a non-solid decorative tile. Properties live with the art; renumber the tileset and the property travels with the tile.

:::interactive_editor

```js
const tileset = {
  tiles: [
    { id: 1, props: { kind: "floor" } },
    { id: 12, props: { kind: "wall", collides: true } },
    { id: 13, props: { kind: "wall", collides: true } },
    { id: 25, props: { kind: "torch" } },
    { id: 27, props: { kind: "chest" } }
  ]
};

function setCollisionByProperty(layer, predicate) {
  return layer
    .map((row) =>
      row.map((gid) => {
        const tile = tileset.tiles.find((t) => t.id === gid);
        const collides = !!tile && predicate(tile.props);
        return collides ? "X" : ".";
      })
    )
    .map((row) => row.join(" "))
    .join("\n");
}

const layer = [
  [12, 12, 12, 12, 12],
  [12, 1, 25, 1, 12],
  [12, 1, 27, 1, 12],
  [12, 12, 13, 12, 12]
];
console.log(setCollisionByProperty(layer, (p) => p.collides === true));
```

:::

The cost of the property scan is one O(width × height) walk at scene start. After that, every tile carries a cached `collides` flag, and arcade physics uses it directly. Even on a 200×200 map (40,000 tiles), this is a one-time millisecond cost, not a per-frame cost.

There is one nonobvious case: tilesets sometimes ship with collision flags already baked into the tile id ranges (Tiled's "Tile Collision Editor"). Those are different — they store *per-tile rectangles*, not the boolean we covered here. The next lecture is about those.

# --questions--

## --text--

You change a tile's id from `12` to `27` to make room for a new wall variant. Which collision API survives without code edits?

## --answers--

`setCollisionByProperty({ collides: true })` — properties travel with the tile, so the renumber is invisible to gameplay.

---

`setCollision(12)` — just keep both ids in the call.

### --feedback--

You still have to remember to update the call. Property-based collision needs zero gameplay edits.

---

`setCollisionBetween(12, 23)` — the range stretches automatically.

### --feedback--

It does not. Ranges are static numbers; the range needs an explicit edit.

---

`setCollisionByExclusion([0, 1])` — any nonzero tile collides.

### --feedback--

Then your decorative torch tile (id `25`) collides too, which is wrong.

## --text--

A level has 8 wall variants (ids `12..19`), 4 floor variants (`1..4`), and a single torch tile (`25`). Which `setCollisionByExclusion` argument is correct?

## --answers--

`[0, 1, 2, 3, 4, 25]` — every non-collidable tile must be in the exclusion list, including the torch.

---

`[0]` — only the empty tile is non-collidable.

### --feedback--

That would make the floor and the torch collide. Decorative tiles must be excluded too.

---

`[1, 2, 3, 4]` — exclude floors only.

### --feedback--

Then `0` (empty) and `25` (torch) collide, which is wrong.

---

`[12, 13, 14, 15, 16, 17, 18, 19]` — exclude walls.

### --feedback--

That inverts the meaning. Exclusion lists name the *non*-collidable tiles; you would be naming the walls.

## --text--

When does index-based collision (`setCollision(12)`) stop being a maintenance headache?

## --answers--

When the tileset is small, frozen, and you control both the art and the gameplay code — typically a one-developer prototype.

---

When the tileset is small.

### --feedback--

Size is part of it, but the bigger factor is whether you (or anyone) will ever renumber tiles.

---

Never — index-based collision is always wrong.

### --feedback--

Too strong. For tiny prototypes with a frozen tileset, `setCollision(12)` is fine.

---

When the level is procedurally generated.

### --feedback--

Procgen is exactly when property-based collision wins biggest, because the generator only needs to know "is this tile solid?" — a property — not which id it picked.
