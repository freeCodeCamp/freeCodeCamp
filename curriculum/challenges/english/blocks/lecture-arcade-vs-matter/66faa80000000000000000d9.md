# --interactive--

Phaser 3 ships *two* physics engines side-by-side: **Arcade** and **Matter**. They're not competitors — they're tools for different jobs. Picking the wrong one is the most common cause of "my physics feels weird" bugs.

A trade-off table, line by line:

| Trait               | Arcade                                                                     | Matter                                                              |
| ------------------- | -------------------------------------------------------------------------- | ------------------------------------------------------------------- |
| Body shapes         | Axis-aligned rectangle or circle                                           | Rectangle, circle, polygon, compound, custom verts                  |
| Rotation collisions | Ignored — bodies never rotate when colliding                               | Full angular response (torque, angular velocity, friction)          |
| Constraints         | None native (you fake them with `setVelocity`)                             | Pin, distance, spring, weld — the whole Matter.js library           |
| Stacking            | Unstable past 2-3 layers                                                   | Solver iterates until stable; 20+ layer stacks work                 |
| Performance         | Microseconds per body. Thousands of bodies fine.                           | ~10× the cost. Hundreds of bodies; profile if you exceed 200       |
| Determinism         | Deterministic per-tick                                                     | Iterative solver; small float drift across runs                     |
| Mental model        | "Player is a velocity, world is a grid"                                    | "Everything obeys Newton; constraints connect things"               |
| API surface         | `body.setVelocity`, `setBounce`, `setCollideWorldBounds`                   | `body.applyForce`, `setAngularVelocity`, `Constraint.create`        |

Arcade was designed for arcade games — top-down shooters, side-scrollers, brick-breakers. Bodies are shapes that move in straight lines and stop when they touch each other. No spinning, no realistic stacking, no ragdolls.

Matter was originally [Matter.js](https://brm.io/matter-js/), a full 2D rigid-body engine. Phaser embeds it. You get realistic mass, friction, restitution, joints, springs, and a constraint solver that handles "stack 20 boxes and shoot them with a cannonball."

A practical heuristic: if your game looks like *Crossy Road*, *Galaga*, or *Super Mario Bros 1*, use Arcade. If it looks like *Angry Birds*, *Cut the Rope*, or *Crayon Physics*, use Matter.

You can run only one default physics engine per scene. Pick one in the scene config; switching mid-scene is not supported.

```js
const config = {
  physics: {
    default: "arcade",
    arcade: { gravity: { y: 600 }, debug: false }
  }
};
```

```js
const config = {
  physics: {
    default: "matter",
    matter: { gravity: { y: 1 }, debug: false }
  }
};
```

A subtle but important detail: Matter's gravity is in *different units*. Arcade's `gravity.y: 600` means "600 pixels per second squared." Matter's `gravity.y: 1` is a multiplier on Matter's internal gravity (`9.8 m/s²`-ish, scaled to pixel space). A Matter `gravity.y: 1` produces a felt acceleration similar to Arcade's `~600`. You can't compare the numbers directly.

:::interactive_editor

```js
function pickEngine(game) {
  const ranks = {
    arcade: 0,
    matter: 0
  };

  if (game.needsStacking) ranks.matter += 3;
  if (game.needsConstraints) ranks.matter += 3;
  if (game.needsRotationOnCollision) ranks.matter += 2;
  if (game.bodies > 500) ranks.arcade += 3;
  if (game.style === "platformer") ranks.arcade += 2;
  if (game.style === "topdown") ranks.arcade += 2;
  if (game.style === "physics-puzzle") ranks.matter += 3;

  return ranks.matter > ranks.arcade ? "matter" : "arcade";
}

console.log("Angry Birds clone:", pickEngine({
  needsStacking: true,
  needsConstraints: true,
  needsRotationOnCollision: true,
  bodies: 60,
  style: "physics-puzzle"
}));

console.log("Top-down dungeon:", pickEngine({
  needsStacking: false,
  needsConstraints: false,
  needsRotationOnCollision: false,
  bodies: 200,
  style: "topdown"
}));

console.log("Bullet hell shmup:", pickEngine({
  needsStacking: false,
  needsConstraints: false,
  needsRotationOnCollision: false,
  bodies: 800,
  style: "topdown"
}));
```

:::

The bullet hell case is interesting — even though it's *technically* "things hitting things," the bullets don't need to bounce, stack, or rotate. Arcade is dramatically faster for raw collision counts.

Don't fight the engine. If you need spring-loaded constraints, *use Matter*. If you need a tight platformer with predictable jumping, *use Arcade*. Forcing the wrong engine doubles the code and halves the feel.

# --questions--

## --text--

Why does an Angry Birds clone *need* Matter and not Arcade?

## --answers--

Slingshots are spring constraints, blocks need to stack and rotate on impact, and the whole feel depends on rigid-body collisions with realistic torque. Arcade can't do any of those.

---

Matter has better sprite-rendering performance than Arcade.

### --feedback--

Rendering is unrelated to physics — both use Phaser's renderer. The difference is body simulation.

---

Arcade can't load sprite sheets.

### --feedback--

Both engines work with any sprite source.

---

Matter is required for any game with gravity.

### --feedback--

Arcade has gravity too. The decider is constraints, stacking, and rotation.

## --text--

A top-down twin-stick shooter has 600 active bullets, 30 enemies, and a player. Which engine is the right call and why?

## --answers--

Arcade. None of the bodies stack, rotate on collision, or use constraints. Arcade handles 600+ bodies cheaply; Matter would burn frame budget on a constraint solver no body needs.

---

Matter, because it's the newer engine.

### --feedback--

"Newer" isn't a reason. Arcade is actively maintained and well-suited to this workload.

---

Both at once, with arcade for bullets and matter for the player.

### --feedback--

Phaser supports one default physics engine per scene. Don't dual-wield.

---

Arcade, but only if you disable gravity.

### --feedback--

Disabling gravity is unrelated to engine choice. Top-down games use `gravity.y: 0` regardless.

## --text--

You set Matter's `gravity.y: 600` expecting the same feel as Arcade's `gravity.y: 600`. Why does the projectile fall like a feather?

## --answers--

Matter's `gravity.y` is a *multiplier*, not a pixel-per-second-squared value. `1` is a normal Earth-like gravity; `600` is way too high *and* still wrong because the units don't compare. Use `gravity.y: 1` and tune from there.

---

Phaser bug — Matter's gravity is broken at high values.

### --feedback--

Not a bug. The unit system is just different.

---

The body's mass is too high.

### --feedback--

Default mass is fine. The issue is the gravity unit.

---

Matter requires `gravityScale` instead of `gravity`.

### --feedback--

`gravityScale` is per-body fine-tuning. The world-level config key is `gravity`.
