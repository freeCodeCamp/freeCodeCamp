# --interactive--

The "which engine?" decision is rarely close. Most game ideas land cleanly in one camp. Use this guide as a fast lookup.

**Pick Arcade when…**

- The world is grid-aligned, top-down, or a side-scroller with predictable jumping.
- Bodies don't rotate when they collide (a bullet hits a wall and stops; doesn't spin).
- You need 200+ active bodies and the frame budget matters.
- Determinism matters — replays, network rollback, deterministic level scripting.
- The genre is platformer, top-down shooter, twin-stick, brick-breaker, runner, shmup, snake, or arcade-style sports (Pong, Breakout, Pinball-arcade-style).

**Pick Matter when…**

- The game *is* the physics — the puzzle is "how do I knock these blocks down?"
- You need stacking that survives 5+ levels of boxes.
- You need constraints: springs, ropes, hinges, pins, welds, distance joints.
- Bodies must rotate on collision (a log rolls when it hits a slope).
- The genre is physics puzzle (Angry Birds, Cut the Rope, Crayon Physics), ragdoll (toy ragdoll, Happy Wheels), softbody (Crush the Castle), Rube Goldberg simulation, or 2D rigid-body construction games.

A few canonical examples broken down:

| Game                | Why this engine                                                                                       |
| ------------------- | ----------------------------------------------------------------------------------------------------- |
| *Super Mario Bros*  | Arcade. Mario is a velocity. Goombas are velocities. World is a tile grid. No rotation, no stacking. |
| *Donkey Kong (1981)* | Arcade. Same reasoning + Mario's hitbox is a fixed rectangle.                                        |
| *Angry Birds*       | Matter. Springs (slingshot), stacking (towers), rotation (logs), torque (pigs spin when hit).        |
| *Cut the Rope*      | Matter. Distance constraints (the rope), gravity, candy spinning on impact.                          |
| *Geometry Wars*     | Arcade. Hundreds of bodies, no rotation on collision, no stacking.                                   |
| *Worms*             | Matter for the explosion-throws-worms moments; could be hybrid but Matter dominates.                 |
| *Hill Climb Racing* | Matter. Spring-loaded suspension, rotating wheels, deformable terrain edges.                         |
| *Stardew Valley*    | Arcade. Top-down, axis-aligned hitboxes, no spring/joint/rotation needed.                            |

A common mistake: starting in Arcade because "it's simpler" then needing one of {stacking, springs, rotation} late in development. Migrating mid-project is *painful* — the entire collision callback API differs (Arcade has `physics.add.collider`, Matter has `world.on('collisionstart', ...)`). Decide up front.

When you're unsure, ask: **does the simulation itself create the gameplay?** If yes, Matter. If the simulation is just "things that move and bump", Arcade.

:::interactive_editor

```js
const games = [
  { name: "Brick Breaker", needsStacking: false, needsConstraints: false, needsRotateOnHit: false, bodies: 30 },
  { name: "Bridge Builder", needsStacking: true, needsConstraints: true, needsRotateOnHit: true, bodies: 80 },
  { name: "Top-down RPG", needsStacking: false, needsConstraints: false, needsRotateOnHit: false, bodies: 100 },
  { name: "Slingshot Puzzle", needsStacking: true, needsConstraints: true, needsRotateOnHit: true, bodies: 60 },
  { name: "Bullet hell", needsStacking: false, needsConstraints: false, needsRotateOnHit: false, bodies: 800 }
];

games.forEach((g) => {
  const matterReasons = [];
  if (g.needsStacking) matterReasons.push("stacking");
  if (g.needsConstraints) matterReasons.push("constraints");
  if (g.needsRotateOnHit) matterReasons.push("rotation");
  const pick = matterReasons.length > 0 ? "Matter" : "Arcade";
  console.log(`${g.name}: ${pick} (${matterReasons.length ? matterReasons.join("+") : "no Matter-only feature"})`);
});
```

:::

The Bullet hell case (800 bodies, no Matter-only feature) is interesting — even though Matter *can* do it, paying the constraint solver tax for 800 collision-only bodies wastes ~9× the CPU. Arcade is the right choice every time none of the Matter-only features are needed.

There's no "best" engine — there's only "best for the workload." Read your game design doc, find the load-bearing feature, pick the engine that supports it natively. The rest is implementation detail.

# --questions--

## --text--

You're building Pinball. Bumpers spin balls off at random angles, paddles flick the ball with torque, and the ball must lose energy realistically with each bounce. Which engine?

## --answers--

Matter. Spinning bodies, rotation-on-collision, restitution loss per bounce, and the "feel" of pinball *is* rigid-body physics. Arcade can't model the angular response.

---

Arcade — pinball is a classic arcade game.

### --feedback--

Cabinet name is unrelated to engine choice. Pinball is rigid-body simulation; Arcade can't simulate the spinning ball.

---

Hybrid: Arcade for the ball, Matter for the bumpers.

### --feedback--

Phaser doesn't support hybrid physics. Pick one default per scene.

---

Either works equally well.

### --feedback--

Arcade fundamentally can't reproduce angular momentum on collision.

## --text--

You started a top-down dungeon crawler with Arcade. Halfway through, the lead designer asks for "barrels that roll when you hit them." What's the lowest-pain path forward?

## --answers--

Push back on the design or fake it: rotate the sprite visually with a tween while keeping the Arcade body axis-aligned. Migrating to Matter mid-project means rewriting every `physics.add.collider` callback and restructuring scene boot.

---

Migrate the entire project to Matter today.

### --feedback--

The project will likely take 2x longer than the rest of development to migrate. Cosmetic-only rolling is usually accepted.

---

Use Matter just for barrels and Arcade for everything else.

### --feedback--

Phaser doesn't support multiple physics engines per scene.

---

Switch to a different game engine.

### --feedback--

Vastly disproportionate. Cosmetic rotation handles 90% of "rolling" feedback.

## --text--

Your game has a slingshot, 80 stackable blocks, and the rest of the gameplay is "watch the chain reaction." What's the engine and the load-bearing reason?

## --answers--

Matter. The slingshot needs a spring constraint, and the blocks need stable stacking — both Matter-only features. The chain-reaction feel comes free from Matter's rigid-body solver.

---

Arcade with manual force-application math.

### --feedback--

You'd be re-writing Matter from scratch. Skip the busywork.

---

A custom physics engine.

### --feedback--

Use Matter unless you have an extraordinary reason; reinventing physics is months of work.

---

Matter, but only because of the slingshot.

### --feedback--

Half-right — the *stacking* is at least as important. Either feature alone justifies Matter.
