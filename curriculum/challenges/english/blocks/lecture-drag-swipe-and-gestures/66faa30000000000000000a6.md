# --interactive--

A keyboard arrow press lights up the next pixel within a frame or two. A finger on a touchscreen does not. Between your finger landing and the GPU drawing the response, the OS samples the touch, debounces hardware noise, hands the data to the browser, the browser fires `touchmove`, Phaser turns it into a `pointermove`, your update runs, and only *then* does the renderer ship pixels. Total cost: 30–80 ms on a good phone, easily 120 ms on a cheap one.

You can't make the OS faster, but you can make the response *feel* faster by smoothing the visible motion. The trick: don't move the paddle directly to `pointer.x` — chase a smoothed target.

Linear easing is the workhorse:

```js
update() {
  this.paddle.x = Phaser.Math.Linear(this.paddle.x, this.targetX, 0.2);
}
```

`Phaser.Math.Linear(a, b, t)` returns `a + (b - a) * t`. With `t = 0.2`, the paddle moves 20% of the remaining distance every frame. The motion looks smooth, the paddle settles quickly, and any jitter in `pointer.x` averages out.

Three easing values worth memorizing:

- `0.05` — slow, syrupy chase. Great for cinematic camera follow, terrible for input.
- `0.2` — responsive but smooth. The default for paddle controls.
- `0.5` — almost-instant. Use only when raw responsiveness wins over visual polish.

For a snappier feel, mix two easing rates:

```js
const speed = Math.abs(this.targetX - this.paddle.x) > 60 ? 0.4 : 0.15;
this.paddle.x = Phaser.Math.Linear(this.paddle.x, this.targetX, speed);
```

Far movements race to catch up; close movements settle gently.

A different strategy is **rolling-average smoothing**. You buffer the last *N* samples of `pointer.x` and use the average:

```js
this.samples.push(pointer.x);
if (this.samples.length > 5) this.samples.shift();
this.targetX = this.samples.reduce((a, b) => a + b, 0) / this.samples.length;
```

This shines when raw input is noisy — old capacitive touchscreens, sketchy Bluetooth pens. Modern hardware is clean enough that lerp alone is usually plenty.

A trade-off worth understanding: any smoothing **adds latency**. If the player flicks the paddle 100 px to the right, a 0.2 lerp takes ~5 frames (~80 ms at 60 FPS) to converge. Players feel that. Match the smoothing to the genre — twitch shooters want minimum smoothing; puzzle games tolerate plenty.

## --text--

What is the role of `Phaser.Math.Linear(this.paddle.x, this.targetX, 0.2)` in a paddle update loop?

## --answers--

It moves the paddle 20% of the distance toward the target every frame, smoothing input jitter.

---

It teleports the paddle directly to the target, ignoring the previous position.

### --feedback--

`Phaser.Math.Linear(a, b, 1)` would teleport. With `t = 0.2`, the paddle approaches the target gradually.

---

It limits the paddle's max velocity to 20 pixels per frame.

### --feedback--

`Phaser.Math.Linear` does not impose a velocity cap; it returns a blended value.

---

It plays a tween from `paddle.x` to `targetX` over 200 ms.

### --feedback--

That would be `this.tweens.add(...)`, not `Linear`. `Linear` is a one-shot lerp; you call it every frame yourself.

## --text--

You set the lerp factor to `0.05` for a paddle. The player reports the paddle "lags" behind their finger. What's the fix?

## --answers--

Increase the lerp factor — `0.05` chases far too slowly for input control.

---

Decrease the lerp factor — slower lerp produces snappier feel.

### --feedback--

Smaller factors mean *less* movement per frame, so they feel more sluggish.

---

Switch to a per-frame teleport.

### --feedback--

That works but throws away all smoothing benefit; a higher lerp (~0.2) keeps the feel and lowers latency.

---

Change the canvas size — lerp scales with width.

### --feedback--

Lerp is unitless (a fraction of remaining distance); canvas size is irrelevant.

## --text--

What's the central trade-off of input smoothing?

## --answers--

Smoothing reduces jitter and looks better, but it adds latency between input and visible response.

---

Smoothing is always free; modern GPUs eliminate the cost.

### --feedback--

Smoothing is cheap to compute, but the *latency* cost is fundamental — averaging requires past samples.

---

Smoothing makes input *less* accurate, so it should be avoided in production.

### --feedback--

Smoothing reduces hardware noise, which usually *improves* perceived accuracy. The cost is added latency, not lost accuracy.

---

Smoothing only works on touchscreens; mouse input cannot be smoothed.

### --feedback--

Smoothing is mathematical; it works on any pointer source.
