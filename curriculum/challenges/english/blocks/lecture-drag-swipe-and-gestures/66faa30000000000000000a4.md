# --interactive--

Most input is "click here, fire there." Drag is different: the player presses, holds, and *moves with* the cursor or finger. Inventory rearrangement, slingshots, dial-spinners, paddle controls — all drag.

Phaser ships drag as an opt-in flag on `setInteractive`:

```js
const card = this.add.image(240, 320, "card");
card.setInteractive({ draggable: true });
```

That single flag wires up three new events on the object:

- `dragstart` — the user pressed the object and started moving.
- `drag` — fires every frame the user moves while still pressed; you get the new `dragX`/`dragY` for free.
- `dragend` — the user released.

The simplest move-with-finger pattern is one line in the `drag` handler:

```js
card.on("drag", (pointer, dragX, dragY) => {
  card.setPosition(dragX, dragY);
});
```

Phaser already adjusts `dragX`/`dragY` for the offset where the press first hit the sprite, so the cursor stays exactly where the user grabbed — no jumping.

Constraint patterns you'll reach for constantly:

- Lock to a single axis: `card.setPosition(dragX, card.y)` ignores vertical motion.
- Snap to a grid: `card.setPosition(Math.round(dragX / 32) * 32, card.y)`.
- Clamp to bounds: `card.setPosition(Phaser.Math.Clamp(dragX, 40, 440), card.y)`.

Combine all three for a paddle that snaps along a horizontal track. That's exactly what the workshop builds.

A subtle but important rule: `setInteractive({ draggable: true })` makes the object draggable for the **scene**, not for any specific pointer. If the player presses with the mouse and then taps with a finger, only one drag is active at a time — Phaser hands the drag to whichever pointer started it.

Two pitfalls that catch every beginner:

1. **Listening on the scene instead of the object.** `this.input.on("drag", …)` fires for *every* draggable in the scene, with the dragged object passed as a fourth argument. The per-object listener (`card.on("drag", …)`) is usually what you want.
2. **Forgetting `setInteractive` returns the game object.** You can chain `card.setInteractive({ draggable: true }).on("drag", …)`. Many code samples in the wild do this.

# --questions--

## --text--

Which call enables dragging on a sprite?

## --answers--

`sprite.setInteractive({ draggable: true })`

---

`sprite.setDraggable(true)`

### --feedback--

There is no `setDraggable` method directly on game objects. The flag is part of `setInteractive`'s config object.

---

`sprite.draggable = true`

### --feedback--

Setting a property doesn't register listeners. You need `setInteractive` so Phaser hit-tests it.

---

`this.input.enableDrag(sprite)`

### --feedback--

There's no `enableDrag` on the input plugin in Phaser 3.

## --text--

You want a card to drag horizontally only — vertical motion should be ignored. What does the `drag` callback look like?

## --answers--

`card.on("drag", (pointer, dragX) => card.setPosition(dragX, card.y));`

---

`card.on("drag", (pointer, dragX, dragY) => card.setPosition(dragX, dragY));`

### --feedback--

That moves on both axes. To lock vertical, ignore `dragY` and pass the existing `card.y`.

---

`card.on("drag", (pointer, dragY) => card.setPosition(card.x, dragY));`

### --feedback--

That locks to vertical, the opposite of what's asked.

---

`card.on("drag", () => card.setPosition(card.x + 1, card.y));`

### --feedback--

That ignores the pointer entirely and moves the card right one pixel per drag tick.

## --text--

What's the difference between `card.on("drag", …)` and `this.input.on("drag", …)`?

## --answers--

The per-object handler fires only when *that* card is dragged; the scene handler fires for every draggable in the scene.

---

The per-object handler fires every frame; the scene handler fires once per pointer release.

### --feedback--

Both fire every frame the pointer moves while down.

---

The scene handler is faster because it skips per-object hit testing.

### --feedback--

Both rely on the same hit testing for drag detection. There is no speed difference.

---

The two listeners fire in undefined order; you should never use both.

### --feedback--

The order is well-defined (scene first, then object) and you can use both safely.
