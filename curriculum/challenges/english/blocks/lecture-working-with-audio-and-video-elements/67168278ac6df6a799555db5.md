# --interactive--

A neural network can only be trusted if its performance on new, unseen data has been properly measured. Splitting your data into three separate sets is the standard way to do this honestly.

### Why You Need Multiple Splits

If you train a model on all available data and then measure its performance on that same data, you will get an unrealistically optimistic score. The model has already seen every example — of course it performs well. You need data the model has never touched to get a true picture of how it will perform in the real world.

### The Three Sets

**Training set** — the data the model learns from. The weights are updated based on this data. Typically 70–80% of the total dataset.

**Validation set** — data the model never trains on, but which is used during training to monitor performance after each epoch. It helps you detect overfitting early and tune hyperparameters like learning rate and network size. Typically 10–15% of the total dataset.

**Test set** — data the model never sees until training is completely finished. It gives the final, unbiased measure of performance. Typically 10–15% of the total dataset. You should only evaluate on the test set once — if you use it to make decisions during development, it becomes contaminated and is no longer a fair measure.

### A Common Split

```python
import tensorflow as tf
import numpy as np

(x_train, y_train), (x_test, y_test) = tf.keras.datasets.mnist.load_data()

# Further split training data to create a validation set
val_size = int(0.1 * len(x_train))  # 10% for validation

x_val   = x_train[:val_size]
y_val   = y_train[:val_size]
x_train = x_train[val_size:]
y_train = y_train[val_size:]

print(f"Train: {len(x_train)}, Val: {len(x_val)}, Test: {len(x_test)}")
# Train: 54000, Val: 6000, Test: 10000
```

Or more simply, using the `validation_split` parameter in `model.fit()`:

```python
model.fit(x_train, y_train, epochs=10, validation_split=0.1)
```

### Reading the Training Curves

During training, a good sign is when both training loss and validation loss decrease together. Problems to watch for:

| Pattern | Meaning |
|---|---|
| Both losses decrease together | Healthy training |
| Training loss decreases, validation loss increases | Overfitting — stop early |
| Both losses remain high | Underfitting — try a larger model |
| Validation loss is lower than training loss | Normal early in training; check data split |

### The Contamination Problem

A subtle but important mistake is **data leakage** — when information from the validation or test set accidentally influences training. Common causes:

- Normalising the entire dataset (including test) using statistics computed from the whole dataset. You should compute mean and standard deviation from the training set only, then apply those same values to validation and test.
- Making model decisions (architecture, hyperparameters) based on test set performance. The test set should be locked away until the very end.

### Key Points

- Training set: used to update weights
- Validation set: used during training to monitor for overfitting and tune hyperparameters
- Test set: used once at the very end to measure final performance
- Never use the test set to make decisions during development
- Data leakage (test information influencing training) produces dishonestly optimistic results
