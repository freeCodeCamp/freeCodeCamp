# --interactive--

The **world** is Phaser's coordinate system for the simulation. It can be much larger than the canvas — your 320 × 240 viewport might look out into a 4000 × 240 side-scrolling level. The **world bounds** define the simulation's outer rectangle: physics bodies are clamped to it (when configured), `worldbounds` events fire when they touch an edge, and Phaser uses it as the spatial reference for collision broad-phase.

The default world bounds match the canvas size. For anything wider than the canvas, you have to set them explicitly:

```js
this.physics.world.setBounds(0, 0, 4000, 240);
```

That declares a 4000 × 240 simulation area starting at `(0, 0)`. The arcade physics engine now knows the level is wide.

To make a body actually stop at the edges, opt in per-body:

```js
this.player.body.setCollideWorldBounds(true);
```

Without this, the player can walk straight off the right side of the world — physics doesn't enforce bounds on a body unless you ask.

:::interactive_editor

```js
const world = { x: 0, y: 0, width: 4000, height: 240 };
const player = { x: 3995, y: 120, width: 12, height: 12, vx: 5, vy: 0 };

function step() {
  player.x += player.vx;
  player.y += player.vy;
  if (player.x + player.width / 2 > world.x + world.width) {
    player.x = world.x + world.width - player.width / 2;
    player.vx = 0;
  }
  if (player.x - player.width / 2 < world.x) {
    player.x = world.x + player.width / 2;
    player.vx = 0;
  }
}

step();
console.log("after step 1:", { ...player });
step();
console.log("after step 2:", { ...player });
```

:::

The first step pushes the player past the right edge, the bounds clamp them flush against it, and the velocity zeroes — exactly what `setCollideWorldBounds(true)` does in the engine.

`setBounds` takes optional arguments to control which sides actually collide. The default is all four; pass `setBounds(0, 0, 4000, 240, true, true, false, false)` to disable the bottom and right edges (a useful trick for endless runners that scroll past the right edge of the world).

A common mistake is moving things via velocity but forgetting to enlarge the world. The level renders correctly because Phaser is happy to draw at any coordinate, but the camera follow + collision feels broken — the player slides into invisible walls at the canvas edge because the physics world is still 320-wide. Always update the world bounds first when you grow the level.

# --questions--

## --text--

You build a 4000-pixel-wide level and wonder why the player runs into an invisible wall at `x = 320`. What did you forget?

## --answers--

`this.physics.world.setBounds(0, 0, 4000, 240)` — the physics world is still the default canvas size.

---

`this.cameras.main.setBounds(0, 0, 4000, 240)` — the camera is the wrong size.

### --feedback--

That fixes the camera, not the wall. The invisible wall is a *physics* clamp, not a camera issue. Both must be set, but the wall is the world bounds.

---

`this.player.setVelocityX(higher value)` — the player is too slow.

### --feedback--

The wall is not about speed; it is a hard clamp. Increasing speed has the player running into it faster.

---

Disable arcade physics and use Matter.js.

### --feedback--

The wall would still exist in Matter for the same reason. Switching engines does not solve the bounds issue.

## --text--

You call `physics.world.setBounds(0, 0, 4000, 240)` but the player still walks off the right edge into negative-coordinate territory. What's missing?

## --answers--

`this.player.body.setCollideWorldBounds(true)` — bounds-clamping is per-body and opt-in.

---

You need to call `physics.world.enable(true)`.

### --feedback--

Physics is already enabled if you used `physics.add.sprite`. The missing piece is per-body bounds.

---

`physics.world.setBoundsCollision(true)`.

### --feedback--

That toggles which world-edge sides are collidable, not whether a specific body respects them.

---

`physics.world.setBounds` clamps automatically.

### --feedback--

It defines the rectangle but does not enforce it on bodies. Each body opts in via `setCollideWorldBounds(true)`.

## --text--

A side-scrolling endless runner streams chunks in from the right and the player should fall off the right edge into the next chunk. Which `setBounds` call gives that behaviour?

## --answers--

`physics.world.setBounds(0, 0, 4000, 240, true, false, true, true)` — top, bottom, left collide; right does not.

---

`physics.world.setBounds(0, 0, 4000, 240, true, true, true, true)` — all sides collide.

### --feedback--

That blocks the player at the right edge — the opposite of what an endless runner wants.

---

`physics.world.setBounds(0, 0, 4000, 240, false, false, false, false)` — no sides collide.

### --feedback--

That removes the floor too; the player would fall forever.

---

`physics.world.setBounds(-Infinity, 0, Infinity, 240)` — open horizontal bounds.

### --feedback--

`setBounds` does not accept `Infinity`. The cleanest pattern is finite bounds with the right edge disabled.
