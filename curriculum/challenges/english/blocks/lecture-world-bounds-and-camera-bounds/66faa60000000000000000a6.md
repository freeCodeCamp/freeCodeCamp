# --interactive--

World bounds and camera bounds are independent, but they almost always need to be set together. They protect against two different failure modes — and forgetting either one produces a player-visible bug.

**Forgetting world bounds:** the player can walk off the right side of the level into empty world coordinates. The level data was 4000 pixels wide, but to the physics simulation the world is still the default canvas size. Bullets, enemies, and the player drift off into negative or far-positive coordinates. Collisions stop firing because the broad-phase quadtree was sized for the old, smaller bounds.

**Forgetting camera bounds:** the level is correctly 4000 pixels wide and the player walks the full length of it, but the camera happily follows them past `x = 4000` into empty world. The screen shows half-level, half-blank. At the start of the level the same problem appears mirrored on the left.

The standard pair, paste-ready:

```js
this.physics.world.setBounds(0, 0, 4000, 240);
this.cameras.main.setBounds(0, 0, 4000, 240);
```

The arguments are usually identical. When they are not, you almost certainly want to know why — non-matching bounds is the kind of thing that produces "physics works but the camera does the wrong thing" or vice versa, and both are confusing to debug.

:::interactive_editor

```js
const world = { x: 0, width: 4000 };
const camera = { x: 0, width: 320, scroll: 0 };
const player = { x: 0, width: 12 };

function tick(steps) {
  for (let i = 0; i < steps; i++) {
    player.x += 50;
    if (player.x + player.width / 2 > world.x + world.width) {
      player.x = world.x + world.width - player.width / 2;
    }
    camera.scroll = player.x - camera.width / 2;
    if (camera.scroll < 0) camera.scroll = 0;
    if (camera.scroll > world.width - camera.width) {
      camera.scroll = world.width - camera.width;
    }
  }
}

tick(100);
console.log("after 100 ticks:", { player: player.x, scroll: camera.scroll });
```

:::

The simulation still works after the player tries to push past the right edge: world bounds clamp the player; camera bounds clamp the scroll. Both clamps fired at the same place because both bounds were the same.

A useful diagnostic when something looks off: dump both bounds to the console.

```js
console.log("world:", this.physics.world.bounds);
console.log("camera:", this.cameras.main.getBounds());
```

If they disagree by accident, the bug is now obvious.

There is one legitimate case to use different bounds: a HUD area at the top or bottom of the screen. The cleanest pattern there is a separate UI camera that ignores the gameplay layer (Chapter 6 Module 2 covers multi-camera setups). Resist the temptation to fix HUD layout by shrinking the gameplay camera bounds — you'll fight the camera every time you change the canvas size.

## --text--

A 4000-pixel-wide level: the player can walk to the end, but the camera follows them past `x = 4000` and shows blank world. What's missing?

## --answers--

`this.cameras.main.setBounds(0, 0, 4000, 240)` — camera bounds, separate from world bounds.

---

`this.physics.world.setBounds(0, 0, 4000, 240)` — world bounds.

### --feedback--

You said the player can walk to the end, so collisions are working. The problem is the camera, which is governed by *camera* bounds.

---

`this.cameras.main.setZoom(2)` — zoom in to hide the empty area.

### --feedback--

That hides one symptom but the underlying bug remains and breaks for any wider screen.

---

`this.player.body.setCollideWorldBounds(true)`.

### --feedback--

The player is already being clamped (they can walk to the end without escaping). The camera is the missing piece.

## --text--

In what scenario would world bounds and camera bounds intentionally differ?

## --answers--

Almost never. The few legitimate cases (HUD reservations, scrolling chunked levels) are better handled with a second camera or with disabled-side flags. Matching bounds is the boring, correct default.

---

Whenever the player is faster than the camera.

### --feedback--

Camera speed is governed by lerp, not bounds.

---

Always — they measure unrelated things.

### --feedback--

They are independent in implementation but they describe the same level. Matching bounds is the right baseline 99% of the time.

---

Whenever zoom is enabled.

### --feedback--

Zoom does not require mismatched bounds; it changes how the bounds rectangle is rendered.

## --text--

When debugging "the camera shows blank space at the level edge," which two values would you log first?

## --answers--

`this.physics.world.bounds` and `this.cameras.main.getBounds()` — visually compare to confirm they match.

---

`this.player.x` and `this.player.y`.

### --feedback--

Useful, but the bug is bounds-related. Player position alone does not reveal a bounds mismatch.

---

`game.scale.gameSize` and `game.config.width`.

### --feedback--

Those are about the canvas, not the playable area. Camera scroll bugs are in bounds.

---

`scene.events` and `cameras.main.events`.

### --feedback--

Events fire when something happens; the bug is static configuration.
