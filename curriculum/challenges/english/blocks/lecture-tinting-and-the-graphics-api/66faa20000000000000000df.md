# --interactive--

Sometimes the right asset isn't a sprite. A debug box around a hitbox, a damage-number floater, a circle that follows the player — these all want to be code-drawn, not painted. Phaser exposes that through `Phaser.GameObjects.Graphics`.

A `Graphics` object is an immediate-mode canvas you can draw into:

```js
const g = this.add.graphics();
g.lineStyle(2, 0xffffff, 1);   // 2px white outline, full alpha
g.strokeCircle(240, 180, 24);  // ring at scene center, radius 24
g.fillStyle(0xff3366, 0.6);    // pink fill, 60% alpha
g.fillRect(10, 10, 100, 20);   // top-left rectangle
```

Three core ideas:

- **State sticks until you change it.** `lineStyle` and `fillStyle` set the current pen; every subsequent draw call uses them. To draw two shapes with different colors, set the style again before the second shape.
- **Drawing accumulates.** Every call appends to the object's internal command list. To erase and start over, call `g.clear()`. This is the pattern for HUD elements that redraw every frame.
- **One Graphics ≈ one batch.** Every `Graphics` instance you create is a separate render call. Don't make a thousand tiny ones; make one and draw a thousand shapes into it.

The full vocabulary you'll reach for most often:

- `lineBetween(x1, y1, x2, y2)` — straight line.
- `strokeRect`, `fillRect` — rectangles.
- `strokeCircle`, `fillCircle` — circles.
- `strokeRoundedRect`, `fillRoundedRect` — rounded rectangles.
- `strokePoints(points, true)` — outline a polygon from an array of `{ x, y }`.
- `fillPath` / `strokePath` — close the current path and stroke or fill it.

A trap that catches everyone once: `Graphics` doesn't have an `x`/`y` you can set the way a sprite does. The shapes you draw are placed in the coordinates you pass to the draw call. If you want to move the whole graphics object, use `g.setPosition(x, y)` and draw your shapes relative to (0, 0) instead of a world position.

```js
const ring = this.add.graphics();
ring.setPosition(this.player.x, this.player.y);
ring.lineStyle(2, 0xffffff, 1);
ring.strokeCircle(0, 0, 24);   // local coords; setPosition handles the world offset
```

That pattern — set position once per frame, draw at local 0, 0 — is how you make a `Graphics` follow a moving sprite without re-clearing and re-drawing every frame.

When you do redraw every frame (e.g. a debug visualization that changes shape), the loop is `g.clear() → g.lineStyle/fillStyle → g.strokeXxx/fillXxx`. Skipping the `clear()` is the single most common bug — your shapes stack up and the canvas turns into a smear.

# --questions--

## --text--

You want a HUD box that re-renders every frame to show the player's current health. What's the right shape of your `update` loop?

## --answers--

`g.clear()`, then `g.fillStyle(...)`, then `g.fillRect(...)` — clear before redrawing.

---

`g.fillStyle(...)` then `g.fillRect(...)` — Phaser auto-clears each frame.

### --feedback--

It does not auto-clear; you have to call `g.clear()` yourself or shapes pile up.

---

`g.destroy()` then `this.add.graphics()` again every frame.

### --feedback--

Destroying and recreating every frame allocates garbage and tanks performance.

---

`g.setVisible(false)` then `g.setVisible(true)` — hide while redrawing.

### --feedback--

Visibility doesn't reset the command list; you'd still need `clear`.

## --text--

You want a glowing ring to follow the player. Which approach minimizes per-frame work?

## --answers--

Create one `Graphics`, draw the ring at local (0, 0), and call `setPosition(player.x, player.y)` each frame.

---

Create a new `Graphics` each frame at the player's position.

### --feedback--

Allocates a new game object every frame — guaranteed garbage-collection thrash.

---

Draw the ring at the player's world position and never move the graphics object.

### --feedback--

Then the ring stays at the spot where the player started, not where they are.

---

Draw the ring once and add a tween that translates the graphics with `tweens.add({ targets: g, x: ... })` every frame.

### --feedback--

A tween isn't built for an unbounded follow; `setPosition` directly is the right tool.

## --text--

In a single `Graphics` object you draw a red filled circle and then a blue filled circle, with no `fillStyle` call between them. What color is the second circle?

## --answers--

Red — the fill style stays at red until you call `fillStyle` again.

---

Blue — Phaser auto-detects the next intended color.

### --feedback--

Phaser doesn't infer colors from anywhere; the current pen is whatever you last set.

---

Black — the second draw call resets the fill.

### --feedback--

Draws don't reset the fill style.

---

Both circles share the average color — pink.

### --feedback--

Phaser doesn't blend pen colors across draw calls.
