# --interactive--

A tween is the engine's way of saying "interpolate this property from its current value to a target value over time." You stop writing per-frame `update` math and let Phaser do the easing. Internally it boils down to one call: `this.tweens.add({ ... })`.

Seven knobs cover ~95% of every tween you'll ever write:

| Field      | Type                                | Job                                                                                |
| ---------- | ----------------------------------- | ---------------------------------------------------------------------------------- |
| `targets`  | object \| object[]                  | What gets animated. Sprites, plain objects, even literals like `{ value: 0 }`.     |
| `props`    | inline keys (`x`, `alpha`, â€¦)       | Which fields to interpolate, and to what value.                                    |
| `duration` | ms                                  | How long the tween takes from current â†’ target.                                    |
| `ease`     | name string                         | Curve shape: `Linear`, `Cubic.easeOut`, `Bounce.easeOut`, `Elastic.easeIn`, ...    |
| `repeat`   | int (`0` default, `-1` = forever)   | How many extra cycles after the first. `-1` loops forever.                         |
| `yoyo`     | boolean                             | After hitting the target, animate back to the start. Combined with `repeat: -1`, you get an infinite ping-pong. |
| `delay`    | ms                                  | Wait this long before starting.                                                    |

A typical "pop-in" tween:

```js
this.tweens.add({
  targets: this.gem,
  scale: { from: 0, to: 1 },
  duration: 250,
  ease: "Back.easeOut"
});
```

The `{ from, to }` form bypasses "start from current value" and forces an explicit start point â€” useful when the gem was just instantiated and you want to guarantee it pops from `scale: 0` regardless of the current state.

Phaser also gives you `onStart`, `onUpdate`, `onComplete` callbacks. Use `onComplete` to trigger the next animation, destroy the target, or unlock input â€” just don't put gameplay logic *during* `onUpdate`, that's what `update()` is for.

:::interactive_editor

```js
function tweenDescriptor(opts) {
  const props = Object.entries(opts).filter(
    ([k]) => !["targets", "duration", "ease", "yoyo", "repeat", "delay", "onComplete"].includes(k)
  );
  return {
    target: opts.targets?.constructor?.name ?? "object",
    animatedProps: props.map(([k, v]) => `${k} â†’ ${JSON.stringify(v)}`),
    duration: opts.duration ?? 1000,
    ease: opts.ease ?? "Power0",
    yoyo: !!opts.yoyo,
    repeat: opts.repeat ?? 0
  };
}

console.log(tweenDescriptor({
  targets: { x: 0 },
  x: 200,
  duration: 600,
  ease: "Cubic.easeOut"
}));

console.log(tweenDescriptor({
  targets: { alpha: 1 },
  alpha: 0,
  duration: 200,
  yoyo: true,
  repeat: -1
}));
```

:::

The first descriptor is a one-shot slide. The second is an infinite alpha ping-pong â€” useful for "hint" pulses on a viable swap in match-3.

A common trap: forgetting that tweens *mutate* their targets directly. After the tween finishes, `this.gem.scale` is `1`, not "back to whatever it was before." If you need the value preserved, stash it before the tween (`const original = this.gem.scale`) and restore in `onComplete`, or use `yoyo: true` if it's a one-shot symmetric animation.

# --questions--

## --text--

Why prefer `tweens.add({ targets, scale, duration })` over manually incrementing `gem.scale` in `update()`?

## --answers--

Tweens are declarative â€” Phaser handles the per-frame interpolation, easing, and frame-rate independence for you. Your `update` stays free for game logic, and the easing curve gives you "feel" for free.

---

`update()` runs slower than tweens.

### --feedback--

`update()` runs every frame and is just as fast. The win is in correctness and clarity, not raw speed.

---

You can't combine tweens with arcade physics.

### --feedback--

You can â€” physics moves the body, tweens animate visual properties like `scale`, `tint`, or `alpha`.

---

Tweens skip rendering when off-screen.

### --feedback--

Off-screen culling happens at the renderer level, regardless of tween source.

## --text--

What does `repeat: -1` paired with `yoyo: true` produce?

## --answers--

An infinite ping-pong: the target animates to the goal, back to the start, to the goal again, forever â€” until you call `tween.stop()` or destroy the target.

---

A tween that runs once then stops.

### --feedback--

`repeat: -1` is "infinite repeats", not "no repeats". Default `repeat: 0` means one play.

---

A tween that always runs in slow motion.

### --feedback--

Slow motion is `time.timeScale`, not a tween config option.

---

A tween that throws an error after one cycle.

### --feedback--

Phaser silently honors any negative repeat as "forever".

## --text--

You instantiate a gem at `scale: 1` and tween `scale` to `0` with `duration: 200, onComplete: () => gem.destroy()`. Why is this safer than `gem.destroy()` alone?

## --answers--

The 200ms scale-down gives the player a visual confirmation that the gem matched. An instant destroy reads as a "glitch" â€” the brain doesn't have time to register the cause.

---

`destroy()` throws an error if called on a sprite without a tween.

### --feedback--

`destroy()` works on any game object regardless of tween state.

---

Phaser caches sprites and refuses to free memory without a tween.

### --feedback--

Phaser frees the memory either way; the tween is purely cosmetic.

---

`scale: 0` improves performance versus immediate destruction.

### --feedback--

Both end at zero render cost. The win is communication to the player, not performance.
