# --interactive--

A single tween animates one *segment* of motion. Real animations are usually multi-segment: gem swaps to position, then scale-pops, then the score text counts up. You can manually chain via `onComplete`, but it gets ugly fast — three nested callbacks for a three-stage animation.

Phaser 3.60+ has `this.tweens.chain({ targets, tweens: [...] })`. It's a tween manager that runs an array of mini-tweens back-to-back on the same target, automatically waiting for each to finish before starting the next.

```js
this.tweens.chain({
  targets: gem,
  tweens: [
    { x: 200, duration: 200, ease: "Cubic.easeOut" },
    { scale: 1.4, duration: 80, yoyo: true },
    { y: gem.y - 30, duration: 250, ease: "Bounce.easeOut" }
  ]
});
```

Three stages, no callbacks. The gem slides right, pops, then bounces up. Each stage is a separate tween config — same fields as `tweens.add` (`duration`, `ease`, `yoyo`, etc.).

A different option, `tweens.add({ ... })` with `delay`, can fake a chain when stages animate *different* properties:

```js
this.tweens.add({ targets: gem, x: 200, duration: 200 });
this.tweens.add({ targets: gem, scale: 1.4, duration: 80, yoyo: true, delay: 200 });
this.tweens.add({ targets: gem, y: gem.y - 30, duration: 250, delay: 280 });
```

Three independent tweens with computed `delay` values. This works for parallel-with-timing patterns but breaks if any stage's duration changes — you'd have to re-math every later `delay`. `tweens.chain` solves that by pulling the next start-time from the previous tween's actual end.

For a 50-step match-3, you'll use chains in two places:

| Use case        | Pattern                                                                |
| --------------- | ---------------------------------------------------------------------- |
| Cascade drop    | `tweens.chain({ tweens: dropStages })` — gems fall, settle, bounce.    |
| Special creation| `tweens.chain({ tweens: scaleUpStages })` — pulse, pulse, glow, settle. |

:::interactive_editor

```js
function chainTimeline(stages) {
  let elapsed = 0;
  return stages.map((s) => {
    const start = elapsed;
    elapsed += (s.delay ?? 0) + (s.duration ?? 0);
    return { ...s, startsAt: start, endsAt: elapsed };
  });
}

console.log(chainTimeline([
  { name: "slide",  duration: 200 },
  { name: "pop",    duration: 80, yoyo: true },
  { name: "bounce", duration: 250 }
]));
```

:::

Note `yoyo: true` doubles a stage's effective duration in the timeline — Phaser's chain accounts for this internally. The naive computation above does not, which is one reason to prefer `chain` over hand-rolling `delay` values.

`tweens.chain` returns a single TweenChain object. You can call `.stop()` on it to halt the entire sequence, or pass `onComplete` at the chain level to fire when all stages finish. Useful for "unlock input after the cascade settles" — wrap the entire cascade in a chain and resume input on its `onComplete`.

# --questions--

## --text--

Why use `tweens.chain` instead of stacking `tweens.add` with computed `delay` values?

## --answers--

`tweens.chain` reads the next stage's start-time from the previous stage's actual end, so editing one duration doesn't force you to recompute every downstream `delay`. It's also one object you can `stop()` or attach an `onComplete` to.

---

`tweens.chain` is faster.

### --feedback--

The two paths perform identically; the chain just pulls the math out of your code.

---

Phaser refuses to run more than three concurrent tweens.

### --feedback--

There's no such limit; you can run hundreds.

---

`delay` doesn't exist in Phaser 3.

### --feedback--

It does — and you'll use it for one-off staggered enters. Chains are for sequential pipelines on one target.

## --text--

In a match-3 cascade, ten gems drop simultaneously to fill empty cells. Should each gem be its own chain or share one chain?

## --answers--

Each gem gets its own chain (or a single tween) because they animate the same property (`y`) but on different targets. `tweens.chain` runs stages serially on a *single* target — it can't fan out to ten gems in parallel.

---

One chain with all ten gems as `targets`.

### --feedback--

`targets` accepts an array, but the chain still runs each *stage* serially on the whole array — all ten gems would have to drop, then all ten scale, etc. Not the parallel drop you want.

---

Three chains, one per column.

### --feedback--

Closer, but doesn't generalize — eight columns means eight chains. Per-gem (or one drop tween per gem) is simpler.

---

A single `tweens.add` with all ten gems as `targets`.

### --feedback--

That's actually the right call when all stages are identical. The wrong answer here was "use a chain for parallel motion" — chains are for *sequence*, not parallelism.

## --text--

You want a 200ms scale-down then a 100ms color flash on the same gem. The flash should *not* start until the scale-down finishes. Best approach?

## --answers--

`tweens.chain({ targets: gem, tweens: [{ scale: 0, duration: 200 }, { tint: 0xffffff, duration: 100 }] })` — sequencing on one target is exactly the chain use case.

---

Two parallel `tweens.add` calls.

### --feedback--

Parallel calls would both start at frame 0, so the flash would compete with the scale-down for the entire 200ms. You wanted sequence, not overlap.

---

Inside `update()`, decrement `gem.scale` then bump `gem.tint`.

### --feedback--

Manual interpolation in `update` is what tweens replace. You're working harder than the framework requires.

---

Use `setTimeout` to schedule the flash 200ms later.

### --feedback--

DOM timers don't account for `time.timeScale` or scene pauses. Use Phaser's `time.delayedCall` or the chain — both stay locked to the scene clock.
