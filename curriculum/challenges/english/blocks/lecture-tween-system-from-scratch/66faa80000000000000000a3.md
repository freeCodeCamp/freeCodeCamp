# --interactive--

`ease` shapes the *feel* of a tween. Same `duration`, same `targets` — different `ease` and the gem drops like a feather (`Sine.easeOut`), springs into place (`Back.easeOut`), or smashes into the floor (`Bounce.easeOut`). Picking the right curve is design, not math.

The ease takes a normalized time `t` in `[0, 1]` and returns a normalized progress `[0, 1]`. `Linear` returns `t` unchanged. Every other curve bends that line.

The curves you'll reach for in match-3:

| Family    | When to use it                                                                  |
| --------- | ------------------------------------------------------------------------------- |
| `Linear`  | Steady drift. Almost never the right call — looks robotic.                      |
| `Cubic`   | Most "natural" easing. `easeOut` for entrances, `easeIn` for exits.             |
| `Sine`    | Softest curve. Subtle bobs, slow drifts, hint pulses.                           |
| `Back`    | Overshoots and settles. Pop-ins, gem-creation flourishes.                       |
| `Elastic` | Big overshoot with oscillation. Special-gem creation. Use sparingly.            |
| `Bounce`  | Multi-step bouncy landing. Coin-drops, gems hitting the floor of the playfield. |

Pair them with a direction:

- `easeIn`: starts slow, ends fast. Use for objects *leaving* a state.
- `easeOut`: starts fast, ends slow. Use for objects *entering* a state.
- `easeInOut`: slow at both ends, fast in the middle. Use for objects passing through.


A practical rule: 90% of the time, `easeOut` is the right choice — entering motion settles, which feels stable. `easeIn` looks unfinished unless the next thing happens fast.

```js
this.tweens.add({ targets: gem, scale: { from: 0, to: 1 }, duration: 250, ease: "Back.easeOut" });
this.tweens.add({ targets: gem, scale: 0, duration: 200, ease: "Cubic.easeIn" });
this.tweens.add({ targets: gem, y: floorY, duration: 600, ease: "Bounce.easeOut" });
```

Three of the most common patterns:

- Gem creation pop: `Back.easeOut` so it settles past 100% then returns.
- Match destruction: `Cubic.easeIn` so it starts slow then accelerates into nothing.
- Cascade drop: `Bounce.easeOut` so it lands with weight.


:::interactive_editor

```js
function easeName(value, ease) {
  const curves = {
    "Linear":         (t) => t,
    "Cubic.easeOut":  (t) => 1 - Math.pow(1 - t, 3),
    "Cubic.easeIn":   (t) => t * t * t,
    "Sine.easeOut":   (t) => Math.sin((t * Math.PI) / 2),
    "Bounce.easeOut": (t) => {
      if (t < 1 / 2.75) return 7.5625 * t * t;
      if (t < 2 / 2.75) { t -= 1.5 / 2.75; return 7.5625 * t * t + 0.75; }
      if (t < 2.5 / 2.75) { t -= 2.25 / 2.75; return 7.5625 * t * t + 0.9375; }
      t -= 2.625 / 2.75; return 7.5625 * t * t + 0.984375;
    }
  };
  return curves[ease](value);
}

[0, 0.25, 0.5, 0.75, 1].forEach((t) => {
  console.log(t.toFixed(2), {
    linear: easeName(t, "Linear").toFixed(3),
    cubic:  easeName(t, "Cubic.easeOut").toFixed(3),
    sine:   easeName(t, "Sine.easeOut").toFixed(3),
    bounce: easeName(t, "Bounce.easeOut").toFixed(3)
  });
});
```

:::

Notice `Cubic.easeOut(0.5)` is already `0.875` — the tween is 87.5% done at the halfway point. That's the "fast in, settles slow" feel. `Linear` reads as `0.500` at `t = 0.5` — boringly even.

`Bounce.easeOut` reads as `0.766` at `t = 0.5`, then `0.926` at `t = 0.75`, then back to `0.984` at `t = 1` — those non-monotonic samples are the bounces. The eye reads them as discrete impacts on a surface.

A common over-use: `Elastic` on every animation. It's loud — the overshoot is ~30% past the target before settling. Reserve it for *announcements* (special gem creation, level-up, win screen). Daily-use animations should be `Cubic.easeOut`.

# --questions--

## --text--

Why does `Back.easeOut` feel "snappy" while `Sine.easeOut` feels "soft"?

## --answers--

`Back.easeOut` overshoots past the target (~10%) before pulling back, giving a small visual "snap" at the end. `Sine.easeOut` is monotonic — it slows smoothly into the target with no overshoot, so the eye reads it as gentle.

---

`Back.easeOut` is faster.

### --feedback--

Both finish at the same time; the difference is the curve shape, not duration.

---

`Sine.easeOut` is buggy in Phaser 3.80.

### --feedback--

It's stable and well-tested.

---

Browser rendering rounds `Sine` curves to whole pixels.

### --feedback--

Sub-pixel rendering applies the same to both.

## --text--

A gem just matched and is being destroyed via `tweens.add({ scale: 0, duration: 200, ease: ??? })`. Which `ease` reads as "the gem is *leaving*"?

## --answers--

`Cubic.easeIn` — starts slow, accelerates as it disappears. Reads as "fading away" / "leaving the stage".

---

`Bounce.easeOut`.

### --feedback--

Bouncing reads as *arrival*, not departure. Wrong direction.

---

`Back.easeOut`.

### --feedback--

`Back.easeOut` overshoots past the target. Visually it'd scale below zero then back to zero — not the "leaving" feel.

---

`Linear`.

### --feedback--

Linear works but is the most boring choice. `Cubic.easeIn` sells the departure better.

## --text--

For a "hint pulse" that gently scales a viable swap up and down forever to draw the eye, which combination is correct?

## --answers--

`{ scale: 1.1, duration: 600, ease: "Sine.easeInOut", yoyo: true, repeat: -1 }` — soft sine curve, ping-pongs between scales, runs forever until you stop it.

---

`{ scale: 1.1, duration: 600, ease: "Bounce.easeOut", yoyo: true, repeat: -1 }`.

### --feedback--

Bounce reads as impact, not breath. You want soft for a hint pulse.

---

`{ scale: 1.1, duration: 600, ease: "Linear", yoyo: false, repeat: -1 }`.

### --feedback--

`yoyo: false` won't pulse — it'd just snap back to scale 1 each loop. And linear feels mechanical.

---

`{ scale: 1.1, duration: 600, ease: "Sine.easeInOut", yoyo: true, repeat: 1 }`.

### --feedback--

`repeat: 1` plays exactly twice. For "until the player makes a move", you want `repeat: -1` and stop the tween manually.
