# --interactive--

## Pointer events

Phaser papers over `mousedown`, `touchstart`, and `MSPointerDown` with one **pointer** API. The same listener fires whether the user clicks with a mouse, taps with a finger, or presses with a stylus. The events live on `this.input` inside any scene and the callback receives a `Pointer` object whose `x`/`y` are already in scene coordinates.

:::interactive_editor

```js
function create() {
  this.cursor = this.add.circle(0, 0, 6, 0xffffff);
  this.input.on("pointermove", pointer => {
    this.cursor.setPosition(pointer.x, pointer.y);
  });
}
```

:::

For per-object input, opt the game object into the input pipeline with `setInteractive()` and listen on the object directly. The global scene listener still fires before the per-object handler — that order is deterministic.

:::interactive_editor

```js
function create() {
  const button = this.add.rectangle(120, 90, 80, 32, 0x4488ff);
  button.setInteractive();
  button.on("pointerdown", () => {
    button.setFillStyle(0xff8844);
  });
}
```

:::

A single touch is `pointer1`. To track more, opt in with `this.input.addPointer(n)` once and Phaser exposes `pointer2`, `pointer3`, … with the same API. Each has a stable `id` you can use to track which finger started a gesture.

## Drag and swipe

Drag is a flag on `setInteractive`. Set `{ draggable: true }` and Phaser dispatches `dragstart`, `drag`, and `dragend` on the object. The `drag` event hands you the pointer's current `(dragX, dragY)` in scene coordinates — set them on the object and the gesture works.

:::interactive_editor

```js
function create() {
  const knob = this.add.circle(120, 90, 12, 0xffaa44);
  knob.setInteractive({ draggable: true });
  knob.on("drag", (pointer, dragX, dragY) => {
    knob.setPosition(dragX, dragY);
  });
}
```

:::

A swipe is a drag interpreted as a single gesture: capture the pointer at `pointerdown`, measure the delta at `pointerup`, and only count it if the distance crossed a threshold within a time budget. Anything below the threshold is "tap"; anything slower than the budget is "drag".

:::interactive_editor

```js
function create() {
  const SWIPE_PX = 40;
  const SWIPE_MS = 400;
  let startX, startY, startT;
  this.input.on("pointerdown", pointer => {
    startX = pointer.x;
    startY = pointer.y;
    startT = pointer.downTime;
  });
  this.input.on("pointerup", pointer => {
    const dx = pointer.x - startX;
    const dy = pointer.y - startY;
    const dt = pointer.upTime - startT;
    if (Math.hypot(dx, dy) >= SWIPE_PX && dt <= SWIPE_MS) {
      const dir = Math.abs(dx) > Math.abs(dy)
        ? (dx > 0 ? "right" : "left")
        : (dy > 0 ? "down" : "up");
      console.log("swipe", dir);
    }
  });
}
```

:::

## Action-map keyboard input

Hardcoding `cursors.left.isDown` everywhere couples gameplay to a single device. The action-map pattern flips the relationship: each **action** the game cares about (`left`, `right`, `launch`, `pause`) maps to a list of key names, and an `isDown(action)` helper resolves the list at runtime. Adding WASD support is one line; adding gamepad support later is one resolver. Gameplay code never branches on device.

:::interactive_editor

```js
const ACTIONS = {
  left: ["LEFT", "A"],
  right: ["RIGHT", "D"],
  launch: ["SPACE"]
};

function create() {
  this.cursors = this.input.keyboard.createCursorKeys();
  this.keys = this.input.keyboard.addKeys("A,D,SPACE");
  this.player = this.add.rectangle(120, 90, 16, 16, 0xffffff);
}

function update() {
  if (this.isDown("left")) this.player.x -= 2;
  else if (this.isDown("right")) this.player.x += 2;
}

function isDown(action) {
  return ACTIONS[action].some(name => {
    const k = this.keys[name] || this.cursors[name.toLowerCase()];
    return k && k.isDown;
  });
}
```

:::

The same pattern lets a player rebind controls: persist the `ACTIONS` object to `localStorage` and load it on boot. The gameplay code does not change.

## Virtual on-screen controls

A virtual D-pad needs its own scene. Running it as a parallel `Scene.launch`'d scene keeps the buttons on top of gameplay without touching `MainScene`'s code, lets you rerun gameplay without rebuilding controls, and isolates touch handlers so they cannot fight `MainScene`'s pointer listeners. Communication between the two scenes runs through the `MainScene.events` emitter — the controls scene emits, the gameplay scene listens.

:::interactive_editor

```js
class MainScene extends Phaser.Scene {
  constructor() { super("MainScene"); }
  create() {
    this.scene.launch("ControlsScene");
    this.events.on("action:left", () => { this.targetX -= 30; });
    this.events.on("action:right", () => { this.targetX += 30; });
    this.targetX = 120;
    this.player = this.add.rectangle(this.targetX, 160, 16, 16, 0xffffff);
  }
  update() {
    this.player.x = Phaser.Math.Linear(this.player.x, this.targetX, 0.3);
  }
}

class ControlsScene extends Phaser.Scene {
  constructor() { super("ControlsScene"); }
  create() {
    const left = this.add.rectangle(40, 160, 60, 40, 0x556677, 0.5).setInteractive();
    const right = this.add.rectangle(200, 160, 60, 40, 0x556677, 0.5).setInteractive();
    left.on("pointerdown", () => {
      this.scene.get("MainScene").events.emit("action:left");
    });
    right.on("pointerdown", () => {
      this.scene.get("MainScene").events.emit("action:right");
    });
  }
}
```

:::

Three places — and only three — should still branch on device: button **layout** (a D-pad takes screen real estate that desktop does not have), tutorial **hint copy** ("Tap to fire" vs "Press SPACE to fire"), and input **tuning** (touch usually wants a touch more lerp smoothing than keyboard). The action map and the gameplay loop stay device-agnostic.
