# --interactive--

A tilemap is a level laid out as a 2D grid of small repeating images. Each cell of the grid points at one image — a **tile** — pulled from a single shared sheet — the **tileset**. The map data is just a 2D array of integers; the tileset is a single PNG; the renderer pastes each tile into the correct cell every frame.

That decoupling is the whole point. The map is small, plain, and editable in any text editor or visual tool; the art lives in one PNG that the loader caches once. A 60×40 dungeon is 2,400 numbers and one image — not 2,400 separate sprite objects.

```js
const TILE = 16;
const tileset = "dungeon.png";
const map = [
  [12, 12, 12, 12, 12],
  [12,  0,  0,  0, 12],
  [12,  0,  0,  0, 12],
  [12, 12, 12, 12, 12]
];
```

Each integer is a **global tile id** (gid). Tiled, the editor we use, numbers tiles starting at `1` (id `0` is reserved for "empty"). The gid points at one tile inside one tileset. With a single 12×10 tileset of 120 tiles, gid `1` is the top-left tile of the sheet and gid `13` is the start of the second row.

:::interactive_editor

```js
function renderRoom(map, tilesPerRow) {
  const out = [];
  for (let row = 0; row < map.length; row++) {
    const cells = [];
    for (let col = 0; col < map[row].length; col++) {
      const gid = map[row][col];
      if (gid === 0) {
        cells.push(".");
      } else {
        const localId = gid - 1;
        const tx = localId % tilesPerRow;
        const ty = Math.floor(localId / tilesPerRow);
        cells.push(`(${tx},${ty})`);
      }
    }
    out.push(cells.join(" "));
  }
  return out.join("\n");
}

const map = [
  [13, 13, 13, 13, 13],
  [13,  1,  1,  1, 13],
  [13,  1,  2,  1, 13],
  [13, 13, 13, 13, 13]
];
console.log(renderRoom(map, 12));
```

:::

The reason tiles repeat is data density. A 60×40 room has 2,400 cells; if each cell were a unique image, you would ship 2,400 PNGs. With one tileset of ~120 unique tiles you ship one PNG, the player's GPU caches one texture, and the JSON map is just gid integers. That is why retro and modern 2D games alike — Zelda, Stardew Valley, Hollow Knight — are built on tilemaps.

There is one more thing the integer carries: a **flip flag**. Tiled stores horizontal/vertical/diagonal mirroring in the high bits of the gid, so `(12 with flip-x) ≠ (12 without)`. You rarely have to think about this directly — Phaser unpacks it for you — but it is why tilemap data is sometimes stored as `unsigned int` rather than a plain integer.

# --questions--

## --text--

Why do tilemaps store integers per cell instead of an image per cell?

## --answers--

Because every cell points at one tile from a single shared tileset image, so the map is small and the GPU caches one texture.

---

Because pixel data does not fit in JSON.

### --feedback--

Pixel data fits in JSON as base64; that is not the reason.

---

Because integers render faster than images at the WebGL level.

### --feedback--

Rendering still draws images — the integers are just lookups. The win is one cached texture, not faster integers.

---

Because the WebGL spec forbids more than one image per draw call.

### --feedback--

Made up. Modern WebGL routinely batches many textures per draw.

## --text--

In Tiled, you see a tile id of `0` in your map data. What does it mean?

## --answers--

Empty cell — Tiled reserves gid `0` to mean "no tile here". The first real tile in your tileset has gid `1`.

---

The first tile of the first tileset.

### --feedback--

That is gid `1`. The reserved `0` slot is the empty cell.

---

A glitch — Tiled never emits zero.

### --feedback--

It does, on every empty cell.

---

The flip-x flag set on tile `0`.

### --feedback--

Flip flags live in the high bits of a non-zero gid; gid `0` itself means empty.

## --text--

You have a `12 × 10` tileset (120 tiles). What is the local tile id of gid `25`?

## --answers--

`24` — subtract the firstgid (`1`) from the gid; that gives the 0-based index into the tileset.

---

`25` — gids are already 0-based.

### --feedback--

Tiled gids start at `1` for the first tileset; the local id is gid minus firstgid.

---

`13` — divide by the column count.

### --feedback--

Column count is for converting a local id to (col,row), not gid to local id.

---

It depends on whether the tile is collidable.

### --feedback--

Collidability is a property; it does not affect the id math.
