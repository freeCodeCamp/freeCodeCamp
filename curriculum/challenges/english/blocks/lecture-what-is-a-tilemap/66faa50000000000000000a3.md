# --interactive--

Tile layers are great for terrain. They are terrible for anything that has to be addressed by name — the player's spawn point, the door that leads to the next room, the chest that drops the key. Tiled gives you a second kind of layer, the **object layer**, for non-grid data.

An object layer holds a list of `Object` records. Each record has an id, a free-form `name`, a free-form `type` (sometimes called `class` in newer Tiled exports), an `(x, y)`, and a `width` / `height`. Optional `properties` carry per-object data — door target, coin value, NPC dialog key. Objects are not stored on the grid; they are stored at exact pixel coordinates and can sit anywhere.

```js
const objects = [
  { id: 1, name: "spawn", type: "spawn", x: 32,  y: 192 },
  { id: 2, name: "door",  type: "door",  x: 304, y: 112 },
  { id: 3, name: "coin1", type: "coin",  x: 80,  y: 48  }
];
```

The `type` field is how the runtime decides what to do with each object. The map author writes `"door"` once in Tiled; the game code reads `o.type === "door"` to decide whether to spawn an interactive door, a coin pickup, or an NPC. Naming the same type consistently across every map is what lets one piece of game logic handle every door in the dungeon.

:::interactive_editor

```js
function spawnFromObjects(objects) {
  const result = { player: null, doors: [], coins: [] };
  for (const o of objects) {
    if (o.type === "spawn") {
      result.player = { x: o.x, y: o.y };
    } else if (o.type === "door") {
      result.doors.push({ id: o.id, x: o.x, y: o.y, target: o.properties?.target });
    } else if (o.type === "coin") {
      result.coins.push({ id: o.id, x: o.x, y: o.y });
    }
  }
  return result;
}

const objects = [
  { id: 1, type: "spawn", x: 32,  y: 192 },
  { id: 2, type: "door",  x: 304, y: 112,
    properties: { target: "room-02" } },
  { id: 3, type: "coin",  x: 80,  y: 48 },
  { id: 4, type: "coin",  x: 240, y: 48 }
];

console.log(spawnFromObjects(objects));
```

:::

Phaser exposes object lookup through `map.findObject(layerName, predicate)` (returns the first match) and `map.getObjectLayer(layerName).objects` (returns every object). The standard pattern is one `for ... of` loop over the array, dispatched by `o.type`. That keeps level data clean — the map says *what* spawns, the code says *how* it spawns.

Objects also unlock things tile layers cannot do. A door is naturally a 16×32 rectangle (taller than it is wide). A trigger zone might be 64×16. An NPC might be a single point with `point: true` set. Tile layers force everything onto the same grid; object layers do not. Anything that does not fit cleanly on the grid — anything that has a *name* — belongs on an object layer.

# --questions--

## --text--

What does the `type` field on a Tiled object do?

## --answers--

It tags the object with a category so the runtime can dispatch on it (e.g. `"spawn"`, `"door"`, `"coin"`).

---

It tells Phaser which sprite sheet to use.

### --feedback--

Object types are a free-form label — Phaser does not auto-load any sprite from them.

---

It changes how the object is rendered in the editor.

### --feedback--

Tiled may color-code by type, but the practical purpose is runtime dispatch.

---

It must be `"rectangle"`, `"ellipse"`, or `"point"`.

### --feedback--

Those are *shape* flags, separate from `type`. `type` is a free-form string.

## --text--

You want to place a 16×32 door inside a 16×16-tile world. Which kind of layer should hold it?

## --answers--

An object layer — objects are positioned in pixel coordinates and can have any size, so a 16×32 rectangle fits.

---

A tile layer — every game world piece is a tile.

### --feedback--

Tile layers force everything onto the grid; a 16×32 region would have to be split across two tiles.

---

A new tilemap loaded just for the door.

### --feedback--

You almost never need a separate tilemap for a single object. That is what object layers are for.

---

A `decoration` tile layer with a custom shader.

### --feedback--

Custom shaders for one door is wildly over-engineered.

## --text--

Reading the door's destination room, you write `door.target`. Phaser returns `undefined`. Why?

## --answers--

Custom Tiled properties live under `door.properties` (an array of `{name, type, value}` records); you read `door.properties.find(p => p.name === "target").value` or use Phaser's helper to convert the array to an object.

---

Tiled does not export custom properties on objects.

### --feedback--

Tiled exports them, just nested under `properties`.

---

Phaser strips object properties at load time.

### --feedback--

Phaser preserves them; you just have to read from the right place.

---

Custom properties have to be valid JavaScript identifiers and `target` is reserved.

### --feedback--

`target` is fine. The issue is the lookup path.
