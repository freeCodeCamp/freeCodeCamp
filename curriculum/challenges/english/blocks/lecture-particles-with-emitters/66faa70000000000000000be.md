# --interactive--

Particles in Phaser 3.60+ are a single call:

```js
const emitter = this.add.particles(x, y, key, config);
```

That's the whole API. You pass:

1. `x, y` — emitter origin in world coordinates.
2. `key` — texture key for each particle (a small image; can be runtime-generated).
3. `config` — an object that controls how each particle is born and dies.

The texture is critical: every particle is one quad with that texture. So a 4×4 white circle texture gives bright cinematic dots, a 16×16 sparkle texture gives stars, and so on. For a brawler hit you want something cheap and bright — a 4×4 white circle is usually all you need.

The cleanest way to produce that texture without shipping an asset is `Graphics.generateTexture` in your `BootScene`:

```js
class BootScene extends Phaser.Scene {
  preload() {
    this.load.audio("sfx-hit", ["audio/sfx-hit.ogg", "audio/sfx-hit.mp3"]);
  }

  create() {
    const g = this.add.graphics();
    g.fillStyle(0xffffff, 1);
    g.fillCircle(4, 4, 4);
    g.generateTexture("particle", 8, 8);
    g.destroy();
    this.scene.start("MainScene");
  }
}
```

Now `"particle"` is available as a texture key everywhere. No HTTP request, no decode, runs the same on every device.

The minimal "explosion" emitter:

```js
this.add.particles(enemy.x, enemy.y, "particle", {
  speed: { min: 100, max: 300 },
  lifespan: 400,
  quantity: 1,
  frequency: -1
}).explode(20);
```

`frequency: -1` means "do not auto-emit" — the emitter spawns nothing until you call `.explode(N)`. Combined with `.explode(20)`, that's exactly 20 particles in one burst, then silence.

For a continuous emitter, omit `frequency: -1` and set `quantity: 2` (or whatever rate you want):

```js
this.flames = this.add.particles(torch.x, torch.y, "particle", {
  speed: { min: 50, max: 150 },
  lifespan: 800,
  quantity: 2,
  scale: { start: 0.6, end: 0 },
  alpha: { start: 1, end: 0 }
});
```

This streams 2 particles per frame, each fading out over 800ms.

:::interactive_editor

```js
function describeEmitter(config) {
  const isOneShot = config.frequency === -1;
  const out = isOneShot ? "burst" : "continuous";
  return {
    type: out,
    particlesPerSecond: isOneShot ? "n/a" : (1000 / 16) * (config.quantity ?? 1),
    avgLifespanMs: config.lifespan,
    speedRange: config.speed
  };
}

console.log(describeEmitter({ frequency: -1, lifespan: 400, speed: { min: 100, max: 300 } }));
console.log(describeEmitter({ quantity: 2, lifespan: 800, speed: { min: 50, max: 150 } }));
console.log(describeEmitter({ quantity: 4, lifespan: 200, speed: { min: 200, max: 500 } }));
```

:::

The first config is a one-shot burst, perfect for hit feedback. The second is a continuous low-rate stream, perfect for a torch. The third is a continuous high-rate stream — looks great for fire/sparks, but watch the particle count: 4/frame × 60 FPS × 200ms lifespan = ~48 live particles at any time. That's fine. 4/frame × 60 FPS × 2000ms lifespan = 480 — that's where you start seeing FPS drops on phones.

Particles ALSO require explicit destruction when no longer needed. Continuous emitters live forever unless you destroy them:

```js
this.flames.destroy(); // when the torch is extinguished
```

For one-shot emitters from `.explode(N)`, Phaser auto-destroys the emitter after all particles complete — usually you don't have to think about it.

# --questions--

## --text--

You want to fire 20 particles in a single burst on every enemy hit, then never again. Which config?

## --answers--

`{ speed: ..., lifespan: ..., frequency: -1 }` then call `.explode(20)`.

---

`{ speed: ..., lifespan: ..., quantity: 20 }` and let it run.

### --feedback--

That streams 20 *per frame* continuously. After 1 second you have 1200 live particles instead of 20.

---

`add.particles(...).pause()` after one frame.

### --feedback--

That works mechanically but spams unnecessary state and only stops new particles after at least one frame. `frequency: -1` with `.explode()` is the canonical pattern.

---

`add.particles(..., { burst: true })`.

### --feedback--

There is no `burst` config option in 3.60+. The pattern is `frequency: -1` + `.explode(N)`.

## --text--

What does `quantity: 4` mean for a continuous emitter (no `frequency: -1`)?

## --answers--

4 particles spawned every frame Phaser ticks (~60 per second), so ~240 particles per second.

---

A total of 4 particles will spawn before the emitter stops.

### --feedback--

That's one-shot behaviour. Continuous emitters keep spawning.

---

Particles spawn at most every 4ms.

### --feedback--

`frequency` controls the inter-spawn delay; `quantity` controls per-tick count.

---

Each particle has 4× the visual weight.

### --feedback--

Quantity is a count, not a multiplier on size or alpha.

## --text--

You ship a continuous flame emitter with `quantity: 4` and `lifespan: 2000`. Phones drop to 30 FPS. What's the cheapest fix?

## --answers--

Cut `lifespan` to ~400ms. Live particle count is roughly `quantity × FPS × lifespan / 1000`, so dropping lifespan from 2000 to 400 reduces the count 5× and the GPU load roughly proportionally. Also reduce `quantity` to 2 if needed.

---

Switch to a one-shot `.explode()` per frame.

### --feedback--

That actually creates more emitters per second and worsens performance.

---

Use a smaller texture (1×1 pixel).

### --feedback--

Texture size barely affects fill rate when particle count is the bottleneck.

---

Move particles to a UI scene.

### --feedback--

Scenes don't change rendering cost; total particles still draw on the same GPU.
