# --interactive--

Two patterns dominate particle usage. Knowing which is which saves you from leaking emitters or misfiring effects.

**One-shot burst** — fires N particles immediately, then auto-destroys when the last one dies:

```js
this.add
  .particles(enemy.x, enemy.y, "particle", {
    speed: { min: 100, max: 300 },
    lifespan: 400,
    quantity: 1,
    frequency: -1
  })
  .explode(20);
```

Use for: hit reactions, pickups, explosions, jump-dust, footstep puffs. The emitter is created and discarded in one breath. You don't need a reference. Phaser cleans it up.

**Continuous emitter** — keeps spawning until you explicitly stop or destroy it:

```js
this.flames = this.add.particles(torch.x, torch.y, "particle", {
  speed: { min: 50, max: 150 },
  lifespan: 800,
  quantity: 2,
  scale: { start: 0.6, end: 0 },
  alpha: { start: 1, end: 0 }
});

// ...later, when the torch is extinguished:
this.flames.destroy();
```

Use for: torches, fountains, smoke trails, weather effects. You hold the reference because YOU'RE responsible for stopping it.

The mistake every junior dev makes: confusing the two.

```js
// Wrong: stale reference accumulates after each hit
onHit(enemy) {
  this.particles = this.add.particles(enemy.x, enemy.y, "particle", {...});
  this.particles.explode(20);
}
// Each onHit OVERWRITES the previous reference. The old emitters are
// auto-destroyed (they're one-shot) so this isn't a leak, but the variable
// name `this.particles` is misleading. Drop the assignment:
onHit(enemy) {
  this.add.particles(enemy.x, enemy.y, "particle", {...}).explode(20);
}
```

```js
// Worse: leaking continuous emitters
onHit(enemy) {
  this.add.particles(enemy.x, enemy.y, "particle", {
    quantity: 2  // ← no frequency: -1, this is CONTINUOUS
  });
}
// Each hit creates a new emitter that runs forever. After 100 hits you have
// 100 emitters spawning 200 particles per frame. Game grinds to a halt.
```

The fix for the leak: always pair "I want a burst" with `frequency: -1` + `.explode()`.

```js
// Right:
onHit(enemy) {
  this.add
    .particles(enemy.x, enemy.y, "particle", {
      ...,
      frequency: -1
    })
    .explode(20);
}
```

A useful diagnostic: check for "I'm in a callback that fires often (onHit, onCollision, on every key press) AND I'm calling `add.particles(...)`". If the answer is yes, you better be calling `.explode()` AND have `frequency: -1`. Otherwise audit the lifetime.

For really hot paths (≥60 emissions/sec), even one-shot bursts can be expensive due to emitter creation/teardown overhead. The fix: pre-create one continuous emitter with `frequency: -1, quantity: 1`, hold the reference, and call `.explode(20)` on it for each hit. Same visual, much less allocation.

```js
class MainScene extends Phaser.Scene {
  create() {
    this.hitParticles = this.add.particles(0, 0, "particle", {
      speed: { min: 100, max: 300 },
      lifespan: 400,
      quantity: 1,
      frequency: -1
    });
  }

  onHit(enemy) {
    this.hitParticles.setPosition(enemy.x, enemy.y);
    this.hitParticles.explode(20);
  }
}
```

This is the pattern shipped in modern Phaser games for high-frequency effects.

# --questions--

## --text--

You want a 1-time particle burst when the player picks up a coin. Which is the simplest correct call?

## --answers--

`this.add.particles(coin.x, coin.y, "particle", { ..., frequency: -1 }).explode(20)` — one-shot, auto-destroyed.

---

`this.particles = this.add.particles(coin.x, coin.y, "particle", { quantity: 20 })`.

### --feedback--

That creates a continuous emitter spawning 20 particles per frame forever, plus stores a reference you might forget to destroy.

---

`this.add.particles(coin.x, coin.y, "particle", { ..., quantity: 20 }).pause()`.

### --feedback--

`pause()` stops emission but leaves the emitter alive in the scene. Inelegant.

---

`this.tweens.add({ targets: coin, duration: 200, onComplete: () => coin.destroy() })`.

### --feedback--

That's a tween, not a particle effect; it doesn't spawn debris.

## --text--

Inside an `onHit` callback that fires 5+ times per second, you call `this.add.particles(x, y, "particle", { quantity: 2 })` (no `frequency: -1`). What happens?

## --answers--

Each hit creates a new continuous emitter that lives forever, spawning 2 particles per frame indefinitely. After 100 hits you have 100 emitters × 2 particles/frame = 200 spawn events per frame, plus thousands of live particles.

---

Phaser auto-deduplicates emitters at the same position.

### --feedback--

It does not.

---

The emitter auto-destroys after 1 second.

### --feedback--

There is no auto-timeout on continuous emitters.

---

Each new call cancels the previous emitter.

### --feedback--

`add.particles` is additive; previous emitters keep going.

## --text--

For high-frequency hit effects (60+ per second), what's the optimal pattern?

## --answers--

Pre-create one continuous emitter with `frequency: -1, quantity: 1` once in `create`. Hold the reference. On each hit, `setPosition(x, y)` then `.explode(20)`. Avoids per-hit emitter allocation.

---

Use 60 separate `add.particles(...).explode()` calls per frame.

### --feedback--

That's the slow path; emitter creation/teardown adds up.

---

Use a tween instead of particles.

### --feedback--

Tweens don't spray debris; they animate properties on existing objects.

---

Use a Graphics object instead.

### --feedback--

Graphics is even slower for repeated draws and lacks per-particle physics.
