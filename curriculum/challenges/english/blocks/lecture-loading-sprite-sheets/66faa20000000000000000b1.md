# --interactive--

Defining an animation with `this.anims.create` registers it. Playing it is a separate call on the sprite itself.

Every `Phaser.GameObjects.Sprite` (or `Phaser.Physics.Arcade.Sprite`) has an `anims` controller. The methods you'll touch every workshop:

```js
sprite.anims.play("walk-down");          // start the animation
sprite.anims.play("walk-down", true);    // start, but ignore if it's already this animation
sprite.anims.stop();                     // stop on the current frame
sprite.anims.pause();                    // pause and remember position
sprite.anims.resume();                   // resume from where pause left it
```

The second argument to `play` is `ignoreIfPlaying`. It defaults to `false`, which means every call to `play("walk-down")` restarts the animation at frame 0 — even if it's already playing the same key. Inside an `update` loop that fires 60 times a second, that restart ticks the animation back to frame 0 every frame and the sprite freezes. Pass `true` and Phaser checks "am I already playing this animation? If yes, skip — keep going."

A typical movement pattern looks like:

```js
update() {
  if (this.cursors.left.isDown) {
    this.player.setVelocityX(-120);
    this.player.setFlipX(false);
    this.player.anims.play("walk-left", true);
  } else if (this.cursors.right.isDown) {
    this.player.setVelocityX(120);
    this.player.setFlipX(true);
    this.player.anims.play("walk-right", true);
  } else {
    this.player.setVelocityX(0);
    this.player.anims.play("idle", true);
  }
}
```

`setFlipX(true)` mirrors the sprite horizontally. If your art has a left-facing walk cycle and you want a right-facing one, you can either author both directions or author one and flip. Flipping is cheaper and looks identical for symmetric characters; for asymmetric characters (a sword on the right hip), authoring both reads better.

You can react to animation events:

```js
this.player.on("animationcomplete", (anim, frame) => {
  console.log("finished:", anim.key);
});

this.player.on("animationcomplete-attack", () => {
  this.player.anims.play("idle");
});
```

The first listens to every animation finishing. The second narrows to a single key by name. Useful for one-shot animations: play `attack`, return to `idle` when it finishes.

One last detail: sprites accept a starting frame at creation time. `this.physics.add.sprite(x, y, "hero", 0)` shows frame 0 of the hero sheet — useful when you want the sprite to look right *before* the player presses any key, in the gap before any animation has played.

## --text--

Why does `this.player.anims.play("walk-left", true)` differ from `this.player.anims.play("walk-left")` inside an `update` loop?

## --answers--

The `true` flag is `ignoreIfPlaying`; without it, the animation restarts at frame 0 every tick and the sprite appears frozen.

---

The `true` flag plays the animation in reverse.

### --feedback--

Reverse playback is controlled by `yoyo` or by reversing the frame array, not by this flag.

---

The `true` flag forces the animation onto the GPU.

### --feedback--

Phaser doesn't expose a CPU-vs-GPU flag through `play`.

---

The two calls are equivalent.

### --feedback--

They differ in restart behavior — that's the entire point of the flag.

## --text--

Which method on a sprite mirrors it horizontally without changing its position?

## --answers--

`setFlipX(true)`

---

`setRotation(180)`

### --feedback--

Rotation tips the sprite around — that flips top-to-bottom too.

---

`setOrigin(-1, 0)`

### --feedback--

Origin offset controls anchor point, not mirroring.

---

`setScale(-1, 1)`

### --feedback--

`setScale(-1, 1)` happens to mirror it, but it also negates other transforms; `setFlipX` is the explicit, idiomatic API.
