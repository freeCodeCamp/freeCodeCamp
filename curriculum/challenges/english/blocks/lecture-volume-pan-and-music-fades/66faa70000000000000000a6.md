# --interactive--

When the level transitions from "calm exploration" to "boss arena", cutting one track and starting another sounds amateurish. You crossfade: the outgoing track tweens its volume to `0` while the incoming track tweens up from `0` to its target volume. With a short overlap the listener hears one continuous flow, not a join.

Phaser's tween system drives any numeric property of a `Sound` instance, including `volume`:

```js
function crossfade(scene, outgoing, incoming, durationMs = 1500, targetVolume = 0.6) {
  incoming.setVolume(0);
  incoming.play({ loop: true });

  scene.tweens.add({
    targets: outgoing,
    volume: 0,
    duration: durationMs,
    onComplete: () => outgoing.stop()
  });

  scene.tweens.add({
    targets: incoming,
    volume: targetVolume,
    duration: durationMs
  });
}
```

The first tween fades out and stops the old track on completion (`stop` returns the buffer to a clean state — call it after the fade so the listener doesn't hear an abrupt mute). The second tween fades the new track in. Both run for the same duration, so the curves cross at `durationMs / 2`.

A few production notes.

**Linear vs equal-power.** A simple linear fade has a perceived dip at the midpoint because both tracks are at half volume there but human loudness is logarithmic. The fix is to use a square-root fade — `volume = Math.sqrt(t)` for the incoming track and `volume = Math.sqrt(1 - t)` for the outgoing one. Phaser tweens accept an `ease` function, so you can pass a custom one:

```js
scene.tweens.add({
  targets: incoming,
  volume: targetVolume,
  duration: durationMs,
  ease: t => Math.sqrt(t)
});
```

For most arcade games linear is fine; for music-driven games equal-power is worth the tiny extra effort.

**Sync the playheads.** If both tracks are the same tempo and you want them to *layer* (e.g. add a percussion track on top of the same pad), start the new track at the same `seek` position as the old one. `outgoing.seek` reads the current playback time in seconds; pass it on `play`:

```js
incoming.play({ seek: outgoing.seek, loop: true, volume: 0 });
```

**Don't tween a sound that's about to be destroyed.** A common bug: `outgoing.destroy()` runs before the tween finishes. The tween then writes to a destroyed instance and throws. Either destroy in `onComplete` or use `outgoing.stop()` (safe; tween still completes).

:::interactive_editor

```js
function crossfadeCurves(t) {
  return {
    linearOut: 1 - t,
    linearIn: t,
    eqPowerOut: Math.sqrt(1 - t),
    eqPowerIn: Math.sqrt(t),
    midpointDipLinear: 1 - t + t,
    midpointSumEqPower: Math.sqrt(1 - t) ** 2 + Math.sqrt(t) ** 2
  };
}

console.log("at t=0.5:", crossfadeCurves(0.5));
```

:::

The `midpointSumEqPower` line shows the property: with equal-power curves, the sum of squared volumes is always `1`, so the perceived loudness stays flat across the fade. With linear curves the same sum is `0.5` at the midpoint — quieter than either track alone. Train your ear; once you hear it, you can't unhear it.

**Mute-aware crossfade.** If the player has muted music globally (`scene.sound.mute = true`), a crossfade still fires both tweens but no audio plays. That's fine. The state stays correct, so unmuting later resumes both tracks at the right volumes. Avoid skipping the tween entirely when muted; it desyncs your "what's currently playing" model.

## --text--

You crossfade two music tracks with linear `0 → 1` and `1 → 0` volume tweens. At the midpoint the music sounds noticeably quieter. Why?

## --answers--

Perceived loudness is roughly proportional to the square root of amplitude, so two tracks at `0.5` linear sum to a perceived `~0.7` instead of `1`. The fix is an equal-power curve, e.g. `Math.sqrt(t)`.

---

Phaser introduces a deliberate dip to mark the transition.

### --feedback--

It does not. The dip is a perceptual property of linear crossfades.

---

The audio context drops samples mid-fade.

### --feedback--

Web Audio doesn't drop samples; the curves are doing what you asked.

---

The browser caps the sum of two simultaneous sounds.

### --feedback--

Chrome and Firefox use linear summation. The cap isn't the issue.

## --text--

After the outgoing track fades to `0`, what should you do with it?

## --answers--

Call `stop()` on it (or `destroy()` if you'll never reuse the instance) — leaving it at volume `0` keeps the buffer alive and the tween system tracking it.

---

Nothing — the tween already removed it from the manager.

---

Call `setVolume(1)` to reset before storing it.

### --feedback--

That'd play it at full volume the next frame; you're storing for later, not playing.

---

### --feedback--

`add` creates a new instance; the old one stays orphaned.

## --text--

You want a percussion track to layer on top of the calm pad in time, sharing the same tempo and seek position. Which call?

## --answers--

`incoming.play({ seek: outgoing.seek, loop: true, volume: 0 })` — start the new track at the outgoing track's current playhead so the beats line up, then fade it in.

---

`incoming.play()` and rely on the tracks being the same length.

### --feedback--

Same length isn't enough — the playhead matters. The new track would start from `0` while the old one's beat is mid-bar.

---

`incoming.play({ rate: outgoing.rate })`.

### --feedback--

Rate matches tempo if the tracks are different speeds, but doesn't sync the position.

---

`incoming.play({ delay: outgoing.duration - outgoing.seek })`.

### --feedback--

That waits for the old track to finish — the opposite of layering.
