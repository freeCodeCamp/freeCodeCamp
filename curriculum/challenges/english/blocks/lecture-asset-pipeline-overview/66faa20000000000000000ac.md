# --interactive--

In the space-shooter workshop, you drew the player, enemies, and bullets from a handful of inline SVGs. That works for crisp geometric shapes. It does not work for animation.

A walking character looks like motion only because you flip through several still drawings — frames — fast enough that your eye fills in the in-between. Loading each frame as a separate image works, but it forces the browser to make one HTTP request per frame and uses one GPU texture per frame. That gets expensive fast.

The fix is a **sprite sheet**: a single image that holds many fixed-size frames laid out on a grid. The browser fetches one file. The GPU uploads one texture. Your game tells the renderer "draw the rectangle at row 2, column 3."

Here is the mental model. A 64×64 sprite sheet split into 16×16 cells is a 4×4 grid holding 16 frames, indexed left-to-right, top-to-bottom: row 0 is frames `0..3`, row 1 is frames `4..7`, row 2 is frames `8..11`, and row 3 is frames `12..15`.

If row 0 is the walk-down cycle, row 1 is walk-up, row 2 is walk-left, and row 3 is walk-right, you have a complete 4-direction walk cycle in one file.

Every modern 2D game framework — Phaser, Godot, Unity, Pixi — speaks this language. Phaser exposes it through `this.load.spritesheet`, which you'll meet in the next lecture.

The trade-off: every frame must be the same size. If your characters need wildly different frame sizes (a hulking boss that occupies a 64×96 space alongside a 16×16 enemy), you'll outgrow the fixed grid. That's when you reach for a texture atlas, which the next challenge unpacks.

# --questions--

## --text--

Why does using a sprite sheet outperform loading each animation frame as a separate image file?

## --answers--

The browser fetches one file and the GPU uses one texture, instead of one of each per frame.

---

Sprite sheets are compressed more aggressively than individual PNGs.

### --feedback--

Compression matters, but it isn't the primary win.

---

Sprite sheets render larger sprites than individual files.

### --feedback--

The on-screen size has nothing to do with how the file is loaded.

---

Sprite sheets allow each frame to use a different size.

### --feedback--

That's the *opposite* of what a sprite sheet's grid layout enforces.

A sprite sheet is 64 pixels wide and 32 pixels tall, sliced into 16×16 frames. How many frames does it contain, and how are they indexed?

## --answers--

16 frames in a 4×4 grid, indexed 0..15.

### --feedback--

That's a 64×64 sheet. This one is only 32 pixels tall.

---

8 frames, indexed 0..7 left-to-right, top-to-bottom.

---

4 frames, indexed 0..3 in a single row.

### --feedback--

Two rows, not one — count both.

---

16 frames, indexed 0..15 in a single row.

### --feedback--

A 64×32 sheet at 16×16 frames cannot hold 16 frames.

## --text--

What is the main constraint a sprite sheet places on your art?

## --answers--

Every frame on the sheet must have the same width and height.

---

Every frame must use the same color palette.

### --feedback--

Sprite sheets have no palette restriction.

---

Every frame must depict the same character.

### --feedback--

A single sheet can hold any mix of characters and props as long as the cell size matches.

---

Every frame must be a perfect square.

### --feedback--

Frames can be any rectangle — they just all have to match.
