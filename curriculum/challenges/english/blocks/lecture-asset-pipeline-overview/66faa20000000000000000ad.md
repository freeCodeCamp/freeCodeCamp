# --interactive--

A sprite sheet packs frames on a fixed grid: every frame is the same width and height, addressed by a single integer. That fixed grid is the source of every win — single texture, predictable indexing, trivial loading code — and the source of its single weakness: the grid forces every frame to occupy the same rectangle.

A **texture atlas** keeps the "one image, many frames" idea but drops the grid. Each frame can be any size and live anywhere on the sheet. A small JSON or XML file ships next to the image and tells the engine where each frame lives.

Here is what an atlas frame entry looks like:

```json
{
  "frames": {
    "hero-idle-0": { "frame": { "x": 0, "y": 0, "w": 16, "h": 24 } },
    "hero-idle-1": { "frame": { "x": 16, "y": 0, "w": 16, "h": 24 } },
    "boss-stomp": { "frame": { "x": 0, "y": 24, "w": 64, "h": 96 } }
  }
}
```

You ask the engine for `"hero-idle-0"` or `"boss-stomp"` by name; the JSON tells the renderer the rectangle to slice out.

Choose between them along three axes:

- **Frame uniformity.** If every animation frame is the same size, a sprite sheet is simpler. If frames vary in size — a tall boss alongside short minions — atlases win.
- **Authoring tools.** Sprite sheets you can paint by hand on a grid. Atlases are produced by packers like TexturePacker, Free Texture Packer, or `texture-packer` on npm; the packer writes both the PNG and the JSON.
- **Asset budget.** Packers are smarter than fixed grids — they pack frames tightly, leaving less wasted transparent space. On large asset sets the savings can be 20-40%.

Phaser supports both with one method change:

- `this.load.spritesheet(key, url, { frameWidth, frameHeight })` for grid sheets.
- `this.load.atlas(key, imageUrl, jsonUrl)` for atlases.

Once loaded, both produce frames you can play through `this.anims.create(...)`. The animation system doesn't care which one fed the frames in.

For Chapter 2 you'll work with sprite sheets — every frame is 16×16, which is exactly the kind of art a fixed grid is designed for. Atlases will return in Chapter 5 when you need to mix tilemap art and character art on a single texture.

# --questions--

## --text--

A sprite sheet has fixed-size cells; an atlas does not. When does the atlas's flexibility matter most?

## --answers--

When frames vary significantly in size — for example, a tall boss alongside short enemies.

---

When the game runs on mobile.

### --feedback--

Both atlases and sheets work on mobile.

---

When the game uses more than one color.

### --feedback--

Color count is unrelated to the layout decision.

---

When the artist prefers to draw with a tablet.

### --feedback--

Drawing tools have no bearing on the atlas-vs-sheet decision.

Which Phaser loader call should you use to load a texture atlas — a PNG paired with a JSON manifest?

## --answers--

`this.load.spritesheet(key, url, { frameWidth, frameHeight })`

### --feedback--

That's the loader for fixed-grid sprite sheets, not atlases.

---

`this.load.image(key, url)`

### --feedback--

`load.image` only loads a single still image; it doesn't read frame metadata.

---

`this.load.atlas(key, imageUrl, jsonUrl)`

---

`this.load.json(key, url)`

### --feedback--

That only loads the JSON; it doesn't bind it to an image.

## --text--

Why does an atlas typically waste less space than a sprite sheet?

## --answers--

A texture packer arranges variable-sized frames tightly, while a fixed grid pads every frame to the largest cell.

---

Atlases are stored in JPEG, which compresses better than PNG.

### --feedback--

Atlases use the same image formats as sprite sheets.

---

Atlases drop transparent pixels at runtime.

### --feedback--

Both formats keep transparent pixels in the image.

---

Atlases automatically downscale frames to half their resolution.

### --feedback--

Neither format changes resolution on its own.
