# --interactive--

Jump buffering is the mirror image of coyote time. Coyote time forgives jumps *after* leaving the ground; jump buffering forgives jumps pressed *before* landing.

The pattern is symmetric. Remember the timestamp of the most recent jump press, then on every frame check whether that press is still "fresh" enough to honor:

```js
const BUFFER_MS = 120;
this.lastJumpPressedAt = -Infinity;

function update(time) {
  if (Phaser.Input.Keyboard.JustDown(this.keys.up)) {
    this.lastJumpPressedAt = time;
  }
  const onGround = body.touching.down || body.blocked.down;
  const bufferActive = time - this.lastJumpPressedAt < BUFFER_MS;
  if (onGround && bufferActive) {
    this.player.setVelocityY(-360);
    this.lastJumpPressedAt = -Infinity;
  }
}
```

:::interactive_editor

```js
function update(time) {
  const body = this.player.body;
  if (Phaser.Input.Keyboard.JustDown(this.keys.up)) {
    this.lastJumpPressedAt = time;
  }
  const onGround = body.touching.down || body.blocked.down;
  if (onGround && time - this.lastJumpPressedAt < 120) {
    this.player.setVelocityY(-360);
    this.lastJumpPressedAt = -Infinity;
  }
}
```

:::

Walk through what happens. The player is in the air. They press jump 80 ms before they land. `lastJumpPressedAt` gets stamped to the press time. The player lands. On the very next frame after touchdown, `onGround` becomes `true`, the buffer is still `< 120` because only 80 ms have passed, and the jump fires — instantly, without the player needing to press again. The reset to `-Infinity` after firing prevents a stale press from triggering a phantom second jump.

Combined with coyote time from the previous lecture, the gate becomes:

```js
const canJump = time - this.lastGroundedAt < COYOTE_MS;
const buffered = time - this.lastJumpPressedAt < BUFFER_MS;
if (canJump && buffered) jump();
```

The two together cover both timing failures the naive check has. From the player's perspective, the game now "feels right" — they intend to jump and the character jumps.

Tuning is again ~60–150 ms. Buffer values noticeably bigger than coyote time start to feel weird ("the character keeps jumping when I land long after I let go"). Roughly equal values are the sweet spot. The platformer workshop uses 120 ms for both.

## --text--

After a buffered jump fires successfully, why do you reset `lastJumpPressedAt` to `-Infinity`?

## --answers--

To prevent the same buffered press from firing a second jump on the next frame the player happens to be grounded.

---

Because Phaser deletes the property otherwise.

To force a new jump animation.

### --feedback--

Animation has nothing to do with the timestamp.

To save memory.

### --feedback--

You set `COYOTE_MS = 120` and `BUFFER_MS = 600`. What goes wrong?

## --answers--

The buffer is too long; the player feels their old jump press "remembered" for half a second after they let go, producing phantom jumps on landing.

---

Coyote time fights with the buffer and the player can never jump.

### --feedback--

The two are independent; they do not interfere mathematically.

Phaser refuses to set `BUFFER_MS` above 500.

### --feedback--

It is a plain JS number — Phaser does not validate it.

### --feedback--

It is one number; the value does not change memory cost.
