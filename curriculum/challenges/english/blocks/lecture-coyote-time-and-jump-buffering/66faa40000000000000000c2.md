# --interactive--

Coyote time is the small grace window during which a player can still jump *after* leaving the ground. The name comes from the cartoon convention where Wile E. Coyote runs off a cliff and has a beat of hang-time before he falls — which is exactly what the technique simulates.

The implementation is a remembered timestamp:

```js
const COYOTE_MS = 120;
this.lastGroundedAt = -Infinity;

function update(time) {
  const onGround = body.touching.down || body.blocked.down;
  if (onGround) this.lastGroundedAt = time;

  const canJump = time - this.lastGroundedAt < COYOTE_MS;
  if (Phaser.Input.Keyboard.JustDown(this.keys.up) && canJump) {
    this.player.setVelocityY(-360);
  }
}
```

:::interactive_editor

```js
function update(time) {
  const body = this.player.body;
  const onGround = body.touching.down || body.blocked.down;
  if (onGround) this.lastGroundedAt = time;
  const canJump = time - this.lastGroundedAt < 120;
  if (Phaser.Input.Keyboard.JustDown(this.keys.up) && canJump) {
    this.player.setVelocityY(-360);
  }
}
```

:::

Read the gate carefully. `time - this.lastGroundedAt < 120` is `true` for as long as it has been less than 120 ms since the player was last on the ground. That includes the *current* frame they are still on the ground (delta 0), and stays true for ~120 ms after they leave. So coyote time gives you "the player can jump if they were grounded recently."

Tuning. Most published platformers settle around 60–150 ms. Below ~60 ms, players still feel cheated. Above ~150 ms, you start getting "ghost jumps" where the player jumps clearly off the ledge into nothing. Celeste shipped at ~110 ms; Hollow Knight at ~80; New Super Mario Bros at ~100.

There is one bug to avoid. After a jump, you must *not* keep updating `lastGroundedAt` until the player lands again — otherwise they could chain coyote-time jumps mid-air. The cleanest way is to clear the timestamp the moment you actually fire the jump: `this.lastGroundedAt = -Infinity` immediately after `setVelocityY`.

Coyote time on its own makes the "leaving the ledge" failure go away. The "pressing jump just before landing" failure still needs jump buffering — that is the next lecture.

# --questions--

## --text--

What problem does coyote time fix?

## --answers--

The player presses jump within ~100 ms after stepping off a ledge but the engine refuses because `body.touching.down` is already `false`.

---

The player can hold jump to fly indefinitely.

### --feedback--

That is variable jump height, a different mechanic.

---

The player gets stuck on corners.

### --feedback--

That is a body-size issue, fixed with `setOffset`.

---

The player double-jumps too easily.

### --feedback--

Double-jump is its own gating, separate from coyote time.

## --text--

Why must you reset `lastGroundedAt` to `-Infinity` (or some past sentinel) the moment a jump fires?

## --answers--

Without it, the player could chain multiple coyote-time jumps in the air during the same grace window.

---

`time` resets to zero each frame.

### --feedback--

`time` is the running scene clock and never resets per frame.

---

Phaser garbage-collects unset properties.

### --feedback--

Phaser does not GC instance properties.

---

`Infinity` is faster than `Number.MAX_VALUE`.

### --feedback--

Both are the same primitive type and have identical performance.
