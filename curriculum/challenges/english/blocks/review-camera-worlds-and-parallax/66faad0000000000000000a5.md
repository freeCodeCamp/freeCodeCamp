## Camera follow, lerp, and deadzone

By default, a Phaser camera sits at world `(0, 0)` and never moves. The first cinematic primitive is `cameras.main.startFollow(target)` — the camera centres on the target every frame, and the world scrolls past on the canvas. That works, but a snap-follow camera reads as rigid; the moment the player moves, the camera teleports.

**Lerp** softens the snap. The 3rd and 4th arguments to `startFollow` are the X and Y smoothing factors, in the range `(0, 1]`:

:::interactive_editor

```js
function create() {
  this.player = this.physics.add.sprite(60, 80, "hero", 12);
  this.cameras.main.startFollow(this.player, true, 0.1, 0.1);
}
```

:::

`0.1` means each frame the camera moves 10% of the way toward the target. Lower numbers feel softer; `1.0` snaps. `0.05–0.15` is the typical range — high enough that the camera keeps up, low enough that small player jitter doesn't ripple the screen.

**Deadzones** ignore small target movements entirely. The argument is a rectangle (in screen space), centred on the camera's current position; while the target stays inside that rectangle, the camera does not move at all.

:::interactive_editor

```js
function create() {
  this.player = this.physics.add.sprite(120, 90, "hero", 12);
  this.cameras.main.startFollow(this.player);
  this.cameras.main.setDeadzone(80, 60);
}
```

:::

The deadzone is the difference between a side-scroller that's stable when the player jumps in place and a side-scroller that wobbles up and down with every hop. Combine `startFollow + lerp + deadzone` for cinematic camera that *responds* without *reacting*.

## World bounds and camera bounds

The **world** is the simulation; the **camera** is the window into it. They have independent bounds, and you almost always need both.

:::interactive_editor

```js
function create() {
  this.physics.world.setBounds(0, 0, 4000, 240);
  this.cameras.main.setBounds(0, 0, 4000, 240);
  this.player = this.physics.add.sprite(60, 120, "hero", 12);
  this.player.setCollideWorldBounds(true);
  this.cameras.main.startFollow(this.player, true, 0.1, 0.1);
}
```

:::

`physics.world.setBounds(x, y, width, height)` declares the simulation rectangle. `body.setCollideWorldBounds(true)` opts a body in to clamping. `cameras.main.setBounds(x, y, width, height)` clamps the camera's scroll. Forget the camera bounds and the camera scrolls past the right edge of the level, exposing the void. Forget the world bounds and the player walks off the level. They're independent; you need both.

A useful rule of thumb: in a side-scroller, world bounds and camera bounds match the level's outer rectangle. In a top-down RPG, they match the room or chunk you're in, and you re-set them on every transition.

## Camera shake, flash, and fade

Three one-line APIs handle every common camera-juice effect.

`shake(duration, intensity)` jitters the camera for a fraction of the viewport size:

:::interactive_editor

```js
function takeDamage() {
  this.cameras.main.shake(200, 0.012);
}
```

:::

`intensity` is the fraction of `(width + height) / 2` to displace at peak. `0.012` on a 320 × 240 viewport is `~3.4 px`; on a 1920 × 1080 it's `~18 px`. Tune at your shipping resolution, not your dev one.

`flash(duration, r, g, b)` paints the viewport in a colour that fades from full opacity to zero over `duration` ms — perfect for hit feedback, level transitions, and "magic happens" moments:

:::interactive_editor

```js
function onPickup() {
  this.cameras.main.flash(120, 102, 221, 255);
}
```

:::

`fade(duration, r, g, b)` is the opposite of `flash` — it fades **in** to a fully opaque overlay that *stays*. Use it for scene transitions; pair it with `cameras.main.fadeIn(...)` on the next scene to come back.

The sweet spots: 80–300 ms for in-game effects, 400–1000 ms for transitions. Anything longer competes with the player's input and reads as broken. Tie juice to discrete events (hits, pickups, level changes), never to continuous state (low health, on fire).

## Multi-camera UI overlays

`setScrollFactor(0)` pins individual objects to the screen so they don't scroll, but it does **not** isolate them from camera-level effects — `cameras.main.shake()` rattles every object that camera renders, including pinned ones. The clean fix is a second camera dedicated to the UI:

:::interactive_editor

```js
function create() {
  this.distanceText = this.add
    .text(8, 8, "Dist: 0")
    .setScrollFactor(0)
    .setDepth(120);
  this.hud = this.add.container(0, 0).setScrollFactor(0).setDepth(120);
  this.hud.add(this.distanceText);
  this.uiCam = this.cameras.add(0, 0, 480, 320);
  this.cameras.main.ignore(this.hud);
  this.uiCam.ignore([this.player, this.platforms]);
}
```

:::

`cameras.add(x, y, width, height)` creates a viewport that, by default, sees every game object in the scene. The two missing pieces are `cameras.main.ignore(uiContainer)` (so world effects don't touch the HUD) and `uiCam.ignore([worldObjects])` (so the HUD camera doesn't redraw the world on top of itself).

The default cameras-render-everything behaviour is the source of the most common multi-camera bug: forget to ignore the world on the UI camera and every world object renders twice — once shaken/scrolled by main, once still by UI. Always wire both ignore lists.

Render order is camera-add order: `cameras.main` exists from scene boot, anything you `add()` after renders **on top**. Within a single camera, `setDepth()` controls layering. The HUD always sits above the world because the UI camera was added later — depth values from the world camera don't compete with depth values from the UI camera.

## Parallax with tileSprite

A `tileSprite` repeats a single texture across an arbitrary rectangle. The rectangle stays put on the canvas; the *contents* scroll via `tilePositionX`:

:::interactive_editor

```js
function create() {
  this.sky = this.add
    .tileSprite(240, 160, 480, 320, "sky")
    .setScrollFactor(0);
  this.mountains = this.add
    .tileSprite(240, 200, 480, 160, "mountains")
    .setScrollFactor(0);
  this.ground = this.add
    .tileSprite(240, 288, 480, 64, "ground")
    .setScrollFactor(0);
}

function update(_t, delta) {
  const dt = delta / 1000;
  this.sky.tilePositionX += 5 * dt;
  this.mountains.tilePositionX += 25 * dt;
  this.ground.tilePositionX += 240 * dt;
}
```

:::

Two complementary tools live behind parallax: **scroll factor** and **`tilePositionX`**. They solve different problems but read almost the same to the player.

`setScrollFactor(0.2)` makes a layer move at 20% of the camera scroll — used when the camera follows a player through a wide world. The far mountains scroll slowly because the camera moves; you don't animate them yourself.

`tilePositionX += speed * dt` makes a layer scroll *on its own* even when the camera doesn't move. This is the right tool for an auto-scrolling endless runner where the player is anchored at fixed X and the world flows past. Multiply by `dt` (`delta / 1000`) so the parallax speed is frame-rate independent.

The cardinal mistake is using both at once — `cameras.main.startFollow(player)` plus `tilePositionX += ...` — which doubles the perceived scroll speed. For an auto-scroller, pick `tilePositionX` and skip the camera follow entirely.

## prefers-reduced-motion

Vestibular sensitivities are real, and shipping unprompted screen shake to every player is a known source of motion-sickness reports. The `prefers-reduced-motion` CSS media query is the platform contract for "this user wants less motion":

:::interactive_editor

```js
function create() {
  this.reduceMotion = window.matchMedia(
    "(prefers-reduced-motion: reduce)"
  ).matches;
}

function takeHit() {
  if (!this.reduceMotion) {
    this.cameras.main.shake(200, 0.012);
  }
}
```

:::

`window.matchMedia('(prefers-reduced-motion: reduce)').matches` returns `true` when the user has set their OS-level "reduce motion" preference. Read it once into `this.reduceMotion`, then gate every shake, flash, and dust-particle emitter behind `if (!this.reduceMotion)`.

`prefers-reduced-motion` is **not** "disable all visuals." Keep the HUD, keep the score updates, keep gameplay-critical flashes. The shipping rule: any motion that exists purely for "feel" — shake, screen-fill flashes that aren't game state, dust trails, screen-edge tints — gets disabled. The player can still tell what hit them; they just don't get the involuntary head-jerk on top.
