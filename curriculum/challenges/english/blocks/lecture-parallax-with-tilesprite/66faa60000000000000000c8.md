# --interactive--

`setScrollFactor` is the dial that turns a static game object into a parallax background.

```js
this.background.setScrollFactor(0.2);
```

A scroll factor of `1.0` (the default) means the object scrolls with the camera one-for-one. The camera moves 100 pixels east; the object's *world* position is unchanged but on screen it appears to move 100 pixels west. Boring trees, platforms, and enemies all use `1.0`.

A scroll factor of `0` means the object **does not scroll with the camera at all**. Wherever you placed it, that's where it stays on screen. HUD elements, fullscreen tints, and pause overlays use `0`.

A scroll factor of `0.2` is the secret sauce. The camera moves 100 pixels east; the object appears to move only 20 pixels east on screen. Far-away mountains scroll slowly because — in real life — they're far away. Pixar uses scroll factor in its 2D parallax sequences. Your game can too with one line.

The math is simple. Phaser computes the camera's *effective* scroll for an object as `effective_scroll = camera.scroll * object.scrollFactor`, then renders the object at `world_position - effective_scroll`. With `scrollFactor = 0.2` the object only "feels" 20% of the camera scroll, so it appears to move slowly relative to the foreground.

For a parallax stack, you assign scroll factors based on perceived distance:

| Layer                 | Scroll factor | Why                  |
| --------------------- | ------------- | -------------------- |
| Sky / sun             | `0.05`        | Practically infinite |
| Far mountains         | `0.2`         | Distant horizon      |
| Mid hills             | `0.5`         | Mid-ground           |
| Foreground silhouette | `0.85`        | Just behind action   |
| Player + ground       | `1.0`         | Foreground           |

These are starting values — tune by eye. The biggest mistake is making the gaps between layers too small (everything looks like it scrolls together) or too large (the layers feel like they slide past each other instead of resolving into a single 3D-ish scene).

:::interactive_editor

```js
const layers = [
  { name: "sky", factor: 0.05 },
  { name: "mountains", factor: 0.2 },
  { name: "hills", factor: 0.5 },
  { name: "foreground", factor: 0.85 },
  { name: "ground", factor: 1.0 }
];

const cameraScroll = 1000;

console.log("camera moved 1000 px right; visible parallax shift:");
for (const l of layers) {
  const apparent = cameraScroll * l.factor;
  console.log(`  ${l.name.padEnd(12)} factor=${l.factor.toFixed(2)} → ${apparent} px`);
}
```

:::

For a `tileSprite` background, scroll factor and `tilePositionX` work together but solve different problems. Scroll factor handles "object world position vs. camera scroll." `tilePositionX` handles "the inside of the tile sprite scrolls forever." A typical sky layer:

```js
this.sky = this.add.tileSprite(240, 160, 480, 320, "sky");
this.sky.setScrollFactor(0);
```

`setScrollFactor(0)` because the tile sprite covers the whole viewport — we never want it to scroll relative to the camera. To animate it, we change `tilePositionX` ourselves in `update`:

```js
update() {
  this.sky.tilePositionX += 0.2;
}
```

That gives auto-scroll independent of the player. For *parallax with the player walking*, set the scroll factor to a small fraction (like `0.2`) and *don't* touch `tilePositionX` — Phaser will handle the parallax automatically as the camera follows the player.

## --text--

A camera scrolls 200 pixels east. An object has `setScrollFactor(0.5)`. How much does it move on screen?

## --answers--

100 pixels — half the camera scroll, because `effective_scroll = 200 * 0.5 = 100`.

---

200 pixels — same as the camera.

### --feedback--

That would be `setScrollFactor(1.0)`. With `0.5`, the object moves at half-speed.

---

50 pixels.

### --feedback--

Check the math: `200 * 0.5 = 100`, not 50.

---

0 pixels — the object stays still.

### --feedback--

That would be `setScrollFactor(0)`.

That's no parallax — every layer scrolls with the camera at the same rate.

## --text--

When should you use `tilePositionX += speed` instead of just setting a scroll factor?

## --answers--

When the layer should scroll on its own (e.g., clouds drifting across a still scene) instead of in response to camera movement.

---

When the layer is too small to fill the viewport.

### --feedback--

That's a sizing issue. `tilePositionX` is about animation, not sizing.

---

When you don't have a `tileSprite`.

### --feedback--

`tilePositionX` only exists on `tileSprite` — if you don't have one, the question doesn't apply.

---

Always — `setScrollFactor` is for HUD only.

### --feedback--

Scroll factor is the primary tool for parallax. `tilePositionX` is additive.
