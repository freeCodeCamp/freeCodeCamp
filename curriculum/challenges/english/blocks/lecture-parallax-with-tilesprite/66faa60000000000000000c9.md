# --interactive--

A real parallax background is **four to five tile sprites stacked back-to-front**. Each layer has its own texture, position, scroll factor, and (sometimes) auto-scroll speed. The art looks expensive; the code is short.

The convention reads back-to-front:

```js
this.sky        = this.add.tileSprite(240, 160, 480, 320, "sky").setScrollFactor(0);
this.mountains  = this.add.tileSprite(240, 200, 480, 160, "mountains").setScrollFactor(0.2);
this.hills      = this.add.tileSprite(240, 240, 480, 100, "hills").setScrollFactor(0.5);
this.ground     = this.add.tileSprite(240, 304, 480, 64, "ground").setScrollFactor(1.0);
```

Three things are happening at once:

1. **Add order = render order.** The sky goes in first, so it ends up at the back. Each subsequent layer renders over the previous one. This is why your stack is back-to-front.
2. **Y position positions the layer vertically.** The sky covers the whole canvas; the ground sits at the bottom. If you draw all four at `y = 160`, the sky covers the others.
3. **Scroll factor sets parallax speed.** Smaller numbers move slower (back). The ground at `1.0` keeps pace with the player.

For an *auto-scrolling* runner where the player is at fixed X and the world flows past, you don't want the camera to actually scroll — there's nothing for it to follow. Instead you animate `tilePositionX` directly:

```js
update(_time, delta) {
  const dt = delta / 1000;
  this.sky.tilePositionX += 5 * dt;
  this.mountains.tilePositionX += 20 * dt;
  this.hills.tilePositionX += 60 * dt;
  this.ground.tilePositionX += 240 * dt;
}
```

Notice the speed ratios mirror what scroll factors would do:

| Layer     | Speed (px/s) | Speed ratio |
| --------- | ------------ | ----------- |
| Sky       | 5            | 1×          |
| Mountains | 20           | 4×          |
| Hills     | 60           | 12×         |
| Ground    | 240          | 48×         |

The exact numbers depend on your art and the feel you want. Multiplying `delta` ensures the parallax speed is frame-rate independent — a 30 fps device and a 144 fps device get the same visual scroll speed.

:::interactive_editor

```js
const layers = ["sky", "mountains", "hills", "ground"];
const speeds = { sky: 5, mountains: 20, hills: 60, ground: 240 };

const dt = 1 / 60;

const positions = Object.fromEntries(layers.map(l => [l, 0]));

for (let frame = 0; frame < 5; frame += 1) {
  for (const l of layers) positions[l] += speeds[l] * dt;
}

console.log("after 5 frames at 60 fps:");
for (const l of layers) {
  console.log(`  ${l.padEnd(10)} tilePositionX = ${positions[l].toFixed(2)}`);
}
```

:::

A few more polish moves you'll see in production parallax:

- **Sub-pixel scroll**: if your art is pixel art, snap to whole pixels (`Math.round`) for crisp scrolling. If it's smooth (gradients, photos), let the values be fractional for buttery scroll.
- **Tint shifting**: change a far layer's tint as the day cycles. Subtle, but adds time-of-day flavor for free. `this.sky.setTint(0xffaa66)` for sunset.
- **Optional cloud layer**: add a fifth `tileSprite` of clouds *in front of* the sky but *behind* mountains, with a unique slow horizontal scroll, for additional depth at near-zero cost.

Common bug: you set up the parallax beautifully, then add a `cameras.main.startFollow(player)` for some other reason. Now both your scroll factors *and* the camera scroll happen — the layers move twice. For an auto-scroller, the rule is: **either the camera follows or the tile sprites animate, never both**.
