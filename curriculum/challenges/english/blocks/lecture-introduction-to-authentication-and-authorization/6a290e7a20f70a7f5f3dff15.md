# --description--

In the last two lectures, you learned how to verify the user identity and maintain their session using cookies. However, automatically sending cookies with every request introduces a unique security vulnerability called Cross-Site Request Forgery, CSRF.

CSRF is an attack that forces an authenticated user to execute unwanted actions on a web application where they are currently logged in. In a CSRF attack, a malicious website tricks the victim browser into making a fraudulent request to your backend server, exploiting the server trust in the user browser cookie.

To understand CSRF, consider this step-by-step example of a banking application:

1. **The Setup:** You log into your bank website, say, `trusted-bank.com`. The bank validates your credentials and stores a session token in your browser cookie.

2. **The Trap:** Without logging out of your bank, you open a new browser tab and visit a malicious website, say `evil-hacker-site.com`.

3. **The Trigger:** The malicious website contains a hidden script or a form that automatically submits a request to the bank backend, such as `POST trusted-bank.com/transfer?amount=1000&to=hacker`.

4. **The Execution:** Because you are still logged into your bank, your browser automatically appends your bank session cookie to that request. The bank server sees a valid cookie, assumes you made the request intentionally, and transfers your money.

The critical takeaway here is that the malicious website cannot read your cookie or steal your data directly. Instead, it simply piggybacks on the fact that your browser automatically attaches cookies to any request going to the target server.

There are two primary defenses: the **SameSite Cookie Attribute** and **CSRF Tokens**.

The simplest defense is setting the `SameSite` attribute on your cookies. This tells the browser when it is allowed to send the cookie along with cross-site requests.

- **SameSite=Strict:** The browser never sends the cookie if the request originates from a third-party website.

- **SameSite=Lax:** The browser allows cookies only when a user safely navigates to the target site, such as clicking a standard link, but blocks them on cross-site form submissions or API requests.

For robust protection, backend frameworks use CSRF tokens. When a user loads a page, the backend generates a random, unique, and unpredictable token and attaches it to the frontend form or session.

When the client submits a request, they must include this token in the custom request headers or form data. The backend compares the submitted token against the token saved in the user session. Because a malicious website cannot access your frontend to copy this secret token, any cross-site request they forge will lack this token and fail validation.

By implementing these strategies, you ensure that requests arriving at your backend are intentional actions made by the actual user, rather than unauthorized actions forced by a malicious player.
