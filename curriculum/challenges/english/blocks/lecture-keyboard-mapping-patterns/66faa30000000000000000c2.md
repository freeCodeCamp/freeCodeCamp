# --interactive--

The third escalation of the action-map pattern is letting the player rebind controls themselves. This is non-negotiable for accessibility (one-handed players, RSI sufferers, players using assistive switches), and it's a notable competitive feature for any reflex-driven game. Once the bindings live in a table separate from gameplay code, exposing that table to the player is mostly a UI problem.

The simplest version stores bindings in `localStorage` so they survive reloads:

```js
const DEFAULT_BINDINGS = {
  left: ['LEFT', 'A'],
  right: ['RIGHT', 'D'],
  fire: ['SPACE']
};

const loadBindings = () => {
  try {
    return JSON.parse(localStorage.getItem('bindings')) ?? DEFAULT_BINDINGS;
  } catch {
    return DEFAULT_BINDINGS;
  }
};

const saveBindings = b => localStorage.setItem('bindings', JSON.stringify(b));
```

The rebinding flow is also simple in concept: pause the game, draw the binding table, and when the player clicks "rebind left" listen for the very next keydown event:

```js
rebind(action) {
  this.add.text(240, 320, `Press a key for ${action}…`).setOrigin(0.5);

  this.input.keyboard.once('keydown', event => {
    this.bindings[action] = [event.code.replace(/^Key/, '')];
    saveBindings(this.bindings);
    this.scene.restart();
  });
}
```

Three things to internalize:

- `event.code` returns names like `KeyA`, `ArrowLeft`, `Space`, `Numpad4`. Strip the `Key` prefix to match Phaser's `addKeys` naming, or simply store `event.code` directly and translate at the call site.
- Use **`once`**, not `on`. Otherwise the next ten keys all rebind the action.
- Restart the scene (or re-run `addKeys`) so `this.keys` reflects the new binding immediately.

A few real-world considerations:

- **Conflict detection.** If the player rebinds "fire" to `A` while `A` is already bound to "left", you should warn them or auto-clear the conflict. Two actions resolving to the same key is a recipe for silent bugs.
- **Reset to defaults.** Always offer this. Players will paint themselves into a corner.
- **Mobile.** The rebinding UI doesn't apply on touch — there's no such thing as a "key" to bind. Hide the rebinding section when `Phaser.Device.os.android || Phaser.Device.os.iOS`.
- **Persistence scope.** `localStorage` is per-origin and per-browser. If the player wants their bindings to follow them across devices, you need a server-side profile.

The action map plus a rebinding UI plus `localStorage` is, for most games, **the entire input customization story**. Twenty lines of code, no plugins.

# --questions--

## --text--

Why use `this.input.keyboard.once('keydown', …)` instead of `on`?

## --answers--

Because you want exactly one keydown to rebind the action; `on` would rebind on every subsequent press.

---

Because `on` doesn't fire for arrow keys.

### --feedback--

`on` fires for every keydown, including arrow keys.

---

Because `on` is async and `once` is synchronous.

### --feedback--

Both are synchronous; `once` simply auto-detaches after the first call.

---

Because `once` is faster.

### --feedback--

Performance is irrelevant here. The reason is the lifecycle, not the perf.

## --text--

What's the right policy when a rebinding would conflict with an existing one?

## --answers--

Either prompt the player to confirm clearing the old binding, or display a "conflict" warning — the worst outcome is two actions silently resolving to the same key.

---

Always overwrite silently — players prefer fewer prompts.

### --feedback--

That's how silent bugs ship. The player rebinds "fire" and discovers they've broken movement on a boss fight.

---

Refuse the rebind without telling the player.

### --feedback--

That's even worse — the player thinks the rebinding screen is broken.

---

Phaser handles conflicts automatically.

### --feedback--

It does not. Conflict detection is your job.

## --text--

Where should the bindings live so they persist across page reloads?

## --answers--

`localStorage`, serialized as JSON.

---

A Redux store.

### --feedback--

Redux is in-memory and resets on reload. You'd still need `localStorage` (or similar) underneath.

---

The browser's `sessionStorage`.

### --feedback--

`sessionStorage` clears when the tab closes. The player will lose bindings overnight.

---

A cookie set by the server.

### --feedback--

Possible, but heavy and requires a server. `localStorage` is the right local-first answer.
