# --interactive--

Almost every PC game ships with both arrows and WASD bindings. Players whose right hand owns the mouse use WASD; players who learned to type without their hands ever leaving the home row often prefer arrows. There is no winner — the only wrong answer is forcing one or the other.

With the action-map pattern from the previous lesson, supporting both is a one-line change to the table:

```js
const ACTIONS = {
  left: ['LEFT', 'A'],
  right: ['RIGHT', 'D'],
  up: ['UP', 'W'],
  down: ['DOWN', 'S']
};
```

You wire it inside `create` once:

```js
this.keys = this.input.keyboard.addKeys('LEFT,RIGHT,UP,DOWN,A,W,S,D');
```

And then nothing in `update` cares which key the player pressed:

```js
if (this.isDown('left')) this.player.x -= 4;
else if (this.isDown('right')) this.player.x += 4;
```

A subtlety worth knowing: `addKeys` accepts a **comma-separated string** _or_ an object (`{LEFT: Phaser.Input.Keyboard.KeyCodes.LEFT}`). Both produce the same `Key` instances. The string form is concise; the object form is what you use when you want to alias a binding (`{leftA: Phaser.Input.Keyboard.KeyCodes.A}`).

Don't forget edge cases:

- **Both keys held at once.** `.some` evaluates to `true` if either is held — exactly what you want. The action stays "active" while at least one key is down.
- **Released order.** If the player presses `A`, then presses `LEFT`, then releases `A`, the action stays active as long as `LEFT` is held. Again, `.some` handles this correctly.
- **Diagonal movement.** With `up` _and_ `left` both held, you'll get `1` on each axis. If you don't want diagonal speed to be `√2` × cardinal speed, normalize the resulting vector after composition (Phaser's `Vector2.normalize()` is your friend).

One more practical tip from shipping games: also bind the **arrow keys + numpad numpad-arrows + WASD + ZQSD** if you want a smooth experience for AZERTY (French) keyboards. ZQSD is the WASD equivalent for left-handed-row French layouts, and adding it is one more line in `ACTIONS`.

## --text--

You add `'A'` to `ACTIONS.left` and the player still can't use `A` to move. What's the most likely cause?

## --answers--

You forgot to include `A` in the `addKeys` call, so `this.keys.A` is `undefined`.

---

Phaser's keyboard plugin has its own action layer that overrides yours.

### --feedback--

It does not. The plugin reads raw key state; the action map is your code.

---

`A` is reserved by Gatsby in development mode.

### --feedback--

Made up. There is no such reservation.

---

You need to use `Phaser.Input.Keyboard.KeyCodes.A` everywhere.

### --feedback--

The string `'A'` is fine for `addKeys` — Phaser accepts both forms.

## --text--

What's the right way to accept `LEFT`, `A`, **and** the numpad's `4` key as a "left" binding?

## --answers--

`{ left: ['LEFT', 'A', 'NUMPAD_FOUR'] }` and add `NUMPAD_FOUR` to `addKeys`.

---

`{ left: ['LEFT|A|NUMPAD_FOUR'] }` — Phaser parses the pipe.

### --feedback--

Phaser does not parse pipes. The array-of-strings form is correct.

---

You can't — `addKeys` only accepts one alias per action.

### --feedback--

`addKeys` accepts as many keys as you list. The action map can resolve as many bindings as you put in the array.

---

Use a regex like `/LEFT|A|NUMPAD_FOUR/` against `event.code`.

### --feedback--

That bypasses Phaser's input system. Stay inside it: register the keys with `addKeys`, then OR them in the action map.

## --text--

Why prefer the action-map approach over `if (cursors.left.isDown || keys.A.isDown)` inline?

## --answers--

Because the gameplay logic and the binding table become independent — you can change one without touching the other.

---

Because `||` short-circuits unpredictably for `Key` instances.

### --feedback--

`||` works fine on the booleans returned by `.isDown`. The reason is structural, not behavioral.

---

Because `cursors` and `addKeys` share state internally and conflict.

### --feedback--

They don't conflict. They both read from the same underlying keyboard plugin.

---

Because Phaser will eventually deprecate `cursors`.

### --feedback--

`createCursorKeys` is not deprecated. Both APIs coexist.
