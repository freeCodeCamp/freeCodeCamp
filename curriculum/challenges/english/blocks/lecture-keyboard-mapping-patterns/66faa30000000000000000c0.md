# --interactive--

Beginners almost always ship the same bug. They write code like this:

```js
if (this.cursors.left.isDown) this.player.x -= 4;
```

Then a player asks for WASD support, and the `if` doubles in size. Then a left-handed player asks for arrows-on-the-numpad. Now there are three branches per direction, and the entire `update` body is twisted into a tree of OR operators that the testers have to walk every time they reproduce a bug.

The fix is to invert the relationship. Instead of asking "is this **key** down?" ask "is the **action** active?" Then map actions to keys in **one** place.

```js
const ACTIONS = {
  left: ['LEFT', 'A'],
  right: ['RIGHT', 'D'],
  fire: ['SPACE']
};

create() {
  this.keys = this.input.keyboard.addKeys('LEFT,RIGHT,A,D,SPACE');
}

isDown(action) {
  return ACTIONS[action].some(name => this.keys[name].isDown);
}

update() {
  if (this.isDown('left')) this.player.x -= 4;
  else if (this.isDown('right')) this.player.x += 4;
  if (this.isDown('fire')) this.tryFire();
}
```

Three things happened:

- The gameplay code reads like the design doc: "if `left` is down, move left." The fact that `left` resolves to two physical keys is hidden behind one method call.
- `ACTIONS` is a plain object you can serialize, log, share between scenes, or persist to `localStorage`. Adding a third bind is a one-line change.
- Tests can stub `isDown` directly. You don't need to fake keyboard events.

This is also the abstraction that the rest of the chapter builds on. Virtual on-screen buttons emit "action" events. Gamepad support reads from the same action map. Replays serialize `actions[]` per frame, not `keys[]`. The action map is the **contract** between any input device and the game.

A tip: keep the action names in the present tense and game-domain ("fire", "jump", "interact") rather than physical-domain ("space", "left-click"). When you migrate the same game from desktop to console to mobile, the gameplay code does not change — only the bindings table does.

# --questions--

## --text--

What's the practical reason to map actions to keys rather than keys to actions?

## --answers--

You can rebind keys (or add more keys per action) without touching gameplay logic.

---

It runs faster — `addKeys` is more efficient than `createCursorKeys`.

### --feedback--

Performance is essentially identical. The benefit is structural, not numeric.

---

It avoids a Phaser bug where `cursors.left` only fires the first time.

### --feedback--

There is no such bug. `cursors.left.isDown` is reliable.

---

It's required by the action-map plugin to register actions.

### --feedback--

There's no such required plugin in core Phaser. The pattern is a convention you adopt by hand.

## --text--

In the snippet above, what does `ACTIONS.fire.some(k => this.keys[k].isDown)` do?

## --answers--

Returns `true` if **any** key bound to the `fire` action is currently held.

---

Returns `true` only if **all** keys bound to `fire` are held simultaneously.

### --feedback--

That would be `.every`. `.some` is OR-style.

---

Returns the first key currently held in `ACTIONS.fire`.

### --feedback--

`.some` returns a boolean, not an element. You're thinking of `.find`.

---

Throws — `keys` is a plain object, not an array.

### --feedback--

`this.keys` is the object Phaser hands back from `addKeys`, indexed by key name. `this.keys[k]` is the `Key` instance for that name.

## --text--

You're adding gamepad support. Where does the rebinding code go in the action-map pattern?

## --answers--

You add a second resolver (for `pad.buttons[i].pressed`) and union it with the keyboard resolver inside `isDown`.

---

You rewrite every `if (this.isDown('left'))` in `update` to also check the gamepad.

### --feedback--

That defeats the entire purpose of the indirection. The gameplay code should not change.

---

You replace `ACTIONS` with a gamepad-only object.

### --feedback--

You'd lose keyboard support. `isDown` should resolve true when **any** bound device fires the action.

---

You can't — Phaser's gamepad plugin requires a different pattern.

### --feedback--

The gamepad plugin reads via `pad.buttons`, which slots cleanly into the same OR-of-resolvers pattern.
