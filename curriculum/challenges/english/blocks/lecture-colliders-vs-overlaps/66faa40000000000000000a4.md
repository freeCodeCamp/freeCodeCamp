# --interactive--

Arcade physics gives you two ways to react when bodies meet, and choosing the wrong one is the most common Phaser bug.

A **collider** physically blocks movement. The engine resolves the contact: bodies separate, velocities reflect off each other, and the world stays solid. Use `this.physics.add.collider(a, b, callback)` for player-versus-ground, ball-versus-paddle, enemy-versus-wall — anything where the bodies should not be allowed to occupy the same space.

:::interactive_editor

```js
function create() {
  this.ball = this.add.circle(120, 30, 8, 0xffaa44);
  this.physics.add.existing(this.ball);
  this.ball.body.setBounce(0.7);

  this.floor = this.add.rectangle(120, 170, 240, 12, 0x66ddff);
  this.physics.add.existing(this.floor, true);

  this.physics.add.collider(this.ball, this.floor);
}
```

:::

An **overlap** fires a callback when two bodies share space, but does *not* push them apart. Use `this.physics.add.overlap(a, b, callback)` for pickups (the player walks through a coin), trigger zones (entering a region starts a cutscene), and damage areas where you want the player to keep moving.

:::interactive_editor

```js
function create() {
  this.player = this.add.rectangle(60, 90, 16, 16, 0xffffff);
  this.physics.add.existing(this.player);
  this.player.body.setVelocityX(80);

  this.coin = this.add.circle(180, 90, 6, 0xffee44);
  this.physics.add.existing(this.coin);
  this.coin.body.setAllowGravity(false);

  this.physics.add.overlap(this.player, this.coin, (_p, c) => c.destroy());
}
```

:::

The signatures are mirror images: `collider(a, b, collideCallback, processCallback, context)` and `overlap(a, b, overlapCallback, processCallback, context)`. The same arguments, different physics behavior. The third argument is your "what happened" callback, which receives the two game objects in order — exactly the pair you passed to `add.collider` or `add.overlap`.

Pick the API by asking one question: *should the bodies be able to occupy the same space?* If yes, you want overlap. If no, you want collider.

# --questions--

## --text--

The player should walk through a coin and trigger a callback that destroys it. Which API is right?

## --answers--

`this.physics.add.overlap(player, coin, callback)`.

---

`this.physics.add.collider(player, coin, callback)`.

### --feedback--

That blocks the player's movement on the coin — they would bump into it instead of walking through.

---

A `pointerdown` listener on the coin.

### --feedback--

That fires on click, not on physical contact.

---

A scene-event subscription on `coin:pickup`.

### --feedback--

That works only if you manually emit it; nothing wires it to physics.

## --text--

You set up `this.physics.add.collider(ball, floor)` but did not pass any callback. What happens when the ball lands on the floor?

## --answers--

The ball stops at the floor, bouncing per its `setBounce` value, and no user callback runs.

---

Phaser throws because `collider` requires a callback.

### --feedback--

The collide and process callbacks are both optional.

---

The ball passes through because no callback opted in to the collision.

### --feedback--

A collider always blocks movement; the callback is only for *reacting* to the contact.

---

The collision is silently ignored.

### --feedback--

The physics resolution still runs even without a callback.

## --text--

You want a "kill zone" rectangle: the player keeps moving when they enter it, but `loseLife` runs on entry. Which API matches?

## --answers--

`this.physics.add.overlap(player, killZone, () => loseLife())`.

---

`this.physics.add.collider(player, killZone, () => loseLife())`.

### --feedback--

That blocks the player at the edge of the zone instead of letting them enter and continue.

---

A bare `if (player.body.touching.up)` check inside `update`.

### --feedback--

`touching` flags require a registered collider/overlap to populate.

---

`this.events.on("kill")` and emit it from `update`.

### --feedback--

Events do not detect spatial intersection on their own.
